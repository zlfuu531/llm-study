## 1. 推理阶段：prefill 和 decode

Prefill：

- 输入完整 prompt。
- 一次性计算所有输入 token。
- 生成每层 KV Cache。
- 更偏计算密集。

Decode：

- 每次生成一个新 token。
- 只计算新 token 的 Q/K/V。
- 读取历史 KV Cache。
- 更偏内存带宽和调度瓶颈。

**面试答案：prefill 和 decode 区别？**  
prefill 是处理 prompt，能并行；decode 是自回归逐 token 生成，依赖 KV Cache，串行性更强。prefill 影响首 token 延迟，decode 影响持续输出速度。

## 2. 推理指标

| 指标 | 含义 | 用户体感 |
| --- | --- | --- |
| TTFT | Time To First Token，首 token 延迟 | 用户等多久开始看到输出 |
| TPOT | Time Per Output Token，每 token 耗时 | 输出流不流畅 |
| QPS | 每秒请求数 | 服务能接多少请求 |
| tokens/s | 每秒 token 吞吐 | GPU 产出能力 |
| P95/P99 | 尾延迟 | 高峰期体验 |

**面试答案：优化看哪些指标？**  
交互体验看 TTFT 和 TPOT，系统吞吐看 tokens/s 和 QPS，稳定性看 P95/P99、错误率、超时率、GPU 利用率和显存。

## 3. KV Cache

KV Cache 是 decode 阶段的核心。

缓存：

- 每一层。
- 每个历史 token。
- K 和 V。

显存估算：

```text
KV cache ≈ batch * seq_len * layers * kv_heads * head_dim * 2 * bytes
```

降低 KV Cache：

- MQA/GQA/MLA。
- 限制上下文长度。
- KV Cache 量化。
- PagedAttention。
- Prefix cache。
- Sliding window。

## 4. PagedAttention

问题：

- 每个请求长度不同。
- KV Cache 需要连续显存会造成浪费和碎片。

PagedAttention：

- 把 KV Cache 拆成 block。
- 像操作系统分页一样管理。
- 提高显存利用率。
- 支持更高并发。

**面试答案：PagedAttention 解决什么？**  
解决 KV Cache 显存碎片和浪费，让不同长度请求的缓存能更灵活地分配和复用，是 vLLM 高吞吐的重要基础。

## 5. Continuous Batching

传统 batching：

- 等一批请求一起跑。
- 某些请求先结束，其他请求还在跑。
- GPU 利用率不稳定。

Continuous batching：

- decode 过程中动态加入新请求。
- 完成的请求退出。
- 未完成的继续。

**面试答案：Continuous Batching 是什么？**  
它是动态批处理，让在线推理在输出长度不一致时仍能保持较高 GPU 利用率和吞吐。

## 6. Prefix Cache

很多请求共享相同前缀：

- 系统 prompt。
- 固定角色设定。
- 相同文档上下文。
- 同一模板。

Prefix cache 缓存公共前缀的 KV Cache，后续请求复用。

**面试答案：prefix cache 有什么用？**  
减少重复 prefill 计算，降低 TTFT 和成本，适合固定系统提示词、RAG 长上下文模板和多用户共享前缀。

## 7. 量化

量化把权重从高精度变低精度：

- FP16/BF16
- INT8
- INT4
- GPTQ
- AWQ

优点：

- 降低显存。
- 降低带宽压力。
- 可能提升推理速度。

风险：

- 精度下降。
- 幻觉增加。
- 数学/代码能力下降。
- 格式遵循变差。

**面试答案：INT4 一定比 FP16 快吗？**  
不一定。量化能减少显存和带宽，但是否更快取决于硬件、kernel、batch、模型结构和解量化开销。必须实测。

## 8. GPTQ 和 AWQ

GPTQ：

- 后训练量化。
- 基于近似二阶信息逐层优化量化误差。

AWQ：

- Activation-aware。
- 保护对激活更重要的通道。

**面试够用版：**  
GPTQ 和 AWQ 都是常见 PTQ 方法，目标是在少量校准或不重新训练的情况下压缩模型权重并尽量保持效果。

更完整的量化公式、GPTQ/AWQ/SmoothQuant、QLoRA、FP8、KV Cache 量化、GGUF 和 INT4 提速取舍，见：[37_模型量化_低比特推理_GPTQ_AWQ_SmoothQuant面试.md](../37_模型量化_低比特推理_GPTQ_AWQ_SmoothQuant面试.md)。

## 9. Speculative Decoding

流程：

1. 小模型快速草拟多个 token。
2. 大模型并行验证。
3. 接受的 token 一次性提交。
4. 拒绝时回退。

**面试答案：怎么加速？**  
减少大模型逐 token 调用次数。如果小模型草稿被大模型高比例接受，就能明显加速。

推测解码、EAGLE、Medusa、MTP、accept rate 和无损验证的完整复习见：[38_推测解码_DraftModel_EAGLE_Medusa_MTP面试.md](../38_推测解码_DraftModel_EAGLE_Medusa_MTP面试.md)。

限制：

- 小模型要足够接近大模型。
- 接受率低则收益小。
- 系统实现更复杂。

## 10. 长上下文为什么贵

Prefill：

- attention 复杂度近似 `O(T^2)`。

Decode：

- KV Cache 随 T 线性增长。

其他问题：

- 上下文越长，模型可能忽略关键内容。
- prompt 成本增加。
- 延迟上升。

**优化：**

- RAG 精简上下文。
- prompt compression。
- sliding window。
- long-context scaling。
- cache reuse。

## 12. 高并发怎么做

手段：

- 请求队列。
- 优先级调度。
- 限流。
- 超时取消。
- 降级小模型。
- 限制 max tokens。
- 限制上下文长度。
- continuous batching。
- prefix cache。
- 热点答案缓存。

**面试答案：高并发下如何降级？**  
先保护系统稳定：限制低优先级请求、缩短输出长度、降低上下文长度、路由到小模型或缓存结果，必要时直接拒绝并提示稍后重试。

## 14. 显存估算

权重：

```text
params * bytes_per_param
```

例子：

- 7B FP16 约 14GB。
- 14B FP16 约 28GB。
- INT4 权重大约四分之一，但有 scale/metadata 和 runtime overhead。

总显存：

```text
模型权重 + KV Cache + 激活/临时 buffer + 框架开销
```

**面试答案：部署 7B 需要多少显存？**  
FP16 权重大约 14GB，但实际还要加 KV Cache 和运行时开销。如果并发、上下文较大，显存需求会明显上升。

## 15. vLLM 面试图谱：为什么它常被问

如果面试官问“vLLM 为什么快”，不要只背 PagedAttention。更完整的图谱是：

```text
显存管理：PagedAttention / block KV cache
请求调度：continuous batching / 动态进出 batch
重复前缀：prefix cache / automatic prefix caching
长输入：chunked prefill / prefill 调度
长输出：高效 decode kernel / KV cache 读写优化
部署服务：OpenAI-compatible server / metrics / tracing / 多卡并行
```

**面试答案：vLLM 为什么吞吐高？**  
核心是把在线推理当成“动态 token 调度 + KV Cache 管理”问题。PagedAttention 用 block 管理 KV Cache，减少不同长度请求带来的显存碎片；continuous batching 让 decode 阶段可以动态加入新请求；prefix cache 避免相同系统 prompt 或 RAG 前缀重复 prefill。它们组合起来提升并发下的 GPU 利用率和显存利用率。

**追问：PagedAttention 和 OS 分页像在哪里？**  
相似点是都把连续的大块逻辑空间切成固定大小 block/page，通过映射表管理，减少连续分配压力。不同点是 PagedAttention 管的是 Transformer 推理里的 KV Cache，不是通用内存管理；它还要配合 attention kernel 高效读取历史 K/V。

## 16. Chunked Prefill 与调度权衡

prefill 和 decode 的资源特征不同：

- prefill：一次处理很多 prompt token，矩阵乘大，偏计算密集。
- decode：每次生成一个 token，要频繁读历史 KV Cache，偏内存带宽和调度。

长 prompt 会把 prefill 占满，导致短请求的 TTFT 变差。Chunked prefill 的思想是把长输入拆成小块，让 prefill 和 decode 更公平地穿插调度。

```text
长请求 prefill 全量占用 GPU
  -> 短请求排队
  -> TTFT 变差

长请求 prefill 分块
  -> decode 和短请求能插进来
  -> 吞吐、TTFT、TPOT 之间做权衡
```

**面试答案：为什么不能只追吞吐？**  
吞吐高不代表体验好。如果系统为了 tokens/s 把 batch 做得很大，用户可能 TTFT 很慢；如果过度照顾低延迟，GPU 利用率又会下降。线上调参要同时看 TTFT、TPOT、P95/P99、tokens/s、队列长度和取消率。

## 17. 容量估算与上线调参

面试里被问“你怎么估一台卡能扛多少并发”，可以这样拆：

```text
总显存 = 模型权重 + KV Cache + 临时 buffer + 框架开销
可用于 KV 的显存 = 总显存 - 权重 - 预留
单请求 KV ≈ layers * seq_len * kv_heads * head_dim * 2 * bytes
可承载请求数 ≈ 可用于 KV 的显存 / 单请求 KV
```

如果用 GQA/MQA：

- MHA：`kv_heads = attention_heads`，KV Cache 最大。
- GQA：多个 query head 共享一组 K/V，`kv_heads` 变小。
- MQA：所有 query head 共享一组 K/V，KV Cache 更小。

调参顺序：

1. 固定模型、硬件、量化方式，先跑基准压测。
2. 分别测短输入短输出、长输入短输出、短输入长输出、长输入长输出。
3. 观察 TTFT、TPOT、tokens/s、P95/P99、显存、GPU 利用率、队列长度。
4. 调 `max_model_len`、`max_num_batched_tokens`、并发上限、输出上限、prefix cache、量化和路由。
5. 用真实业务流量回放，而不是只看随机 prompt。

**面试答案：显存够但吞吐上不去怎么办？**  
先看瓶颈。GPU 利用率低可能是调度、batch 太小、网络或 tokenizer 慢；显存占用高但 tokens/s 低可能是长上下文 KV 读写、decode 带宽或 kernel 不合适；P95 高可能是排队、长请求阻塞或 max tokens 太大。不要只说“加机器”，先用指标定位。

## 19. 本章高频问答

### Q1：decode 为什么常受内存带宽影响？

decode 每步只生成一个 token，计算规模小，但每层都要读取历史 KV Cache。长上下文和高并发时，读写缓存成为瓶颈。

### Q2：PagedAttention 和 FlashAttention 区别？

FlashAttention 优化 attention 计算和显存读写，主要减少中间 attention 矩阵存储；PagedAttention 优化 KV Cache 管理，解决推理时缓存碎片和显存利用率。

### Q3：怎么降低推理成本？

量化、小模型蒸馏、KV Cache 优化、prefix cache、continuous batching、限制 max tokens、RAG 精简上下文、热点缓存、模型路由。

如果面试继续追“线上如何做多模型路由、token 限流、trace、成本归因、fallback 和灰度回滚”，转到 [20_LLMOps模型网关与可观测性面试.md](../20_LLMOps模型网关与可观测性面试.md) 系统复习。

更完整的端侧小模型、蒸馏、剪枝、GGUF/llama.cpp、本地部署和端侧 eval 复习见：[14_端侧小模型与模型压缩面试.md](../14_端侧小模型与模型压缩面试.md)。

更完整的低比特推理、GPTQ/AWQ/SmoothQuant、FP8 和 KV Cache 量化复习见：[37_模型量化_低比特推理_GPTQ_AWQ_SmoothQuant面试.md](../37_模型量化_低比特推理_GPTQ_AWQ_SmoothQuant面试.md)。

### Q4：量化后怎么评估？

不仅看平均准确率，还要看幻觉、格式遵循、代码/数学能力、安全、边界 case、延迟和吞吐。

### Q5：vLLM 为什么适合在线服务？

因为它围绕在线请求的真实形态做了优化：请求长短不一、输出长度不一、KV Cache 大、重复前缀多、需要流式输出和监控。PagedAttention、continuous batching、prefix cache、服务端指标和 OpenAI-compatible 接口让它更适合工程落地。

### Q6：chunked prefill 解决什么问题？

解决长 prompt prefill 阶段阻塞调度的问题。把长输入拆块后，系统能在 prefill 和 decode、长请求和短请求之间更公平地调度，改善 TTFT 和尾延迟，但参数不当也可能影响吞吐。

### Q7：并发上不去一定是显存不够吗？

不一定。可能是 KV Cache 显存限制，也可能是 decode 内存带宽、batch 调度、tokenizer、网络、输出太长、P95 排队或安全过滤耗时。面试中要用指标定位，而不是直接给结论。

## 20. 面试前背诵版

大模型推理分 prefill 和 decode。prefill 处理 prompt，偏计算密集；decode 逐 token 生成，依赖 KV Cache，常受内存带宽和调度限制。优化方向包括 PagedAttention 管理 KV Cache、Continuous Batching 提升 GPU 利用率、Prefix Cache 复用前缀、Chunked Prefill 改善长输入调度、量化降低显存、Speculative Decoding 用小模型草拟大模型验证。线上还要看 TTFT、TPOT、tokens/s、P95/P99、队列长度、显存、错误率、取消率和成本。
