# 完整学习章 06：项目、面经与冲刺

## 你学完要能做到什么

这一章是把知识变成面试表现。学完后你要能：

- 准备 3 分钟自我介绍。
- 每个项目讲 2 分钟版和 8 分钟版。
- 用真实面经反推复习重点。
- 做完整模拟面试。
- 面试后复盘错题、项目漏洞和手撕问题。

## 1. 自我介绍

模板：

```text
面试官您好，我叫 X，目前主要方向是大模型相关的算法/应用开发。
我比较熟悉 Transformer、RAG、Agent、微调和推理部署这些内容。
项目上，我做过 X 项目，主要解决 Y 问题，我负责 Z 部分，里面涉及 A、B、C 技术。
最近我重点学习的是大模型相关方向，所以也系统复习了手撕 Attention、RoPE、LoRA、DPO 以及 RAG/Agent 的工程评估。
```

原则：

- 不超过 3 分钟。
- 主动引导到你准备最好的项目。
- 少罗列工具，多讲方向和贡献。

## 2. 项目讲法结构

按这个顺序：

1. 背景：为什么要做。
2. 目标：解决什么问题。
3. 方案：系统怎么设计。
4. 你的工作：你负责什么。
5. 难点：遇到什么问题。
6. 指标：效果如何。
7. 复盘：还能怎么改。

不要只说：

> 我用了 LangChain 和向量数据库。

要说：

> 我解决的是知识库问答中召回不准和幻觉的问题，具体在 chunk、混合召回、rerank、引用校验和 bad case 回流上做了优化。

### 2 分钟版和 8 分钟版怎么切换

2 分钟版只讲主线：

```text
场景 -> 问题 -> 方案 -> 我的工作 -> 指标 -> 难点
```

8 分钟版要讲细节：

```text
业务约束
  -> baseline 是什么
  -> 为什么旧方案不够
  -> 方案拆成哪些模块
  -> 每个模块怎么选型
  -> 做了哪些实验/评估
  -> 上线后怎么监控
  -> bad case 怎么回流
  -> 还有什么局限
```

可以用这个模板：

```text
这个项目的背景是 X，用户原来的痛点是 Y。最初 baseline 是 Z，但它在 A/B/C 三个问题上不够好。
我负责的是其中的 M 模块，主要做了 1/2/3 三件事。
技术上我没有只堆组件，而是先定义评估集和指标，再分别优化召回、生成、延迟或成本。
上线后看到了指标变化：核心指标从 a 到 b，bad case 主要集中在 c。
我最后的复盘是：这个方案解决了主要问题，但在 d 场景还有局限，后续可以用 e 继续改。
```

### 项目证据包

面试官最怕听到“效果不错”。你要准备一个证据包：

| 证据 | 你要准备什么 |
| --- | --- |
| 任务定义 | 用户是谁，输入输出是什么，成功标准是什么 |
| baseline | 原来方案、prompt-only、纯向量检索、无 rerank、无量化等 |
| 指标 | Recall@k、faithfulness、任务完成率、TTFT、P95、成本等 |
| 数据 | 数据来源、规模、清洗、脱敏、训练/验证/测试划分 |
| 实验 | 做过哪些 ablation，哪一项最有效 |
| bad case | 至少 3 类失败样例和定位结论 |
| 上线 | 灰度、监控、回滚、日志、权限、安全 |
| 复盘 | 当前方案局限和下一步 |

**面试答案：项目没有线上指标怎么办？**  
可以诚实说是课程/实验/实习内部项目，但要补离线证据：构造了多少条评估集、baseline 是什么、哪些 case 改善、哪些没改善、延迟和成本怎么估算。不要编线上 DAU、转化率这类无法自洽的数字。

项目评估、私有 eval、LLM-as-judge、ablation 和上线回归的完整讲法，集中看 [../16_大模型评测与实验设计面试.md](../16_大模型评测与实验设计面试.md)。

## 3. RAG 项目话术

2 分钟版：

```text
我做的是一个面向 X 场景的知识库问答系统。离线侧对文档做解析、清洗和结构化切分，保留标题、来源和页码等 metadata，然后用 embedding 建向量索引，同时保留 BM25 做混合召回。在线侧用户问题先做 query rewrite，再走向量召回和关键词召回，候选片段用 reranker 精排，最后把 top 证据拼进 prompt，让模型基于证据回答并返回引用。评估上我看 Recall@k、faithfulness、引用正确率、人工满意度和延迟成本。主要难点是 chunk 粒度、专有名词召回和幻觉控制。
```

必备追问：

- 为什么不用微调？
- chunk size 怎么选？
- embedding 模型怎么选？
- 为什么要 rerank？
- 召回为空怎么办？
- 文档更新怎么处理？
- RAG 幻觉怎么定位？
- 怎么评估？

## 4. Agent 项目话术

2 分钟版：

```text
我做的是一个能调用工具完成 X 任务的 Agent。系统分成任务理解、规划、工具调用、状态管理和结果生成几部分。模型根据用户目标生成下一步动作，按 tool schema 调用检索、数据库或业务 API。每次工具返回后进入 observation，模型决定下一步。为了稳定性，我加了参数校验、最大步数、失败重试、权限控制和 trace 日志。评估上看任务完成率、工具调用准确率、平均步骤数、失败率和成本。
```

必备追问：

- 为什么用 Agent，不用固定工作流？
- 怎么避免死循环？
- 工具调用失败怎么办？
- 权限怎么控制？
- 记忆怎么设计？
- 怎么评估？

## 5. 微调项目话术

2 分钟版：

```text
我做的是一个面向 X 任务的模型微调项目。首先判断 prompt/RAG 是否能解决，确认需要行为适配后，我们收集并清洗了 X 类数据，统一成 instruction-response 格式，对敏感信息做脱敏，并划分训练、验证和测试集。训练上选择 LoRA/QLoRA，冻结基座模型，只训练低秩适配参数。评估上除了自动指标，还做人工 case 评估和业务指标对比。主要难点是数据质量、过拟合和模型在边界问题上的幻觉。
```

必备追问：

- 为什么不用全量微调？
- LoRA rank 怎么选？
- 数据怎么清洗？
- loss mask 怎么做？
- 怎么防止过拟合？
- 安全性变差怎么办？

## 6. 推理部署项目话术

2 分钟版：

```text
我参与的是一个大模型推理服务优化项目。系统需要支持多用户并发和流式输出，所以我们重点优化模型加载、请求调度、KV Cache、批处理和监控。指标上看 TTFT、TPOT、tokens/s、P95 延迟和 GPU 利用率。优化手段包括量化、prefix cache、continuous batching、限制上下文长度和超时降级。主要难点是吞吐和延迟的平衡，以及长上下文带来的 KV Cache 显存压力。
```

必备追问：

- prefill 和 decode 区别？
- P95 延迟高怎么排查？
- 长上下文显存不够怎么办？
- 流式输出怎么实现？
- 量化后怎么评估？

## 7. 指标怎么讲

不要说“效果不错”。要说具体指标。

RAG：

- Recall@k。
- Faithfulness。
- 引用正确率。
- 幻觉率。
- 解决率。
- 转人工率。

Agent：

- 任务完成率。
- 工具调用准确率。
- 参数准确率。
- 平均步骤数。
- 失败率。
- 成本。

推理：

- TTFT。
- TPOT。
- tokens/s。
- P95/P99。
- GPU 利用率。
- 单次成本。

微调：

- 自动指标。
- 人工评估。
- 格式遵循率。
- bad case 通过率。
- 业务指标。

### 指标回答万能结构

```text
我会把指标分成三层：
第一层是技术指标，判断模块有没有做好；
第二层是端到端指标，判断系统输出是否可用；
第三层是业务指标，判断用户是否真的受益。
```

例子：

- RAG 技术指标：Recall@k、MRR、rerank 后命中率。
- RAG 端到端指标：faithfulness、引用正确率、拒答准确率。
- RAG 业务指标：解决率、转人工率、人工满意度、单次成本。

### bad case 怎么讲

不要只说“我看了 bad case”。要按定位链路说：

```text
先判断 gold 文档是否存在
  -> 再判断是否被召回
  -> 再判断是否被 rerank 排到前面
  -> 再判断 prompt 是否包含证据
  -> 再判断模型是否忠于证据
```

常见定位：

- 没有 gold 文档：知识库覆盖问题。
- 有 gold 但没召回：chunk、embedding、query rewrite 或 BM25 问题。
- 召回但没进 top：rerank 问题。
- 进了 prompt 但答错：生成约束、上下文太长或证据冲突。
- 答对但没引用：citation 逻辑问题。

## 8. 项目最容易被追问穿的地方

- 不知道 chunk 怎么切。
- 不知道 embedding 为什么选这个。
- 不知道 rerank 作用。
- 微调不知道 loss mask。
- LoRA rank 随便说。
- 没有评估集。
- 只说提升，没有数字。
- 没有 bad case。
- 不了解上线延迟和成本。
- Agent 没有失败处理。

### 四类项目的高压追问回答框架

RAG 项目：

| 追问 | 回答要点 |
| --- | --- |
| 为什么不用微调 | 知识更新、引用、私有知识优先 RAG；行为适配再微调 |
| chunk 怎么选 | 先按结构切，再用 Recall@k 和 bad case 调 |
| embedding 怎么选 | 语言、领域、长度、速度、维度、业务评测 |
| rerank 有什么用 | 召回求全，rerank 求准，减少无关上下文 |
| 幻觉怎么治理 | 召回、rerank、证据约束、引用、拒答、bad case 回流 |

Agent 项目：

| 追问 | 回答要点 |
| --- | --- |
| 为什么不用 workflow | 流程固定用 workflow；路径不确定、需工具反馈才用 Agent |
| 怎么防死循环 | 最大步数、状态机、终止条件、失败次数、人工兜底 |
| 工具错了怎么办 | schema、参数校验、错误重试、降级、人工确认 |
| 权限怎么控 | 最小权限、工具白名单、高风险确认、审计日志 |
| 怎么评估 | task success、tool accuracy、argument accuracy、steps、cost、safety |

微调项目：

| 追问 | 回答要点 |
| --- | --- |
| 为什么要微调 | prompt/RAG 不足以改变稳定行为或格式 |
| 数据怎么清洗 | 去重、脱敏、纠错、统一模板、去泄露 |
| loss mask 怎么做 | 只训 assistant answer，prompt/pad 设为 -100 |
| LoRA rank 怎么选 | 小 rank 起步，看欠拟合/过拟合和验证集 |
| 安全退化怎么办 | 加拒答/安全样本，评估安全集，灰度回滚 |

推理部署项目：

| 追问 | 回答要点 |
| --- | --- |
| TTFT 高 | prefill 长、排队、prefix cache 未命中、batch 调度 |
| TPOT 高 | decode 带宽、KV Cache、batch、kernel |
| 显存不够 | 量化、限制上下文、GQA/MQA、KV cache 优化、降并发 |
| 量化怎么评估 | 效果、幻觉、格式、数学/代码、安全、延迟吞吐 |
| 高并发怎么降级 | 限流、缩短输出、路由小模型、缓存、拒绝低优先级 |

## 9. 面经怎么用

面经不是用来制造焦虑，而是用来校准重点。

只记录三类：

- 重复出现。
- 自己不会。
- 和自己项目相关。

不要记录：

- 情绪贴。
- 资料包引流。
- 和岗位完全不匹配的题。

每日流程：

1. 看 3-5 篇 2025-2026 面经。
2. 标注岗位。
3. 摘题。
4. 打标签。
5. 判断自己能否 2 分钟回答。
6. 不会就加入错题库。

### 2026 岗位题型地图

| 岗位方向 | 面试最常追 | 项目怎么包装 | 高频踩坑 |
| --- | --- | --- | --- |

### 同一个项目如何按岗位改讲法

假设你做的是 RAG 问答项目：

- 算法岗：重点讲 embedding/rerank 模型选择、评估集、Recall@k、消融实验、失败样本分布。
- 应用岗：重点讲文档解析、chunk、混合召回、rerank、引用、拒答、权限、安全、线上闭环。
- 推理岗：重点讲长上下文成本、prefix cache、batch、TTFT/TPOT、限流降级、模型路由。
- 搜广推岗：重点讲召回/排序、多路召回融合、NDCG/MRR、query 改写、线上 A/B。

**面试答案：为什么要按岗位改讲法？**  
因为面试官在判断你是否匹配岗位。算法岗想看你能不能做实验和理解模型；应用岗想看你能不能把系统落地；推理岗想看你能不能扛住性能和成本；搜广推岗想看你是否懂排序指标和业务闭环。同一个项目要保留事实不变，但突出不同证据。

### 目标岗位不清楚时怎么准备

如果 JD 写得很泛，比如“大模型算法工程师 / AI 应用工程师”，就按这个优先级准备：

1. RAG + Agent 项目必须能讲 8 分钟。
2. Transformer、LoRA、DPO、ORPO、GRPO 能讲公式和直觉。
3. 至少 8 道 LLM 手撕能脱稿。
4. 推理指标 TTFT/TPOT/KV Cache/vLLM 能说清。
5. 公共基础 AUC、交叉熵、AdamW、LayerNorm/RMSNorm 不能丢。

这样即使岗位在算法和应用之间摇摆，也不会一问就露出断层。

## 16. 面试前背诵版

项目面要讲清背景、目标、方案、你的工作、难点、指标和复盘。RAG 项目重点讲 chunk、混合召回、rerank、幻觉治理和评估；Agent 项目重点讲工具 schema、参数校验、最大步数、权限和 trace；微调项目重点讲数据清洗、loss mask、LoRA 配置和评估；推理项目重点讲 TTFT、TPOT、KV Cache、batching 和显存。面经只用来反推重点，不能被资料流带跑。
