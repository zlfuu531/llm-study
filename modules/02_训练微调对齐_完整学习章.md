## 1. 预训练

Decoder-only LLM 常用 Causal Language Modeling：

```text
给定 x_1...x_t，预测 x_{t+1}
```

训练目标是 next token cross entropy。

**面试答案：预训练解决什么？**  
预训练让模型从海量语料中学习语言规律、世界知识、推理模式和代码模式。它解决“模型有没有基础能力”的问题。

**追问：训练时是不是也逐 token 生成？**  
不是。训练时用 causal mask，可以并行计算所有位置的 next token loss；推理时才逐 token 生成。

## 2. SFT

SFT，即 Supervised Fine-Tuning，用指令-回答数据训练模型。

典型数据：

```json
{
  "instruction": "请解释 KV Cache",
  "input": "",
  "output": "KV Cache 是..."
}
```

**面试答案：预训练和 SFT 的区别？**  
预训练解决“模型会不会语言和知识”，SFT 解决“模型会不会按指令回答”。预训练语料更大更杂，SFT 数据更少但质量更高。

**为什么只对 answer 算 loss？**  
我们希望模型学习回答，而不是学习复述用户问题。工程上通常把 prompt 部分 label 设为 `-100`，只对 answer token 计算 cross entropy。

## 3. 数据清洗

微调效果很大程度取决于数据质量。

要清洗：

- 重复数据。
- 低质量回答。
- 格式不一致。
- 隐私和敏感信息。
- 事实错误和互相矛盾样本。
- 泄露测试集的数据。

**面试答案：为什么数据清洗重要？**  
SFT 和偏好对齐数据量相对预训练小，脏数据占比稍高就会明显改变模型行为，导致幻觉、格式混乱、安全性下降或过拟合。

更完整的训练显存估算、DDP、ZeRO/FSDP、3D 并行、activation checkpointing 和通信排查复习见：[17_大模型训练系统与分布式训练面试.md](../17_大模型训练系统与分布式训练面试.md)。

## 4. LoRA

原线性层：

```text
y = Wx
```

LoRA 冻结 `W`，训练低秩分支：

```text
y = Wx + scale * B(Ax)
scale = alpha / rank
```

shape：

```text
W: [out_dim, in_dim]
A: [rank, in_dim]
B: [out_dim, rank]
```

参数量：

```text
rank * in_dim + out_dim * rank
```

**面试答案：LoRA 为什么有效？**  
预训练模型已经有通用能力，下游任务通常只需要在少数方向上调整行为，权重更新可能近似低秩。因此只训练低秩矩阵就能达到不错的适配效果。

**rank 和 alpha：**

- rank 控制 LoRA 分支容量。
- alpha 控制 LoRA 更新强度。
- 常见 scale 是 `alpha / rank`。

**LoRA 加在哪里？**  
常见加在 attention 的 `q_proj`、`v_proj`，也可以加 `k_proj`、`o_proj` 和 FFN 的 `up/down/gate_proj`。资源少先加 Q/V，效果不够再扩大范围。

## 5. QLoRA

QLoRA 的核心是：

- 冻结的基座模型用 4-bit 等低比特量化存储。
- LoRA 参数仍以较高精度训练。
- 只更新 LoRA 参数。

**面试答案：QLoRA 为什么省显存？**  
它把大头的基座权重量化存储，并且不保存全量训练参数和优化器状态。显存主要省在冻结基座权重和 optimizer state 上。

**误区：**  
不是所有计算都变成 4-bit，也不是 LoRA 参数一定是 4-bit。

## 6. RAG 还是微调

判断：

| 问题类型 | 优先方案 |
| --- | --- |
| 知识更新快 | RAG |
| 企业私有知识 | RAG |
| 需要引用来源 | RAG |
| 输出格式不稳定 | SFT/LoRA |
| 需要学领域任务模式 | SFT/LoRA |
| 模型不会某种风格 | SFT/LoRA |
| 既要知识又要格式 | RAG + SFT/LoRA |

**面试答案：怎么选？**  
RAG 更适合补知识，微调更适合改行为。业务知识库问答通常先 RAG；如果模型不会按格式输出或领域表达差，再用 LoRA/SFT 做行为适配。

## 7. RLHF

RLHF 流程：

1. SFT 得到基础指令模型。
2. 对同一 prompt 采样多个回答。
3. 人类标注哪个回答更好。
4. 训练 Reward Model。
5. 用 PPO 优化策略模型。
6. 加 KL 约束防止偏离 reference model。

**面试答案：RLHF 解决什么？**  
SFT 只能模仿标准答案，RLHF 通过人类偏好让模型更有帮助、更安全、更符合人类期待。

## 8. Reward Model 与 PPO

Reward Model 输入 prompt + response，输出一个标量 reward。

训练偏好损失：

```text
loss = -log sigmoid(r_chosen - r_rejected)
```

PPO 做什么：

- 根据 reward 更新策略模型。
- 用 KL 约束限制偏离。
- 用 advantage 估计哪些回答该鼓励。

**面试答案：为什么 RLHF 需要 KL？**  
防止模型为了骗 Reward Model 而偏离原模型太远，导致语言质量下降或 reward hacking。

## 9. DPO

DPO 不显式训练 Reward Model，直接用 chosen/rejected 偏好对训练。

核心：

```text
pi_logratio = logp_policy_chosen - logp_policy_rejected
ref_logratio = logp_ref_chosen - logp_ref_rejected
loss = -logsigmoid(beta * (pi_logratio - ref_logratio))
```

**面试答案：DPO 和 PPO 的区别？**  
PPO 是强化学习方法，需要 Reward Model、采样和 advantage，训练复杂。DPO 直接利用偏好对做优化，不显式训练 Reward Model，更像监督式偏好优化，流程更简单。

**为什么需要 reference model？**  
reference model 是锚点，防止当前模型为了偏好数据过度漂移。DPO 优化的是 policy 相对 reference 的偏好变化。

## 10. ORPO

ORPO，即 Odds Ratio Preference Optimization。它也是离线偏好优化，用的数据仍然是：

```text
(prompt, chosen, rejected)
```

但它和 DPO 的关键区别是：**ORPO 不需要 reference model**，而是把 SFT 和偏好优化合成一个目标。

直觉：

- SFT 部分：继续提高 chosen answer 的似然。
- Odds-ratio 部分：让 chosen 的 odds 高于 rejected 的 odds。
- 单阶段训练：不用先 SFT 再 DPO，也不用额外 reward model / value model。

简化写法：

```text
L_ORPO = L_SFT + lambda * L_OR
L_OR = -log sigmoid(log odds(chosen) - log odds(rejected))
odds(y|x) = P(y|x) / (1 - P(y|x))
```

其中 `P(y|x)` 通常用回答 token 的平均 logprob 转成序列概率近似，避免长回答天然概率更低。

**面试答案：ORPO 是什么？**
ORPO 是一种 reference-free 的偏好优化方法。它不显式训练 Reward Model，也不需要 DPO 那样保留 reference model，而是在 SFT 目标之外加入 odds ratio 偏好项：提高 chosen 的概率，同时压低 rejected 的相对 odds。它适合有 chosen/rejected 偏好对、希望简化 post-training 流程的场景。

**和 DPO 的区别？**

- DPO：比较 policy 相对 reference 是否更偏好 chosen，通常需要冻结 reference model。
- ORPO：直接在当前 policy 上做 SFT + odds-ratio 偏好惩罚，不需要 reference model。
- DPO 更像 SFT 后的偏好优化阶段；ORPO 更像把 instruction tuning 和 preference tuning 合并。

注意：ORPO 不是“没有约束就一定更稳”。它少了 reference model 这个显式锚点，所以更依赖偏好数据质量、`lambda` 权重、学习率、长度归一化和评估回归。

## 11. GRPO

GRPO 常和 DeepSeek-R1、reasoning model 一起被问。

核心：

- 对同一个 prompt 采样一组回答。
- 用组内相对奖励估计优势。
- 减少对 value model 的依赖。

**面试答案：GRPO 为什么最近常被问？**  
它和强化 reasoning model 的训练有关。相比 PPO，GRPO 用组内相对表现估计 advantage，降低了 value model 的复杂度和训练成本。

一句话：

> GRPO 是同题多答案、组内比较，好的鼓励，差的压低。

## 12. 对齐算法公式地图：PPO、DPO、ORPO、GRPO

先把三个算法放在一张图里：

| 方法 | 数据来源 | 需要 Reward Model | 需要 Value/Critic | 是否在线采样 | 核心约束 |
| --- | --- | --- | --- | --- | --- |
| PPO/RLHF | prompt + 模型采样 + reward | 通常需要 | 通常需要 | 是 | KL / clip |
| DPO | 离线 chosen/rejected 偏好对 | 不显式需要 | 不需要 | 否 | reference model |
| ORPO | 离线 chosen/rejected 偏好对 | 不需要 | 不需要 | 否 | SFT + odds ratio |
| GRPO | prompt + 组内多个采样 + reward 函数/模型 | 可用 reward 函数或 RM | 不需要 value model | 是 | group advantage + KL |

### PPO 公式直觉

PPO 不是只让 reward 变大，还要防止策略一步改太猛。简化目标：

```text
r_t(theta) = pi_theta(a_t | s_t) / pi_old(a_t | s_t)
L = E[min(r_t * A_t, clip(r_t, 1-eps, 1+eps) * A_t)]
```

在 LLM 里可以理解为：

- `a_t` 是第 t 个 token。
- `s_t` 是 prompt + 前面已生成 token。
- `A_t` 是这个 token/回答相对平均水平好不好。
- clip 防止新策略和旧策略差太远。
- KL 约束防止模型为了 reward 牺牲语言质量。

### DPO 公式直觉

DPO 的训练样本是：

```text
(prompt, chosen, rejected)
```

先算四个 logprob：

```text
log pi_theta(chosen | prompt)
log pi_theta(rejected | prompt)
log pi_ref(chosen | prompt)
log pi_ref(rejected | prompt)
```

然后比较“当前模型相对 reference 是否更偏好 chosen”：

```text
pi_logratio  = logp_theta_chosen - logp_theta_rejected
ref_logratio = logp_ref_chosen   - logp_ref_rejected
loss = -log sigmoid(beta * (pi_logratio - ref_logratio))
```

如果当前模型已经比 reference 更偏好 chosen，loss 变小；如果它更偏好 rejected，loss 变大。

### ORPO 公式直觉

ORPO 仍然看 chosen/rejected，但不引入 reference model。

先看 SFT 部分：

```text
L_SFT = -log pi_theta(chosen | prompt)
```

再看偏好部分：

```text
odds_chosen   = P_theta(chosen | prompt) / (1 - P_theta(chosen | prompt))
odds_rejected = P_theta(rejected | prompt) / (1 - P_theta(rejected | prompt))
L_OR = -log sigmoid(log odds_chosen - log odds_rejected)
```

合在一起：

```text
L_ORPO = L_SFT + lambda * L_OR
```

直觉是：chosen 既作为标准答案被学习，又在偏好项里相对 rejected 被拉高；rejected 不参与 SFT 监督，而是在 odds-ratio 项里被压低。

### GRPO 公式直觉

GRPO 对同一个 prompt 采样一组回答：

```text
o_1, o_2, ..., o_G
```

每个回答拿到 reward：

```text
r_1, r_2, ..., r_G
```

组内 advantage 常写成：

```text
A_i = (r_i - mean(r_1...r_G)) / (std(r_1...r_G) + eps)
```

直觉：

- 同一题下，超过组内平均的回答被鼓励。
- 低于组内平均的回答被压低。
- 不再额外训练 value model 来估计 baseline。

**面试答案：GRPO 省在哪里？**  
PPO/RLHF 通常要 policy、reference、reward model、value/critic 等组件，工程复杂、显存和训练成本高。GRPO 用同题多回答的组内相对 reward 估计 advantage，减少 value model 依赖，所以在 reasoning RL 里更受关注。

## 13. 微调训练细节：sequence logprob、loss mask、packing

DPO/ORPO/GRPO/RLHF 都绕不开一个底层问题：怎么计算一段回答的 log probability。

语言模型输出：

```text
logits: [B, T, V]
labels: [B, T]
```

对每个位置：

```text
log_probs = log_softmax(logits, dim=-1)
token_logp = gather(log_probs, labels)
```

回答序列 logprob：

```text
sequence_logp = sum(token_logp * answer_mask)
```

为什么要 `answer_mask`：

- prompt 部分不该参与回答偏好。
- padding 部分不该参与 loss。
- 多轮对话里可能只训练 assistant 的 token。

**追问：DPO 里 logprob 是平均还是求和？**  
常见实现会对 answer token 求和，也有长度归一化变体。面试要说清楚：求和更贴近整段回答概率，但会受到长度影响；平均能缓解长度差异，但改变了优化偏好。实际要和框架实现、数据长度分布、评估结果一致。

### Packing 为什么会影响 loss mask

packing 是把多个短样本拼进一个长序列，提高 GPU 利用率：

```text
[sample1 prompt+answer] [sample2 prompt+answer] [sample3 prompt+answer]
```

注意：

- 不同样本之间不能互相 attention，或至少 loss mask 必须正确。
- prompt token、padding token 要设成 `-100` 或 mask 掉。
- 拼接时 EOS、role token、chat template 不能丢。

**面试答案：微调 loss mask 写错会怎样？**  
如果 prompt 也算 loss，模型会学会复述用户问题；如果 padding 没 mask，会污染梯度；如果多轮 assistant/user 边界错了，模型会学用户说话或输出角色混乱。

## 14. 2026 高频追问：分别需要几个模型、训练哪些参数？

### LoRA/QLoRA

训练时：

- base model 冻结。
- LoRA A/B 训练。
- optimizer state 只为 LoRA 参数保存。
- QLoRA 中 base model 低比特存储，梯度通过量化基座传到 LoRA。

推理时：

- LoRA 可以单独加载。
- 也可以 merge 到 base 权重里。
- QLoRA 的训练量化不等于部署一定要同样量化。

### DPO

训练时通常有：

- policy model：要训练的模型。
- reference model：冻结的锚点模型。

不显式需要：

- reward model。
- value model。
- 在线 rollout。

训练参数：

- 全参 DPO：更新 policy 全部参数。
- LoRA DPO：只更新 LoRA 参数，更常见。

### ORPO

训练时通常有：

- policy model：要训练的模型。
- chosen/rejected 偏好对：chosen 同时用于 SFT，rejected 用于 odds-ratio 偏好项。

不需要：

- reference model。
- reward model。
- value model。
- 在线 rollout。

训练参数：

- 全参 ORPO：更新 policy 全部参数。
- LoRA ORPO：只更新 LoRA 参数，用于低成本偏好微调。

高频追问：

- `lambda` 太大：模型可能过分压 rejected，语言质量或多样性变差。
- `lambda` 太小：接近普通 SFT，偏好学习不明显。
- chosen/rejected 长度差异大：要做长度归一化或按框架实现解释清楚。

### GRPO

训练时通常有：

- policy model：要训练。
- reference model：计算 KL 或约束偏移。
- reward function / reward model：给每个采样回答打分。

不需要：

- 单独 value/critic model。

训练参数：

- policy 的全参或 LoRA 参数。
- reward function 一般不训练；reward model 如果有，通常在 RL 阶段冻结使用。

**面试答案：DPO、ORPO、GRPO 都能用 LoRA 吗？**
可以。LoRA 是参数高效训练方式，不是对齐算法本身。SFT、DPO、ORPO、GRPO 都可以只训练 LoRA adapter。区别在于训练目标不同：SFT 学标准答案，DPO 学相对 reference 的偏好对，ORPO 学 SFT + odds-ratio 偏好，GRPO 用组内 reward 做在线策略优化。

## 15. 蒸馏

蒸馏是用强模型 teacher 训练小模型 student。

常见：

- response distillation：蒸馏回答。
- logit distillation：拟合概率分布。
- reasoning distillation：蒸馏推理轨迹。

**面试答案：蒸馏有什么用？**  
降低部署成本，把大模型能力迁移到小模型，也可以让小模型学习大模型的 reasoning、格式遵循或领域回答风格。

## 16. 业务微调流程

如果面试官问“给你一批业务数据怎么微调”，按这个答：

1. 明确任务：问答、分类、抽取、客服、代码还是格式生成。
2. 判断是否需要微调：能否用 prompt/RAG 解决。
3. 数据清洗：去重、脱敏、纠错、统一格式。
4. 数据构造：instruction/input/output，控制长度。
5. loss mask：只对 answer 部分算 loss。
6. 选择方法：LoRA/QLoRA 优先。
7. 训练配置：rank、lr、batch、epoch、max length。
8. 评估：自动指标、人工 bad case、业务指标。
9. 部署：merge LoRA、量化、灰度、回滚。
10. 监控：幻觉、安全、格式、用户反馈。

## 17. 本章高频问答

### Q1：LoRA 和 Adapter 区别？

LoRA 在权重更新上加低秩分支，推理时可以 merge；Adapter 是在网络层中插入小模块，会增加推理路径。LoRA 更常用于当前 LLM 微调。

### Q2：DPO 为什么不需要 Reward Model？

DPO 通过数学变换把偏好优化直接写成 chosen/rejected 的 log probability 目标，不需要显式训练一个 reward model 再做 PPO。

### Q3：微调后幻觉变严重怎么办？

检查数据错误、过拟合、loss mask、训练轮数和学习率。加入拒答样本，清洗事实错误数据，结合 RAG 提供证据，用人工 bad case 和事实一致性评估。

### Q4：QLoRA 会不会无损？

不能默认无损。量化可能影响效果，必须用业务验证集评估格式、事实、幻觉、安全和边界问题。

### Q5：GRPO 和 DPO 的区别？

DPO 是离线偏好对优化；GRPO 更接近强化学习，对同一 prompt 采样多个回答，用组内相对奖励优化策略，常用于 reasoning model 语境。

### Q6：ORPO 和 DPO 的区别？

ORPO 和 DPO 都用离线 chosen/rejected 偏好对。DPO 需要 reference model，优化的是 policy 相对 reference 的偏好差；ORPO 不需要 reference model，把 SFT 的 chosen 监督和 odds-ratio 偏好惩罚合在一个 loss 里。面试记忆：DPO 是“相对参考模型更偏好 chosen”，ORPO 是“单模型 SFT + odds ratio 拉开 chosen/rejected”。

### Q7：LoRA 的 rank 越大越好吗？

不一定。rank 大容量更强，但显存、训练时间和过拟合风险也更高。面试可以说先从小 rank 做验证，如果任务复杂、数据足够、欠拟合明显，再增加 rank 或扩大注入层。

### Q8：QLoRA 的 NF4、double quantization、paged optimizer 分别解决什么？

NF4 更适合近似正态分布的权重；double quantization 继续压缩量化常数；paged optimizer 缓解训练中 optimizer state 和梯度检查点带来的显存峰值。

### Q9：DPO 的 beta 怎么理解？

beta 控制偏好优化强度。beta 大，模型更用力地区分 chosen/rejected，但可能偏离 reference；beta 小，更新更保守。它类似偏好学习里的温度/约束强度。

### Q10：GRPO 只适合数学推理吗？

不是。它常因 reasoning model 被问，但本质适合“能对多个候选回答打分”的任务，例如数学、代码、工具调用、可验证任务。关键是 reward 是否可靠。

### Q11：微调效果怎么评估？

不要只看 loss。要看指令遵循、格式、领域准确率、幻觉、安全、拒答、长尾 case、人工偏好、线上业务指标。DPO/ORPO/GRPO 还要看 reward hacking、长度偏置和输出多样性。

def lora_linear(x, weight, A, B, alpha, rank):
    base = x @ weight.T
    update = (x @ A.T) @ B.T
    return base + update * (alpha / rank)

import torch.nn.functional as F

def log_odds(sequence_logp, eps=1e-6):
    # sequence_logp 通常来自 answer token 的平均 logprob
    prob = sequence_logp.exp().clamp(eps, 1 - eps)
    return torch.log(prob) - torch.log1p(-prob)

sft_loss = -chosen_logp.mean()

### Sequence Logprob

```python
import torch
import torch.nn.functional as F

def sequence_logprob(logits, labels, mask):
    # logits: [B, T, V], labels/mask: [B, T]
    log_probs = F.log_softmax(logits, dim=-1)
    token_logp = log_probs.gather(-1, labels.unsqueeze(-1)).squeeze(-1)
    return (token_logp * mask).sum(dim=-1)
```

### GRPO Advantage

```python
def grpo_advantage(rewards, eps=1e-6):
    # rewards: [B, G], G 是每个 prompt 的采样回答数
    mean = rewards.mean(dim=-1, keepdim=True)
    std = rewards.std(dim=-1, keepdim=True)
    return (rewards - mean) / (std + eps)
```

## 19. 面试前背诵版

预训练让模型学习语言和知识，SFT 让模型学会按指令回答，RLHF/DPO/ORPO/GRPO 让模型更符合偏好。LoRA 冻结原模型，只训练低秩更新分支；QLoRA 把冻结基座量化后训练 LoRA 参数，降低显存。PPO/RLHF 通常需要 reward model 和 value/critic，并用 KL/clip 防止策略漂移。DPO 直接用 chosen/rejected 偏好对优化，不显式训练 Reward Model，但通常需要 reference model；ORPO 不需要 reference model，把 SFT 和 odds-ratio 偏好项合成一个 loss；GRPO 对同题多个回答做组内相对比较，减少 value model 依赖，常用于 reasoning model 训练讨论。业务上 RAG 用来补知识，微调用来改行为；真正上线要看数据质量、loss mask、评估集、灰度和回滚。
