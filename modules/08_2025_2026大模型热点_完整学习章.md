- DeepSeek 为什么被频繁讨论？

## 1. DeepSeek 系为什么高频

DeepSeek 相关内容在 2025-2026 面试中高频，主要因为它把几个热点绑在一起：

- MoE：用稀疏激活扩大模型容量。
- MLA：降低 KV Cache 成本。
- GRPO：降低 reasoning RL 训练复杂度。
- 推理模型：强化长思考和数学/代码推理。
- 成本效率：强调训练和推理性价比。

**面试答案：DeepSeek 为什么值得关注？**  
DeepSeek 的价值不只是模型效果，而是围绕成本效率做了系统设计：MoE 提升容量但控制每 token 计算，MLA 降低 KV Cache 压力，GRPO 降低 RL 对齐训练复杂度。这些都对应大模型落地中的核心矛盾：能力、成本、显存和推理效率。

## 2. MLA

MLA，即 Multi-head Latent Attention。

普通 MHA 的问题：

- 每个 head 都存 K/V。
- KV Cache 随层数、上下文长度和 head 数增长。
- 长上下文和高并发下显存压力大。

MQA/GQA：

- 通过共享 K/V 减少 KV Cache。

MLA：

- 把 K/V 相关信息压缩到 latent 空间。
- 推理时缓存更紧凑的 latent 表示。
- 目标是降低 KV Cache，同时尽量保持表达能力。

**面试答案：MLA 解决什么问题？**  
MLA 主要解决推理阶段 KV Cache 显存和带宽压力。它不是简单让所有 head 共享 K/V，而是把 K/V 信息压缩到低维 latent 表示，在长上下文和高并发场景下降低缓存成本。

**追问：MLA 和 GQA 区别？**  
GQA 是分组共享 K/V，减少 KV head 数；MLA 是 latent 压缩思路，目标也是降 KV Cache，但机制更进一步。

## 3. DeepSeekMoE / MoE

MoE，即 Mixture of Experts。

核心：

- 模型有很多专家。
- 每个 token 只路由到少数专家。
- 总参数量大，但每 token 激活参数量有限。

优点：

- 扩大模型容量。
- 控制计算成本。
- 专家可学习不同模式。

难点：

- 路由负载均衡。
- expert parallel 通信。
- token dispatch 和 combine 成本。
- 热门专家拥堵。
- 推理部署复杂。

**面试答案：MoE 为什么能降低计算？**  
MoE 不是降低总参数，而是稀疏激活。模型有很多专家，但每个 token 只经过 top-k 个专家，所以每 token 计算量不随总参数线性增长。

**追问：MoE 的负载均衡为什么重要？**  
如果大量 token 路由到少数专家，专家会拥堵，GPU 利用率不均，训练和推理都变慢。因此需要 load balancing loss、路由约束或工程调度。

## 4. GRPO

GRPO 常和 reasoning model 一起被问。

PPO：

- 需要 reward。
- 通常需要 value/critic。
- 训练复杂。

GRPO：

- 对同一 prompt 采样一组回答。
- 用组内相对 reward 估计 advantage。
- 减少对 value model 的依赖。

直觉：

```text
同一道题生成多个答案。
比组平均好 -> 鼓励。
比组平均差 -> 抑制。
```

**面试答案：GRPO 和 PPO 的区别？**  
PPO 通常需要 value model 估计 advantage，而 GRPO 用同一 prompt 的一组回答做相对比较，利用组内 reward 归一化估计优势，从而降低 value model 的依赖和训练复杂度。

**追问：GRPO 和 DPO 区别？**  
DPO 是离线偏好对优化，直接用 chosen/rejected；GRPO 更接近 RL，对同一 prompt 采样多个回答并用 reward 做组内相对优化。

## 5. Reasoning Model

Reasoning model 强调：

- 数学推理。
- 代码推理。
- 多步规划。
- 长思考链路。
- 自我检查。

常见训练方式：

- 高质量推理数据 SFT。
- rejection sampling。
- RL 强化正确答案和推理过程。
- 蒸馏强模型推理轨迹。

**面试答案：推理模型和普通指令模型区别？**  
普通指令模型更关注直接回答和通用对话，推理模型更强调复杂问题的多步思考、验证和答案正确性。训练上通常会引入推理轨迹、可验证奖励或 RL。

**风险：**

- 长思考增加成本和延迟。
- 推理过程不一定忠实。
- 简单问题过度思考。
- 安全和隐私风险。

## 6. MCP

MCP，即 Model Context Protocol。

它解决：

> 模型应用如何标准化连接外部工具、资源和上下文。

Function Calling：

- 模型输出工具名和参数。

MCP：

- 工具/资源/上下文如何被模型应用发现和使用。

**面试答案：MCP 为什么火？**  
Agent 应用越来越多，模型需要连接文件、数据库、浏览器、代码仓库、业务系统。如果每个工具都单独适配，成本很高。MCP 提供统一协议，降低工具接入和生态集成成本。

**安全追问：MCP 有什么风险？**

- 工具权限过大。
- prompt injection 借工具执行危险操作。
- 敏感数据泄露。
- 不可信工具返回恶意内容。

治理：

- 最小权限。
- 用户确认高风险操作。
- 工具结果视为不可信输入。
- 审计日志。
- 隔离执行环境。

## 7. Context Engineering

Prompt Engineering：

- 单次提示词怎么写。

Context Engineering：

- 模型运行时拿到什么信息。
- 检索结果怎么选。
- 工具列表怎么暴露。
- 记忆怎么写入和读取。
- 历史对话怎么压缩。
- 多 Agent 状态怎么同步。
- 成本和缓存怎么控制。

**面试答案：Context Engineering 和 Prompt Engineering 区别？**  
Prompt Engineering 是写提示词，Context Engineering 是设计模型的信息供应系统。RAG/Agent 生产化后，关键不是某一句 prompt，而是如何稳定给模型正确、精简、可控的上下文。

## 8. GraphRAG

普通 RAG：

- 按 chunk 检索。
- 适合局部事实问答。

GraphRAG：

- 抽取实体和关系。
- 构建图结构。
- 支持多跳推理和全局总结。

适合：

- 法规条款关联。
- 金融风控。
- 公司组织关系。
- 医疗知识图谱。
- 长文档全局总结。

**面试答案：GraphRAG 解决什么问题？**  
普通 RAG 对跨文档、多实体、多跳关系不够强。GraphRAG 用图结构显式表达实体关系，更适合需要关系推理和全局聚合的任务。

**局限：**

- 图构建成本高。
- 实体关系抽取可能错。
- 图更新复杂。
- 查询规划更难。

## 9. RAG 评估热点

2025-2026 RAG 面试越来越爱问评估，而不是只问流程。

要分层：

检索：

- Recall@k。
- MRR。
- NDCG。

上下文：

- context relevance。
- context precision。
- 是否包含答案。

生成：

- faithfulness。
- answer relevance。
- citation correctness。
- hallucination rate。

业务：

- 解决率。
- 转人工率。
- 成本。
- 延迟。

**面试答案：为什么 RAG 评估要分层？**  
最终答案错了，可能是没召回、rerank 排错、证据冲突、prompt 没约束或模型幻觉。分层评估能定位问题在哪个环节。

## 10. Agent 评估热点

Agent 不看 demo，要看任务完成。

指标：

- task success rate。
- tool call accuracy。
- argument accuracy。
- average steps。
- cost per task。
- latency。
- human intervention rate。
- safety violation rate。

测试集：

- 正常任务。
- 工具失败任务。
- 权限敏感任务。
- 长任务。
- 多轮任务。
- 恶意输入。

**面试答案：Agent 怎么评估？**  
构造固定任务集，记录每次工具调用 trace，标注工具选择、参数、结果使用是否正确，最终看任务完成率、步骤数、成本和安全违规率。

## 11. 多模态大模型

常见结构：

```text
图像/音频 encoder -> projector/adapter -> LLM
```

视觉模型把图像编码成 patch/token 表示，通过 projector 对齐到 LLM hidden space，再让 LLM 处理。

**面试答案：VLM 怎么把图像接到 LLM？**  
通常用视觉 encoder 提取图像特征，再用线性层、MLP 或 Q-Former 等 projector 把视觉特征映射到语言模型的 embedding 空间，使 LLM 能把图像 token 当上下文处理。

常见任务：

- 图文问答。
- OCR。
- 图像理解。
- 视频理解。
- 多模态 Agent。

难点：

- 视觉和语言对齐。
- OCR 细粒度识别。
- 多图/视频长上下文。
- 幻觉。
- 坐标定位。

更完整的 VLM 面试复习见：[12_VLM多模态面试.md](../12_VLM多模态面试.md)。如果追到高分辨率 OCR、AnyRes/tiling、dynamic resolution、grounding、视频采样和 GUI Agent，看进阶专题：[40_VLM进阶_高分辨率OCR_Grounding_VideoAgent面试.md](../40_VLM进阶_高分辨率OCR_Grounding_VideoAgent面试.md)。

## 12. 长上下文热点

长上下文能力来自多方面：

- 位置编码扩展。
- 数据训练。
- attention 优化。
- KV Cache 管理。
- 上下文压缩。

常见问题：

- lost in the middle。
- 成本高。
- 延迟高。
- KV Cache 大。
- 无关上下文干扰。

**面试答案：长上下文是不是越长越好？**  
不是。长上下文能容纳更多信息，但会增加成本、延迟和干扰。生产系统通常会结合 RAG、摘要、上下文压缩和缓存，只给模型必要信息。

## 14. 安全与对齐热点

常见风险：

- 幻觉。
- prompt injection。
- jailbreak。
- 数据泄露。
- 工具滥用。
- 有害内容。
- 偏见。

治理：

- 安全 SFT / DPO。
- 内容审核。
- 工具权限隔离。
- 检索证据约束。
- 拒答策略。
- 红队测试。
- 日志审计。

**面试答案：RAG/Agent 如何防 prompt injection？**  
检索文档和网页内容都要当作不可信输入，不能让它覆盖系统指令；工具调用要做权限控制和参数校验；高风险操作需要用户确认；记录 trace 便于审计。

## 15. 评测 Benchmark 与安全治理热点

2025-2026 的热点题越来越强调“怎么证明系统可靠”。不要只会说模型榜单，要会区分四类评测：

| 类型 | 看什么 | 例子 |
| --- | --- | --- |
| 基础能力 benchmark | 模型知识、推理、数学、代码 | MMLU、GSM8K、HumanEval 等 |
| Agent benchmark | 工具使用、搜索、浏览、多步任务 | GAIA、AgentBench、SWE-bench 类任务 |
| 应用私有 eval | 业务真实问题是否解决 | RAG 问答集、客服工单、工具调用任务集 |
| 安全红队 eval | 是否越权、泄露、被注入攻击 | prompt injection、jailbreak、数据泄露、过度代理 |

**面试答案：为什么公开 benchmark 不够？**  
公开 benchmark 能看通用能力，但不等于你的业务可靠。RAG/Agent 系统依赖数据、工具、权限、prompt、上下文和日志链路，必须做私有评估集、线上灰度和 bad case 回流。

### GAIA 这类 Agent benchmark 问什么

GAIA 代表一类“通用助手任务”评测，强调：

- 多步推理。
- 搜索或浏览。
- 工具使用。
- 多模态信息处理。
- 最终答案可验证。

**面试怎么用：**  
不要背榜单分数，重点说它提醒我们 Agent 不能只看聊天能力，要看真实任务完成率、工具链稳定性和可验证答案。

### OWASP LLM Top 10 怎么落到项目

常见高频风险可以这样讲：

- Prompt Injection：用户或检索内容试图覆盖系统指令。
- Sensitive Information Disclosure：模型泄露隐私、密钥、内部数据。
- Supply Chain / Data Poisoning：模型、依赖、数据源或知识库被污染。
- Insecure Output Handling：把模型输出直接当 SQL、命令、HTML 执行。
- Excessive Agency：Agent 权限过大，自动执行高风险操作。
- Vector and Embedding Weaknesses：向量库被投毒、相似度召回被攻击。
- Unbounded Consumption：超长输入、死循环工具调用导致成本爆炸。

**面试答案：如何给 LLM 应用做安全设计？**  
把模型输入、检索内容、工具返回都当不可信数据；系统指令和工具权限隔离；高风险工具最小权限和人工确认；输出进入业务系统前做校验；记录 trace、审计和告警；用红队集持续测试 prompt injection、越权、泄露和成本攻击。

更完整的安全评测、红队和上线治理复习见：[13_LLM安全评测与红队.md](../13_LLM安全评测与红队.md)。

### NIST AI RMF / GenAI Profile 怎么回答

如果面试官问“AI 风险怎么体系化管理”，可以不背条文，但要知道框架思路：

```text
治理 -> 映射风险 -> 度量风险 -> 管理风险
```

在大模型项目里对应：

- 治理：谁负责模型、数据、安全、上线审批。
- 映射：识别业务场景、用户、数据、工具、风险边界。
- 度量：构建 eval、红队、日志、人工评审和线上指标。
- 管理：权限、拒答、降级、灰度、回滚、审计、持续改进。

**面试答案：安全和效果冲突怎么办？**  
先按业务风险分级。低风险问答可以更开放，高风险操作必须保守；涉及隐私、资金、医疗、法律、删除/写入等动作，要人工确认或只给建议不自动执行。安全不是单个 prompt，而是权限、评估、监控和流程。


## 17. 本章高频问答

### Q1：DeepSeek 的几个关键词怎么串起来？

MoE 提升容量，MLA 降低 KV Cache 成本，GRPO 降低 reasoning RL 复杂度，整体指向更高性价比的大模型训练和推理。

### Q2：MLA 和 GQA 谁更好？

不能简单说谁更好。GQA 是工程上成熟的 K/V head 共享折中，MLA 是更激进的 latent 压缩设计。选择取决于模型结构、训练策略和推理系统。

### Q3：MCP 是不是取代 Function Calling？

不是。Function Calling 是模型输出工具调用意图的方式，MCP 是工具和上下文接入模型应用的协议。两者可以配合。

### Q4：Context Engineering 为什么重要？

生产级 LLM 应用的质量不只由模型决定，还取决于给模型什么上下文、上下文是否正确、是否太长、工具是否合适、历史和记忆是否污染。

### Q5：GraphRAG 什么时候不适合？

普通事实问答、文档规模小、关系不复杂时，普通 RAG 更简单。GraphRAG 构图、更新和查询都更复杂，不要为了热点硬上。

### Q6：公开 benchmark 分数高，为什么业务效果还可能差？

公开 benchmark 测的是通用能力，业务系统还受数据覆盖、检索质量、prompt、工具权限、延迟、成本、安全策略和用户分布影响。上线前必须做私有 eval 和灰度监控。

### Q7：Agent 的最大安全风险是什么？

不是模型说错一句话，而是“说错后还能调用工具执行”。所以要限制工具权限、校验参数、隔离环境、高风险人工确认、记录 trace，并设置最大步数和成本上限。

### Q8：向量库也会有安全问题吗？

会。知识库可能被投毒，恶意文档可能通过相似度召回进入上下文，诱导模型泄露或执行错误操作。要做数据准入、来源标记、权限过滤、检索结果清洗和引用审计。

### Q9：如何设计一个 LLM 应用的私有 eval？

从真实日志和业务专家样例里抽问题，标注参考答案、允许证据、拒答边界和安全标签；覆盖正常、长尾、恶意、工具失败、权限敏感场景；每次改模型、prompt、检索、工具 schema 都跑一遍。

## 19. 面试前背诵版

2025-2026 大模型热点本质上都围绕能力、成本、评估和工程稳定性。DeepSeek 高频是因为 MoE、MLA、GRPO 共同指向高性价比模型：MoE 扩容量，MLA 降低 KV Cache，GRPO 简化 reasoning RL。MCP 解决工具和上下文标准化接入，Context Engineering 解决模型运行时信息供应，GraphRAG 解决普通 RAG 难以处理的多跳关系问题。RAG/Agent 进入生产后，公开 benchmark 不够，还要做私有 eval、红队、安全权限、trace 审计和 bad case 回流。热点题不要只背名词，要说清问题、机制、收益、代价、风险和项目落地。
