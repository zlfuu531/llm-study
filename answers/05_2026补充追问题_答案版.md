# 答案版 05：2026 补充追问 40 题

这一份对应 `03_高频题单100题.md` 的第 101-140 题。它不是替代完整学习章，而是帮你在模拟面试前快速组织“30 秒结论 + 2 分钟展开 + 项目落点”。

## 101. 如果让你从数学角度解释 Attention 的本质，你会怎么说？

**面试答案：**  
Attention 本质是根据 query 和 key 的相似度，对 value 做加权求和。公式是：

```text
Attention(Q,K,V)=softmax(QK^T/sqrt(d_k))V
```

`QK^T` 计算每个 token 对其他 token 的相关性，softmax 变成权重，最后对 `V` 加权聚合信息。

**追问落点：**  
它不是简单平均，而是内容相关的动态加权；复杂度主要来自 `[T,T]` attention score。

## 102. FlashAttention 和 PagedAttention 分别优化什么，为什么不能混为一谈？

**面试答案：**  
FlashAttention 优化 attention 计算本身，通过分块和 IO-aware 设计减少 HBM 读写和中间 attention 矩阵显存。PagedAttention 优化推理时 KV Cache 的显存管理，把 KV Cache 分成 block，减少碎片和浪费。

**一句话：**  
FlashAttention 管“怎么算 attention 更省”，PagedAttention 管“推理缓存怎么放更省”。

## 103. Lost in the Middle 是什么，长上下文里怎么缓解？

**面试答案：**  
Lost in the Middle 指关键信息放在长上下文中间时，模型更容易忽略，表现为开头和结尾信息利用更好，中间信息利用差。

缓解方法：

- RAG 只放最相关证据，减少无关上下文。
- 把关键证据放在 prompt 更显眼的位置。
- 做 context compression / rerank。
- 分段回答或 map-reduce 汇总。
- 用长上下文评测集专门测。

## 104. Chunked Prefill 解决什么问题，会牺牲什么？

**面试答案：**  
Chunked Prefill 把长 prompt 的 prefill 拆成小块，避免一个长请求长时间占用 GPU，改善短请求 TTFT 和尾延迟。

代价：

- 调度更复杂。
- prefill/decode 资源分配需要调参。
- 参数不合适时可能影响总吞吐。

**项目表达：**  
它是在吞吐、TTFT、TPOT、公平性之间做工程权衡。

## 105. Prefix Cache 什么时候命中率高，什么时候收益不明显？

**面试答案：**  
命中率高的场景：系统 prompt 固定、RAG 模板固定、多轮对话前缀相同、同一文档被多用户反复查询。

收益不明显的场景：

- 每个请求 prompt 都差异很大。
- 前缀很短，prefill 成本本来不高。
- cache 管理开销超过复用收益。
- 上下文包含用户个性化敏感内容，不能共享。

## 106. DPO 训练时 chosen/rejected 长度差异大怎么办？

**面试答案：**  
DPO 常用 sequence logprob，如果直接求和，长回答会天然累积更多 logprob 惩罚或奖励，可能产生长度偏置。

处理方式：

- 数据构造时控制 chosen/rejected 长度差异。
- 明确使用 sum 还是 mean logprob，并和评估保持一致。
- 加长度惩罚或长度归一化。
- 分析模型是否学会偏向短答或长答。

## 107. DPO 的 `beta` 过大或过小会有什么风险？

**面试答案：**  
`beta` 控制偏好优化强度。过大时模型会很激进地区分 chosen/rejected，可能偏离 reference，出现过拟合、格式变坏或安全退化；过小时更新太保守，偏好学习不明显。

**面试抓手：**  
`beta` 类似“偏好信号力度”和“离 reference 多远”的权衡。

## 108. GRPO 的 reward 怎么设计，reward 错了会怎样？

**面试答案：**  
GRPO 依赖同题多回答的组内 reward。reward 可以来自规则、单元测试、答案匹配、LLM judge 或 reward model。

reward 错了会导致：

- reward hacking。
- 鼓励错误推理格式。
- 模型学会钻规则空子。
- 输出变长、啰嗦或模式化。

**项目表达：**  
GRPO 适合 reward 可验证的任务，比如数学、代码、工具调用；开放闲聊类任务要更谨慎。

## 109. QLoRA 里的 NF4、double quantization、paged optimizer 分别解决什么？

**面试答案：**

- NF4：适合近似正态分布权重的 4-bit 量化格式。
- double quantization：继续量化量化常数，进一步省显存。
- paged optimizer：缓解训练中 optimizer state 的显存峰值。

**一句话：**  
QLoRA 的核心是把冻结基座低比特存储，同时用较高精度训练 LoRA 参数。

## 110. Packing 训练时 attention mask 和 loss mask 写错会造成什么后果？

**面试答案：**  
packing 把多个短样本拼进一个长序列。如果 attention mask 错了，不同样本可能互相看见，造成数据污染；如果 loss mask 错了，模型可能学习 prompt、padding 或用户角色 token，导致复述问题、格式混乱和角色错乱。

**排查：**  
看训练样本解码、检查 label 中 `-100` 位置、确认 EOS/role token 和样本边界。

## 111. RAG 召回率低，你如何从 query、chunk、embedding、索引、rerank 分层排查？

**面试答案：**

1. 先确认 gold 文档是否在知识库里。
2. 看 query 是否需要 rewrite，是否有简称、错别字、领域词。
3. 看 chunk 是否切碎、过大或丢标题 metadata。
4. 看 embedding 是否适合语言和领域。
5. 看 ANN 索引参数是否导致近邻召回损失。
6. 看 BM25/dense hybrid 是否覆盖精确词。
7. 看 rerank 是否把正确片段排下去了。

**一句话：**  
RAG 问题要分层定位，不能只改 prompt。

## 112. RAG 评估集怎么构造，gold 文档和 reference answer 怎么标？

**面试答案：**  
从真实日志、业务 FAQ、专家样例和 bad case 中抽 query。每条样本至少标：

- 用户问题。
- 应该命中的 gold 文档或段落。
- 参考答案。
- 是否允许拒答。
- 安全/权限标签。
- 可能的同义问法。

评估时分别看 Recall@k、context precision、faithfulness、引用正确率和业务解决率。

## 113. 如果检索结果被恶意 prompt injection 污染，系统怎么防？

**面试答案：**  
把检索内容当不可信输入，不能让文档内容覆盖系统指令。治理包括：

- system prompt 明确外部内容不具备指令权限。
- 检索内容做来源标记和隔离。
- 工具调用只由受控 schema 和权限系统决定。
- 高风险操作人工确认。
- 对输出和工具参数做校验。
- 记录 trace，支持审计和回滚。

## 114. GraphRAG 构图错了或实体抽取错了，会怎样影响答案？

**面试答案：**  
GraphRAG 依赖实体和关系。如果抽错实体、合并错实体或关系边错了，检索会沿错误路径扩散，导致多跳推理错误、引用错误和全局总结偏差。

处理方式：

- 实体消歧。
- 关系置信度。
- 人工校验高价值实体。
- 图谱版本管理。
- 图检索结果和原文证据一起给模型。

## 115. Agent Harness 包含哪些工程模块，为什么只写 prompt 不够？

**面试答案：**  
Agent Harness 是包住模型的工程外壳，包括任务理解、状态管理、工具注册、权限控制、参数校验、执行器、记忆、trace、评估、失败恢复、成本控制和人工确认。

只写 prompt 不够，因为模型会选错工具、填错参数、误读 observation、死循环、越权或成本失控。生产级 Agent 要可观测、可约束、可回滚。

## 116. Agent 的 memory 怎么设计，短期记忆、长期记忆和检索记忆怎么分？

**面试答案：**

- 短期记忆：当前会话状态、最近几轮对话、当前任务变量。
- 长期记忆：用户偏好、历史任务摘要，需要权限和过期策略。
- 检索记忆：外部知识库、文档、历史案例，通过 RAG 召回。

**风险：**  
记忆会污染上下文，也可能泄露隐私，所以要有写入规则、删除机制和权限隔离。

## 117. MCP 的 Host、Client、Server、Tools、Resources、Prompts 分别是什么？

**面试答案：**

- Host：模型应用宿主，如 IDE、Agent 平台。
- Client：Host 里连接某个 MCP Server 的组件。
- Server：暴露能力的服务。
- Tools：可执行动作，如查库、调用 API。
- Resources：可读取上下文，如文件、schema、文档。
- Prompts：可复用提示模板。

**一句话：**  
MCP 解决模型应用如何标准化发现和使用外部工具、资源、提示模板。

## 118. Function Calling 参数错了，系统应该让模型重试还是直接失败？

**面试答案：**  
看错误类型。

- 可修复错误：类型错、缺字段、格式错，可以把错误信息返回给模型重试。
- 不可修复错误：权限不足、危险操作、业务约束不允许，应该拒绝或让用户补充。
- 多次失败：终止、降级或人工介入。

**工程要点：**  
重试要有次数上限，工具调用要幂等，高风险操作要确认。

## 119. Multi-Agent 为什么可能比 Single-Agent 更差？

**面试答案：**  
Multi-Agent 会增加通信成本、延迟、状态同步难度和错误传播风险。多个弱 Agent 互相讨论不一定比一个强 Agent 稳定，还可能产生重复工作和调试困难。

适合 Multi-Agent 的条件：

- 任务确实可分工。
- 角色边界清晰。
- 有可验证中间产物。
- 并行收益大于协调成本。

## 120. Human-in-the-loop 适合放在哪些 Agent 环节？

**面试答案：**  
适合放在高风险或不可逆环节：

- 执行写数据库、发邮件、下单、删除文件前。
- 权限升级前。
- 模型信心低或证据不足时。
- 多次工具失败后。
- 安全/合规敏感输出前。

**一句话：**  
HITL 不是让人每步都审，而是在风险最高的决策点兜底。

## 121. 线上 TTFT 变高，你会按什么链路排查？

**面试答案：**  
TTFT 主要受排队、prefill 和前置链路影响。排查：

1. 网关、鉴权、限流是否排队。
2. tokenizer 和 prompt 构造是否变慢。
3. prompt 是否变长，RAG 是否塞了太多上下文。
4. prefix cache 是否没命中。
5. prefill batch 调度是否被长请求阻塞。
6. GPU 是否满载，队列长度是否上升。

## 122. 线上 TPOT 变高，你会按什么链路排查？

**面试答案：**  
TPOT 是每个输出 token 的耗时，常和 decode 阶段有关。排查：

- batch 是否过大。
- decode kernel 是否退化。
- KV Cache 是否太长、读写带宽瓶颈。
- 是否长输出请求变多。
- GPU 利用率、显存带宽、cache 命中是否异常。
- 是否开启了低效的安全过滤或后处理。

## 123. 显存够但 tokens/s 上不去，可能是什么瓶颈？

**面试答案：**  
显存够不代表吞吐高。可能瓶颈包括：

- batch 太小，GPU 利用率低。
- decode 受内存带宽限制。
- KV Cache 访问低效。
- tokenizer、网络、后处理拖慢。
- 调度策略导致 GPU 空转。
- 模型结构或 kernel 没优化。

**排查原则：**  
看 GPU 利用率、显存、队列、TTFT、TPOT、tokens/s，不要只看显存。

## 124. 量化后数学/代码能力下降，怎么评估和补救？

**面试答案：**  
评估要单独看数学、代码、格式遵循、幻觉、安全和业务 case，不能只看平均分。

补救：

- 换更高比特，如 INT4 -> INT8/BF16。
- 使用更好的量化方法或校准集。
- 对敏感任务路由到未量化大模型。
- 量化后再做少量校准/微调。
- 保留关键层高精度。

## 125. 模型路由怎么设计：什么时候用大模型，什么时候用小模型？

**面试答案：**  
模型路由按任务难度、风险、成本和延迟选模型。

小模型适合：

- 分类、改写、抽取、简单问答。
- 低风险高频请求。
- draft model。

大模型适合：

- 复杂推理。
- 多工具规划。
- 高价值用户。
- 安全敏感或失败代价高的任务。

路由信号可以来自规则、分类器、置信度、用户等级、上下文长度和历史失败率。

## 126. 模型网关要做哪些事情？

**面试答案：**  
模型网关是模型调用入口，常做：

- 鉴权和租户隔离。
- 路由不同模型。
- 限流、熔断、重试。
- prompt 模板版本管理。
- 日志、监控、成本统计。
- 缓存和降级。
- 安全审核和敏感信息脱敏。
- 灰度发布和回滚。

## 127. 如何设计大模型服务的限流、降级和熔断？

**面试答案：**

- 限流：按用户、租户、接口、token 预算限制。
- 降级：缩短输出、降低上下文、关闭 rerank、路由小模型、返回缓存。
- 熔断：当错误率、P95、队列长度、GPU 利用率异常时，暂停低优先级请求。

**面试补充：**  
大模型服务要按 token 成本限流，不只按 QPS。

## 128. RAG + 长上下文同时使用时，如何控制成本？

**面试答案：**  
不要把所有检索结果都塞进长上下文。控制方式：

- query rewrite 提高召回精度。
- rerank 只保留 top 证据。
- context compression。
- 去重和合并相邻 chunk。
- 引用必要片段，不塞全文。
- prefix cache 复用固定模板。
- 设置 token budget。

## 129. Speculative Decoding 接受率低时为什么可能不加速？

**面试答案：**  
Speculative Decoding 靠小模型草稿被大模型接受来减少大模型调用。如果接受率低，大模型频繁拒绝并回退，还多了小模型生成和验证开销，可能不加速甚至变慢。

**关键因素：**  
draft model 要足够接近 target model，任务分布也要匹配。

## 130. KV Cache 量化有什么收益和风险？

**面试答案：**  
收益是降低 KV Cache 显存和带宽压力，提高长上下文和高并发承载能力。

风险：

- attention 质量下降。
- 长上下文答案变差。
- 数学/代码/细节定位能力受影响。
- 量化 kernel 和硬件支持不佳时收益有限。

上线前要用长上下文、RAG、多轮对话和业务 case 评估。

## 131. 为什么 CrossEntropyLoss 通常接收 logits，而不是 softmax 后的概率？

**面试答案：**  
框架通常把 `log_softmax` 和 NLLLoss 合在一起实现，直接输入 logits 数值更稳定，避免 softmax 后概率下溢或重复计算。手动 softmax 再传入 CrossEntropyLoss 是常见错误。

## 132. softmax + cross entropy 的梯度为什么是 `p - y`？

**面试答案：**  
softmax 把 logits 变成概率 `p`，交叉熵是 `-sum y log p`。二者合并求导后，logits 的梯度简化为：

```text
dL/dz = p - y
```

正确类如果概率低，梯度会把它往上推；错误类概率高，梯度会把它往下压。

## 133. AdamW 为什么要解耦 weight decay？

**面试答案：**  
Adam 的自适应学习率会改变 L2 正则加入梯度后的实际衰减效果。AdamW 把 weight decay 从梯度更新中解耦，直接对权重做衰减，更稳定，也更符合正则化直觉。

常见实践：bias、LayerNorm/RMSNorm 的 scale 通常不做 weight decay。

## 134. ROC-AUC 和 PR-AUC 怎么选？

**面试答案：**  
ROC-AUC 看正负样本整体排序能力；PR-AUC 更关注正类检出质量。

如果类别不平衡很严重、正样本稀少、误报漏报成本敏感，PR-AUC 更有意义。普通排序二分类可以看 ROC-AUC，但最终还要结合业务阈值下的 precision/recall。

## 135. PPL 低为什么不等于模型对话体验好？

**面试答案：**  
PPL 衡量 next-token 预测能力，不直接衡量指令遵循、事实性、安全、推理、工具调用和用户满意度。一个模型 PPL 低，仍可能幻觉、拒答差、格式不稳或不会调用工具。

LLM 应用还要看任务成功率、人工偏好、faithfulness、引用正确率和业务指标。

## 136. tokenizer 词表为什么不能随便换？

**面试答案：**  
tokenizer 决定文本如何映射到 token id，而 embedding 和 LM head 都和 token id 绑定。随便换 tokenizer，会让原模型学到的 token 表示失效，输入分布和输出词表都变了。

除非重新训练、扩词表适配或做 embedding 初始化，否则不能随便换。

## 137. BPE、WordPiece、SentencePiece 的核心区别是什么？

**面试答案：**

- BPE：从字符/字节开始合并高频 pair。
- WordPiece：基于似然/子词收益选择合并，BERT 系常见。
- SentencePiece：直接在原始文本上建模，不依赖空格预分词，多语言友好。

共同目标是在词表大小、序列长度和 OOV 之间折中。

## 138. Encoder-only、Decoder-only、Encoder-Decoder 适合什么任务？

**面试答案：**

- Encoder-only：理解、分类、匹配、检索表征，如 BERT。
- Decoder-only：自回归生成、对话、通用 LLM，如 GPT。
- Encoder-Decoder：输入输出结构差异明显的任务，如翻译、摘要、T5/BART。

现代通用大模型多用 Decoder-only，因为生成接口统一、扩展性强。

## 139. 一个 RAG/Agent 项目 8 分钟深挖版应该讲哪些证据？

**面试答案：**  
按证据包讲：

1. 业务场景和成功标准。
2. baseline 是什么。
3. 系统架构和你的负责模块。
4. 数据来源、清洗、权限和脱敏。
5. 核心技术选择和替代方案。
6. 评估集、指标和 ablation。
7. bad case 类型和定位链路。
8. 上线监控、灰度、回滚和成本。
9. 当前局限和下一步。

**关键：**  
项目要讲证据，不讲“我用了某框架”。

## 140. 如何为 LLM 应用设计私有 eval、红队测试、灰度和回滚？

**面试答案：**  
私有 eval 从真实日志、专家样例和 bad case 构造，覆盖正常、长尾、恶意、权限敏感和工具失败场景。每条样本标参考答案、gold evidence、安全标签和是否允许拒答。

红队测试重点看 prompt injection、越权工具调用、数据泄露、有害输出、成本攻击。

灰度时按用户、流量、场景逐步放量，监控成功率、投诉、P95、成本、安全违规。

回滚要能回到旧模型、旧 prompt、旧索引、旧工具 schema，并保留 trace 方便定位。
