# 答案版 11：大模型评测与实验设计

对应 `03_高频题单100题.md` 的 241-260 题。

## 241. 为什么公开 benchmark 不够？

公开 benchmark 能做通用能力筛选，但不能证明业务可用。它可能被训练数据污染，也不覆盖你的业务文档、权限、工具、延迟和安全风险。面试里要说：先用公开榜单选模型，再用私有 eval 做上线证据，按业务分桶、固定数据版本、对比 baseline、分析 bad case。

## 242. 大模型评测应该按哪些维度拆？

先按对象拆：基座模型、指令模型、RAG、Agent、Function Calling、安全、推理服务和线上产品。再按指标拆：组件指标、端到端指标、业务指标。比如 RAG 要看 Recall@k、faithfulness 和引用正确率；Agent 要看 task success、tool accuracy 和安全违规率；服务要看 TTFT、TPOT、P95 和成本。

## 243. MMLU-Pro、GPQA、SWE-bench、MT-Bench、Chatbot Arena 分别看什么？

MMLU-Pro 看更难的多学科知识和推理，GPQA 看高难科学问答，SWE-bench 看真实软件 issue 修复，MT-Bench 看多轮对话质量，Chatbot Arena 看人类偏好的对战排名。它们是能力维度参考，不是业务上线证明。

## 244. 私有 eval set 怎么构造？

从业务目标出发，抽取真实 query、历史工单、文档问答、工具任务、投诉和 bad case；按任务、难度、风险、来源分桶。每条样本记录 query、gold evidence、reference answer、rubric、risk tags 和 split。valid 用来调参，test 用来最终报告，不能把 test 泄露到 prompt、SFT 或 few-shot。

## 245. Reference answer 和 rubric 怎么写？

reference answer 给理想答案，rubric 给判分规则。开放生成不能只写“回答正确”，要写必须包含什么、不能包含什么、证据不足时是否拒答、引用是否必须支持答案、安全边界是什么。rubric 越明确，人工和 LLM judge 的一致性越高。

## 246. LLM-as-judge 怎么设计？

给 judge 输入用户问题、模型回答、必要的 gold answer 或 evidence，再用明确 rubric 做 pointwise 或 pairwise 评估。要固定 judge 模型、prompt、temperature 和脚本版本。为了控制偏置，A/B 顺序要随机或交换，多 judge 或多次评估取聚合，并用人工抽检校准。

## 247. Pointwise 和 pairwise 评测有什么区别？

Pointwise 是对单个回答打分，适合规则明确、要看绝对达标率的任务。Pairwise 是比较两个回答谁更好，适合模型 A/B 对比和偏好评测。Pairwise 往往更稳定，但要控制位置偏置和成本；pointwise 实现简单，但绝对分数容易漂移。

## 248. Win rate、Elo、Bradley-Terry 怎么理解？

Win rate 是 A 相比 B 的胜率，常把 tie 算半胜。Elo 和 Bradley-Terry 都是用大量 pairwise 对战估计全局能力分。BT 常写成 `P(A beats B)=exp(s_A)/(exp(s_A)+exp(s_B))`。面试重点是：排名要看置信区间，小差距不一定显著，业务分布变化后榜单排名不一定可靠。

## 249. 怎么评估幻觉？

先定义幻觉：答案声称了无法由证据支持的内容。RAG 场景看 faithfulness、citation accuracy、无答案拒答率；通用问答看 factuality、人工抽检和带证据 judge。排查时先看 gold 文档是否召回，召回了还错再查 prompt、context packing、模型能力和拒答策略。

## 250. RAG 评估为什么要拆检索和生成？

因为最终答案错可能有两类原因：检索没拿到证据，或者证据拿到了但模型生成错。检索层看 Recall@k、MRR、NDCG；生成层看 faithfulness、answer relevance、引用正确率和拒答正确率。拆开后才能知道该改 chunk、embedding、rerank，还是改 prompt 和模型。

## 251. Agent 评估要看哪些指标？

Agent 要看任务结果和过程轨迹。任务层看 task success、解决率、平均步骤数、成本和耗时；工具层看 tool selection accuracy、argument accuracy、schema validity、执行成功率；安全层看越权调用、循环调用、敏感信息泄露和需要人工确认时是否停下。

## 252. Function Calling 怎么评测？

每条样本要有 user query、可用工具、expected tool、expected args、forbidden tools 和 success criteria。指标上看函数名准确率、参数 exact match 或 F1、schema 合法率、多轮调用成功率、工具执行成功率和越权率。不能只看最后回复，因为工具轨迹错了也可能生成一个看似正确的文本。

## 253. Ablation 实验怎么设计？

先定 baseline，再一次只加一个模块。比如 RAG：vector only、+BM25、+rerank、+query rewrite、+citation/refusal prompt。每一步固定数据、模型、prompt 和采样参数，看指标、成本、延迟和 bad case。这样才能证明提升来自哪个模块。

## 254. 模型上线前 regression eval 看什么？

看新模型相对当前线上版本是否退化。核心业务分桶要提升或持平，安全违规不能升高，延迟和成本不能超预算，高风险 bad case 不能复现。通过离线 regression 后再灰度，线上监控转人工率、投诉率、错误率、P95/P99 和成本。

## 255. 线上 A/B 和离线 eval 有什么关系？

离线 eval 是上线前门禁，A/B 是真实用户验证。离线能快速排除明显差方案，但不能完全代表用户行为；A/B 能看解决率、满意度、转化、转人工和投诉，但成本和风险更高。正确做法是先离线通过，再小流量灰度，最后扩大 A/B。

## 256. 小样本评测差 1%-2% 能说明提升吗？

不能直接说明。要看样本量、方差、置信区间、win/tie/loss 分布和分桶结果。生成任务还要人工抽检，确认不是 judge 偏置或长度偏置。对安全和合规指标，不能只看平均分，要设硬门槛。

## 257. 怎么防止评测集污染？

评测集不能进入训练、SFT、DPO、prompt few-shot 或调参流程。数据划分要按用户、文档、时间、来源或业务实体隔离，避免同源泄露。公开 benchmark 要做 decontamination，私有 eval 要版本化，记录样本来源、更新时间和使用范围。

## 258. Judge 和人工不一致怎么办？

先抽样分析不一致类型：rubric 不清、judge 忽略事实、偏好长答案、对格式过敏、人工标准不统一。然后改 rubric、加入 gold evidence、交换 A/B 位置、多 judge 聚合，并重新计算 judge-human agreement。关键样本以人工或规则为准。

## 259. Eval 结果怎么版本化？

记录 eval set 版本、样本来源、prompt、模型版本、judge 版本、采样参数、评测脚本 commit、指标结果和 bad case 标签。否则新旧结果不可比，线上退化也难以定位是模型、数据、prompt、retriever 还是 judge 变了。

## 260. 大模型评测项目 8 分钟深挖版讲什么？

按“背景 -> 风险 -> 数据集 -> 指标 -> baseline -> ablation -> bad case -> 上线门禁 -> 线上反馈”讲。举例：为知识库问答构造 200 条私有 eval，标注 gold 文档和参考答案；比较 vector、hybrid、rerank、query rewrite；看 Recall@k、MRR、faithfulness、引用正确率、P95 和成本；上线前做 regression，灰度后把投诉和 bad case 回流到 eval set。
