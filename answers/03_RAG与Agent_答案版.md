# 答案版 03：RAG、Agent、工具调用与 MCP

这一份对应 `03_高频题单100题.md` 的第 41-75 题。

## 41. RAG 是什么，解决什么问题？

**面试答案：**  
RAG 是 Retrieval-Augmented Generation，检索增强生成。它先从外部知识库检索相关证据，再把证据放进 prompt，让模型基于上下文回答。它解决的是模型参数知识不够新、不够私有、不可溯源的问题。

**项目直觉：**  
模型负责表达和推理，检索系统负责提供事实证据。

## 42. RAG 和微调怎么选？

**面试答案：**  
RAG 更适合补知识，尤其是企业私有知识、频繁更新知识、需要引用来源的问答。微调更适合改变模型行为，比如输出格式、任务风格、领域表达方式。实际项目里常常组合：RAG 提供知识，SFT/LoRA 提升指令遵循和领域风格。

## 43. RAG 的离线建库和在线问答流程是什么？

**面试答案：**

离线：

1. 文档采集。
2. 清洗解析。
3. chunk 切分。
4. embedding。
5. 建向量索引/关键词索引。

在线：

1. 用户 query。
2. query rewrite。
3. dense/sparse/hybrid 召回。
4. rerank。
5. 拼接上下文。
6. LLM 生成。
7. 引用和后处理。
8. 日志与评估。

## 44. chunk size 怎么设置？

**面试答案：**  
chunk size 要根据文档结构、问题类型和模型上下文长度调。太小会丢上下文，太大语义稀释、召回不准、成本高。通常先按标题/段落/语义单元切，再结合 overlap 和 Recall@k、答案命中率、人工 bad case 调整。

**一句话：**  
chunk 的目标是包含能回答问题的最小完整语义单元。

## 45. overlap 有什么作用？

**面试答案：**  
overlap 用来缓解切分边界问题。答案可能跨两个 chunk，如果完全硬切，关键上下文会断掉。适当 overlap 能保留边界信息，提高召回完整性。

**注意：**  
overlap 太大也会增加索引冗余、召回重复和成本。

## 46. 文档结构化解析有什么难点？

**面试答案：**  
难点包括 PDF 版式、表格、图片、标题层级、页眉页脚、代码块、脚注、跨页内容和扫描件 OCR。解析不好会导致 chunk 语义混乱，embedding 和召回都变差。

**项目话术：**  
我会保留标题层级、来源、页码、表格结构等 metadata，因为这些信息对召回、引用和后处理都很重要。

## 47. embedding 模型怎么选？

**面试答案：**  
看语言、领域、文本长度、部署成本和评测效果。中文业务优先选中文/多语言效果好的 embedding；领域文本要用自己的 query-doc 测试集评估 Recall@k。还要考虑向量维度、推理速度、是否支持长文本。

**不要只说：**  
“我用了某某 embedding 模型。”要补一句：“我用业务问题集比较过召回效果。”

## 48. dense retrieval 和 sparse retrieval 的区别是什么？

**面试答案：**  
dense retrieval 用 embedding 做语义召回，能找到语义相近但词面不同的内容。sparse retrieval 如 BM25 基于关键词和词频，适合精确词、编号、专有名词、错误码。两者互补，企业 RAG 常用 hybrid retrieval。

## 49. 为什么要混合召回？

**面试答案：**  
因为单独向量召回容易漏掉关键词精确匹配，单独 BM25 又不懂语义。混合召回能同时覆盖语义相似和关键词匹配，提高召回率，尤其适合企业文档里的产品名、制度编号、错误码、表格字段。

## 50. query rewrite 有什么用？

**面试答案：**  
query rewrite 把用户口语化、上下文依赖或不完整的问题改写成更适合检索的问题。比如多轮对话中“那怎么申请？”需要结合历史改写成“差旅报销怎么申请？”。

**风险：**  
改写错误会把检索方向带偏，所以要评估 rewrite 后的召回效果。

## 51. rerank 的作用是什么？

**面试答案：**  
召回阶段追求快和覆盖，rerank 阶段用更强模型对候选文档精排，把真正含答案的片段排到前面。常见是 bi-encoder 召回 + cross-encoder rerank。

**一句话：**  
召回负责“捞得全”，rerank 负责“排得准”。

## 52. cross-encoder reranker 为什么更精确但更慢？

**面试答案：**  
bi-encoder 是 query 和 doc 分别编码后算向量相似度，速度快但交互弱。cross-encoder 把 query 和 doc 拼在一起输入模型，能做 token 级交互，判断更细，但每个 query-doc pair 都要跑模型，所以更慢。

## 53. RAG 为什么还会幻觉？

**面试答案：**  
RAG 幻觉可能来自检索、排序、生成和系统设计。检索可能没召回正确证据，rerank 可能排错，文档可能过时或冲突，prompt 可能没约束基于证据，模型也可能用常识补全无证据内容。

**解决：**

- 提升召回。
- 加 rerank。
- 引用来源。
- 证据不足时拒答。
- 答案-证据一致性校验。
- bad case 回流。

## 54. 如何评估 RAG 检索效果？

**面试答案：**  
检索看 Recall@k、Precision@k、MRR、NDCG，以及 gold 文档是否被召回。实际项目里还要看召回片段是否包含答案，而不是只看语义相关。

**重点：**  
如果没有标注集，至少要构造一批业务 query 和标准证据文档做离线评估。

## 55. 如何评估 RAG 生成效果？

**面试答案：**  
生成看 answer relevance、faithfulness、citation correctness、hallucination rate 和人工满意度。业务上还要看解决率、转人工率、成本和延迟。

**一句话：**  
RAG 评估要分层：检索是否找对、上下文是否有答案、生成是否忠于证据。

## 56. 多轮对话 RAG 怎么管理历史上下文？

**面试答案：**  
不能把所有历史都塞进 prompt。通常做法是保留短期对话窗口、对历史做摘要、抽取用户意图和实体，并在检索前把当前问题改写成独立问题。

**例子：**  
用户问“这个怎么申请？”要结合上一轮“年假”改写为“年假怎么申请？”再检索。

## 57. 文档更新后索引怎么维护？

**面试答案：**  
要维护文档版本、更新时间和 chunk id。新增文档增量写入，修改文档删除旧 chunk 再重建，删除文档要同步删除索引。还要处理缓存失效和引用来源更新。

## 58. 如果召回为空怎么办？

**面试答案：**

- query rewrite 或 multi-query。
- fallback 到 BM25/网页/数据库。
- 降低阈值但控制噪声。
- 明确告诉用户知识库没有相关信息。
- 记录 query 作为知识库补充候选。

**不要做：**  
不要让模型无证据强答。

## 59. 如果召回内容互相矛盾怎么办？

**面试答案：**  
优先看文档版本、发布时间、来源权威度和业务规则。生成时可以提示模型说明存在冲突，并引用不同来源。系统上可以引入 metadata 优先级和冲突检测。

## 60. GraphRAG 适合什么场景？

**面试答案：**  
GraphRAG 适合实体关系复杂、需要多跳推理和全局总结的场景，比如组织关系、金融风控、知识图谱、法规条款关联。它通过图结构表达实体和关系，弥补普通 chunk 检索难以跨文档关联的问题。

## 61. Agent 和普通 Chatbot 有什么区别？

**面试答案：**  
普通 Chatbot 主要是输入问题、输出回答。Agent 是面向目标的执行系统，会规划步骤、调用工具、观察结果、维护状态，并根据反馈决定下一步。

**一句话：**  
Chatbot 是回答器，Agent 是任务执行器。

## 62. ReAct 的流程是什么？

**面试答案：**  
ReAct 是 Reasoning + Acting。模型先思考 Thought，再选择 Action 调用工具，得到 Observation 后继续推理，直到给出最终答案。

```text
Thought -> Action -> Observation -> Thought -> ... -> Final
```

## 63. Plan-and-Execute 适合什么任务？

**面试答案：**  
适合长任务、多步骤任务和路径相对清晰但执行细节复杂的任务。Planner 先拆解步骤，Executor 执行每步，必要时 re-plan。

**风险：**  
计划一开始错了会传导，所以需要执行反馈和重规划机制。

## 64. Function Calling 的本质是什么？

**面试答案：**  
Function Calling 是模型以结构化格式输出它想调用的函数名和参数，真正的函数执行由外部系统完成。模型不直接执行函数，应用侧负责参数校验、权限、执行和错误处理。

## 65. Tool Calling 失败怎么处理？

**面试答案：**

- 参数校验失败：让模型修正参数或返回用户补充信息。
- 工具超时：重试、降级或提示稍后再试。
- 权限不足：拒绝并说明权限。
- 结果为空：换检索策略或明确无结果。
- 高风险操作：人工确认。

## 66. 工具 schema 怎么设计？

**面试答案：**  
工具 schema 要清楚写明工具用途、参数类型、必填字段、枚举范围、返回格式和错误情况。描述要具体，避免多个工具功能重叠导致模型选择困难。

**好 schema 的标准：**  
模型能选对工具，人能校验参数，系统能稳定执行。

## 67. Agent 为什么容易死循环？

**面试答案：**  
原因包括目标不清、终止条件缺失、工具结果不明确、模型没读懂 observation、prompt 让模型过度尝试。解决上要设置最大步数、状态机约束、失败次数限制、明确 stop 条件和人工兜底。

## 68. Agent 记忆怎么设计？

**面试答案：**  
记忆可以分短期记忆、长期记忆和工作记忆。短期记忆是当前对话上下文，长期记忆保存用户偏好或历史事实，工作记忆保存任务状态。关键是决定记什么、何时写入、怎么检索、怎么防止错误记忆污染。

## 69. Agent 如何做权限控制？

**面试答案：**  
工具侧做权限，不相信模型本身。不同工具分权限等级，高风险操作需要用户确认；敏感数据做脱敏；执行前校验用户身份和工具参数；所有调用记录日志，便于审计。

## 70. Agent 如何评估？

**面试答案：**  
看任务完成率、工具选择准确率、参数准确率、平均步骤数、失败率、成本、延迟和安全违规率。要有固定任务集，覆盖正常任务、边界任务、工具失败任务和权限敏感任务。

## 71. MCP 解决什么问题？

**面试答案：**  
MCP，即 Model Context Protocol，解决模型应用和外部工具、资源、上下文之间的标准化连接问题。它让工具和数据源以统一协议暴露给模型应用，减少每个工具都要单独适配的成本。

## 72. MCP 和 Function Calling 的关系是什么？

**面试答案：**  
Function Calling 更关注模型如何结构化表达工具调用意图，MCP 更关注工具、资源和上下文如何以标准协议接入模型应用。两者不是同一层，但可以配合使用。

## 73. 多 Agent 有什么优点和风险？

**面试答案：**  
优点是可以按角色分工，比如 planner、researcher、coder、critic，各自处理不同子任务。风险是通信成本高、错误传播、协调复杂、难评估、延迟和成本增加。

**项目建议：**  
能用单 Agent 或工作流解决，就不要为了炫技上多 Agent。

## 74. 什么时候不应该用 Agent？

**面试答案：**  
流程固定、规则明确、风险高、强一致性要求高、工具调用成本高、延迟敏感时，不应该优先用 Agent。普通工作流更可靠、更便宜、更可控。

## 75. Context Engineering 和 Prompt Engineering 有什么区别？

**面试答案：**  
Prompt Engineering 关注单次提示词怎么写。Context Engineering 关注模型运行时拿到什么上下文、上下文如何组织、检索结果怎么压缩、记忆怎么管理、工具列表怎么选择、成本和缓存怎么控制。

**一句话：**  
Prompt 是一句话怎么问，Context 是整个信息供应系统怎么设计。
