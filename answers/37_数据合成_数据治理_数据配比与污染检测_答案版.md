# 答案版 37：数据合成、数据治理、数据配比与污染检测

对应题单 761-780。建议先看 [42_数据合成_数据治理_数据配比与污染检测面试.md](../42_数据合成_数据治理_数据配比与污染检测面试.md)，再用本答案版做口头复述。

## 761. 为什么大模型数据从“越多越好”转向“数据配方”？

因为低质、重复、污染和不合规数据会浪费训练预算，还可能让模型学到错误格式、幻觉和安全风险。SFT/偏好阶段数据量更小，坏样本影响更直接。

数据配方强调来源、过滤、去重、配比、模板、切分、版本和评估。面试可以说：数据规模是基础，有效学习信号和可控分布才决定能力上限。

## 762. Data recipe 面试怎么讲？

Data recipe 是可复现的数据配方：

```text
sources + filters + dedup + mixture weights + templates
+ label policy + split policy + version
```

回答时按 6 件事讲：数据从哪来、为什么可信、怎么清洗、怎么去重、怎么配比、怎么证明有效。加分点是说“用分桶 eval 和 ablation 证明配方有效”，而不是只说清洗了多少条。

## 763. 预训练数据质量过滤看哪些信号？

常见信号包括语言识别、长度、乱码、重复字符、HTML 残留、广告导航、SEO 内容、毒性、PII、困惑度、质量模型分数、来源可信度、许可证和 benchmark overlap。

面试要强调过滤不是删得越多越好，而是提高有效 token 密度，在质量、覆盖、多样性和偏差之间权衡。

## 764. DataComp-LM、FineWeb、Dolma 这类数据工作给面试什么启发？

它们说明数据集构建不是随便爬网页，而是系统比较数据源、过滤规则、去重、质量分数和训练效果。

面试可以总结三点：

1. 数据质量和配方会显著影响模型能力。
2. 数据处理流程要可复现、可审计。
3. 公开数据也要关注许可证、污染、PII 和来源偏差。

## 765. Exact dedup、near dedup、semantic dedup 怎么区分？

Exact dedup 用 hash 去完全重复。Near dedup 用 SimHash、MinHash、n-gram overlap 去镜像页、模板页、轻微改写。Semantic dedup 用 embedding similarity 去语义重复或改写重复。

训练去重关注有效 token 和减少记忆；评测去重关注防止模型见过题目、答案或同源文档。

## 766. Benchmark contamination 怎么检测和防？

污染是训练或调参过程中见过测试题、答案、解析或同源文档，导致评估虚高。

检测包括 exact match、n-gram overlap、MinHash/SimHash、embedding similarity、prompt-answer pair matching、source blacklist 和时间切分。

防护包括 benchmark blacklist、训练/评测按时间/来源/用户/文档隔离、few-shot 不用测试样本、记录 decontamination 方法。

## 767. 合成数据的典型闭环是什么？

闭环：

```text
seed tasks/docs/schemas
-> teacher generate
-> rule filter
-> judge/verifier filter
-> dedup
-> human spot-check
-> mix with real data
-> train/eval ablation
```

关键是过滤、去重、人工抽检和真实数据校准。合成数据适合补长尾和冷启动，但不能不加验证地灌进训练。

## 768. Self-Instruct、Evol-Instruct、Magpie 怎么区分？

Self-Instruct 从少量 seed tasks 出发自动生成更多 instruction/input/output。

Evol-Instruct 把简单指令逐步改写成更复杂、多约束、多步骤任务。

Magpie 利用强 instruction-tuned model 的行为自动生成指令数据，减少人工 seed 依赖。

共同点是用 teacher 扩数据，真正关键是生成后过滤、去重、分桶、抽检和 ablation。

## 769. UltraFeedback 类偏好数据要注意什么？

偏好数据要看 prompt 分布、chosen/rejected 差异、标注维度、长度偏置、judge 偏差和两个答案是否都错。

过滤要去掉低一致性标签、both-bad pair、重复 prompt，控制 chosen/rejected 长度分布，并按任务和风险分桶抽检。

一句话：DPO/RLHF 的难点不只是 loss，而是偏好标签是否真的代表目标行为。

## 770. 可验证合成数据为什么重要？

因为 teacher 和 judge 都可能幻觉。数学、代码、RAG、tool calling、JSON 等任务可以用 verifier 过滤。

例子：

- 数学：最终答案或符号检查。
- 代码：单元测试和执行。
- RAG：gold evidence 和引用覆盖。
- Tool calling：schema、参数和环境状态。
- JSON：parser 和 schema。

可验证数据能降低错误 trace 污染，也适合 RLVR、rejection sampling 和 hard negative 构造。

## 771. 数据配比怎么设计？

流程：

```text
定义目标能力 -> 建分桶 eval -> 设计 source/task mixture
-> 小规模 ablation -> 看 trade-off -> 调权重
-> 固化 recipe -> 大规模训练
```

配比维度包括通用、领域、代码、数学、多语言、安全、工具、长上下文、结构化输出、RAG/Agent trace。不能只看目标桶提升，还要看通用能力、安全和格式是否退化。

## 772. 合成数据比例过高有什么风险？

风险包括 teacher 风格单一、错误自我强化、难度不真实、回答变啰嗦、领域分布偏、真实用户表达覆盖不足，以及 eval 被 synthetic bias 污染。

治理方法是混入真实数据、控制合成比例、做 teacher 多样化、verifier 过滤、人工抽检，并用真实业务 eval 校准。

## 773. 难例挖掘和 bad case 回流怎么做？

先收集线上失败、eval failed cases、judge disagreement、低置信输出、用户差评、安全红队、工具失败 trace 和 RAG miss。

然后打标签、聚类、去重、归因，先加入 eval 回归集，再决定是否进入训练数据。原则是：bad case 先变成可复现评测，再进入训练闭环。

## 774. Train/valid/test 怎么切才安全？

不要简单随机切。应按用户、文档、工单、业务实体、时间或来源切分，避免同源泄露。

RAG eval 要隔离 gold evidence、query 和 reference answer；Agent eval 要隔离任务模板和环境状态；代码评测要考虑时间切分和仓库级去重。

一句话：大模型样本相似度高，随机切分很容易虚高。

## 775. Data card 要记录什么？

记录数据来源、许可证、时间范围、采集方式、清洗规则、去重方法、PII/safety 处理、schema、模板、标注协议、合成 teacher/judge 版本、train/valid/test split、语言/任务分布和已知限制。

Data card 的价值是可审计、可复现、可解释，也方便出问题时回滚。

## 776. 数据版本化怎么做？

版本化要记录：

```text
dataset_version
source_snapshot
filter_config
dedup_config
template_version
label_policy
split_rule
hash / manifest
changelog
```

实验记录要能从模型 checkpoint 反查用的是哪一版数据、哪套过滤规则、哪版模板和哪个 split。

## 777. 数据质量看板看哪些指标？

看 token 数、样本数、来源分布、语言/领域/任务分布、长度分布、重复率、PII 命中率、毒性/unsafe rate、schema valid rate、judge pass rate、human spot-check pass rate、contamination match rate、train/test overlap。

面试里加一句：我不会只报清洗后剩多少条，还会报过滤率、重复率、任务分布、污染命中率和抽检通过率。

## 778. 模型回归时怎么判断是不是数据问题？

先看哪个 eval bucket 回归，再查数据源、过滤、去重、配比、模板、loss mask、合成 teacher/judge、切分和标签协议是否变化。

典型数据问题包括：模板变化导致格式乱，配比变化导致某能力下降，合成数据幻觉变多，过滤太激进删掉领域样本，eval 被污染导致指标虚高。

## 779. 数据工程项目 8 分钟怎么讲？

结构：

1. 业务目标：提升什么能力。
2. 数据来源：真实、公开、人工、合成、线上日志。
3. 数据治理：清洗、去重、PII、质量过滤、污染检测。
4. 数据配方：来源权重、真实/合成比例、难例比例、replay。
5. 标注和合成：teacher、judge、verifier、抽检。
6. 切分和评估：按用户/文档/时间隔离，分桶 eval。
7. 复盘：哪些桶提升、哪些退化、bad case 怎么回流。

收束句：我做的不是拿数据训练，而是建立了可复现的数据配方和评估闭环。

## 780. 数据合成与治理面试前最后怎么复习？

最后背五条线：

1. 数据配方：source、filter、dedup、mixture、template、split、version。
2. 合成数据：teacher generate、filter、verifier、dedup、human spot-check、ablation。
3. 污染检测：exact、n-gram、MinHash、embedding、time/source split。
4. 配比实验：分桶 eval、小规模 ablation、看 trade-off。
5. 数据闭环：bad case 先入 eval，再决定是否入训练。

30 秒版：

> 数据工程进阶的核心是 data recipe：用可追溯的数据来源、清洗去重、防污染、合成过滤、配比实验和分桶评估，证明数据真的提升目标能力。
