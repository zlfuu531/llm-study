# 答案版 04：推理部署、项目与手撕代码

这一份对应 `03_高频题单100题.md` 的第 76-100 题。

## 76. TTFT、TPOT、QPS、tokens/s 分别是什么？

**面试答案：**

- TTFT：Time To First Token，首 token 延迟。
- TPOT：Time Per Output Token，每个输出 token 平均耗时。
- QPS：每秒请求数。
- tokens/s：每秒生成 token 数，常用于衡量吞吐。

**理解：**  
用户体感很看 TTFT，系统吞吐很看 tokens/s 和 QPS。

## 77. 为什么 decode 阶段常受内存带宽影响？

**面试答案：**  
decode 每次只生成一个 token，矩阵计算规模相对小，但每层都要读取历史 KV Cache。长上下文和高并发下，大量时间花在读写 KV Cache，所以常受内存带宽限制。

## 78. PagedAttention 解决什么问题？

**面试答案：**  
PagedAttention 把 KV Cache 像操作系统分页一样拆成 block 管理，避免不同请求长度不一致导致的大块连续显存浪费和碎片问题，提高显存利用率和吞吐。

## 79. Continuous Batching 是什么？

**面试答案：**  
Continuous Batching 是动态批处理。生成过程中，已完成的请求退出，新请求可以加入 batch，而不是等整批都结束。它能提高 GPU 利用率，适合输出长度不一致的在线推理。

## 80. prefix cache 有什么用？

**面试答案：**  
prefix cache 缓存相同前缀 prompt 的 KV Cache。多个请求共享系统提示词、知识库前缀或长上下文时，可以复用前缀计算，降低 TTFT 和计算成本。

## 81. 量化为什么能省显存？

**面试答案：**  
量化把权重从 FP16/BF16 等高精度表示压到 INT8/INT4 等低比特表示，单个参数占用字节更少，所以模型权重显存降低，也可能减少内存带宽压力。

## 82. INT8 和 INT4 的风险是什么？

**面试答案：**  
风险是精度损失，可能表现为回答质量下降、幻觉增加、格式遵循变差、数学/代码能力下降。INT4 压缩更强，风险通常比 INT8 更大。上线前必须用业务评测集验证。

## 83. GPTQ 和 AWQ 大致区别是什么？

**面试答案：**  
GPTQ 是基于二阶近似的后训练量化方法，逐层优化量化误差。AWQ 强调 activation-aware，保护对激活更重要的权重通道，减少量化对模型输出的影响。

**面试够用版：**  
两者都是常见 PTQ 方法，目标是在不重新训练或少量校准的情况下压缩权重并尽量保持效果。

## 84. Speculative Decoding 怎么加速？

**面试答案：**  
用小模型先快速草拟多个 token，再让大模型并行验证。如果大模型接受这些 token，就一次推进多步，减少大模型逐 token 调用次数。

**限制：**  
小模型要和大模型分布接近，否则接受率低，收益不明显。

## 85. 流式输出怎么实现？

**面试答案：**  
后端生成 token 时边生成边通过 SSE、WebSocket 或 chunked response 推给前端。前端增量渲染文本。服务端要处理取消、超时、异常、敏感词和日志。

## 86. 高并发下如何做限流和降级？

**面试答案：**

- 请求队列和优先级。
- 用户级/租户级限流。
- 最大上下文长度限制。
- 超时取消。
- 降级到小模型。
- 降低生成长度。
- 缓存常见请求。
- 忙时拒绝低优先级任务。

## 87. 大模型服务如何做日志和监控？

**面试答案：**  
监控 TTFT、TPOT、tokens/s、QPS、P95/P99、GPU 利用率、显存、错误率、超时率、队列长度和成本。日志要记录 request id、模型版本、prompt 模板版本、检索结果、工具调用、输出摘要和用户反馈，同时注意隐私脱敏。

## 88. 怎么估算部署一个 7B/14B 模型需要多少显存？

**面试答案：**  
先估权重：FP16 下每参数 2 bytes，所以 7B 约 14GB，14B 约 28GB。再加 KV Cache、激活 buffer、框架开销。INT4 权重大约降到四分之一左右，但还要加 scale/metadata。

**KV Cache 也要算：**

```text
batch * seq_len * layers * kv_heads * head_dim * 2 * bytes
```

## 89. 长上下文为什么贵？

**面试答案：**  
训练/预填充阶段 attention 复杂度和序列长度平方相关，decode 阶段 KV Cache 随序列长度线性增长。长上下文会增加计算、显存、延迟和成本，还可能让模型忽略关键内容。

## 90. 如何降低推理成本？

**面试答案：**

- 量化。
- 小模型蒸馏。
- KV Cache 优化。
- prefix cache。
- continuous batching。
- 限制 max tokens。
- RAG 精简上下文。
- prompt 压缩。
- 热点缓存。
- 路由到不同大小模型。

## 91. 手写 Self-Attention。

**面试答案要点：**  
输入 `x: [B, T, C]`，投影出 Q/K/V，计算 `QK^T / sqrt(D)`，加 causal mask，softmax，乘 V。

**必须说清：**

- score shape 是 `[B, T, T]`。
- softmax 在最后一维。
- mask 在 softmax 前。

## 92. 手写 Multi-Head Attention。

**面试答案要点：**

```text
x: [B, T, C]
qkv: [B, T, 3C]
q/k/v: [B, H, T, D]
score: [B, H, T, T]
out: [B, T, C]
```

**易错点：**

- `C % H == 0`。
- transpose 后 reshape 要注意 contiguous。
- 缩放因子是 `sqrt(head_dim)`。

## 93. 手写 causal mask。

**面试答案要点：**

```python
mask = torch.triu(torch.ones(T, T), diagonal=1).bool()
score = score.masked_fill(mask, float("-inf"))
```

如果 score 是 `[B, H, T, T]`，mask 可 broadcast 成 `[1, 1, T, T]`。

## 94. 手写 RoPE。

**面试答案要点：**

- 生成 `inv_freq`。
- 根据 position 得到角度。
- 算 cos/sin。
- 对偶数维和奇数维做旋转。
- 作用到 Q/K。

**易错点：**  
不要作用到 V；cos/sin 维度要能 broadcast 到 `[B, H, T, D]`。

## 95. 手写 RMSNorm 或 LayerNorm。

**RMSNorm：**

```text
y = x / sqrt(mean(x^2) + eps) * weight
```

沿 hidden dimension 计算，不减均值。

**LayerNorm：**

```text
y = (x - mean) / sqrt(var + eps) * weight + bias
```

## 96. 手写 Top-k / Top-p sampling。

**Top-k：**  
保留 logits 最大的 k 个，其余置为 `-inf`，softmax 后采样。

**Top-p：**  
按概率降序排序，保留累计概率达到 p 的最小集合，其余置为 `-inf`，再采样。

**追问：**  
Top-p 更动态，因为候选集合大小随概率分布变化。

## 97. 手写 BPE tokenizer 简化版。

**面试答案要点：**

1. 从字符级词表开始。
2. 统计相邻 token pair 频率。
3. 合并最高频 pair。
4. 重复直到达到词表大小。

**回答重点：**  
BPE 通过子词合并缓解 OOV，兼顾词表大小和表达能力。

## 98. 手写 LoRA linear layer。

**面试答案要点：**

```text
base = x @ W^T
lora = (x @ A^T) @ B^T * alpha / r
out = base + lora
```

base weight 冻结，只训练 A/B。

## 99. 手写 DPO loss。

**面试答案要点：**

```text
pi_logratio = policy_chosen - policy_rejected
ref_logratio = ref_chosen - ref_rejected
loss = -logsigmoid(beta * (pi_logratio - ref_logratio))
```

核心是让 policy 相对 reference 更偏好 chosen。

## 100. 手写 KV Cache 简化版。

**面试答案要点：**

- prefill 时保存每层 K/V。
- decode 时只计算新 token 的 K/V。
- 将新 K/V append 到 cache。
- 当前 Q attend 到完整 cache。

**Shape：**

```text
k_cache: [B, H_kv, T_cache, D]
v_cache: [B, H_kv, T_cache, D]
q_new:   [B, H_q, 1, D]
```

GQA/MQA 场景要处理 `H_q` 和 `H_kv` 不一致。
