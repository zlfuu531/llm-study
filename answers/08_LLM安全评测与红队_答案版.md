# 答案版 08：LLM 安全评测与红队

这一份对应 `03_高频题单100题.md` 的第 181-200 题。目标是让你能把“安全、评测、红队、灰度、回滚”讲成工程闭环。

## 181. 为什么公开 benchmark 不够？

**面试答案：**  
公开 benchmark 只能看模型通用能力，比如知识、数学、代码和推理，但业务系统还包含私有数据、RAG 检索、工具调用、权限、prompt、上下文和线上反馈。一个模型榜单高，不代表你的知识库问答、客服 Agent 或内部工具调用可靠。上线前必须做私有 eval、红队测试和灰度监控。

## 182. 私有 eval 样本应该怎么构造？

**面试答案：**  
样本来源包括真实用户 query、历史工单、线上 bad case、产品文档、人工边界样本、红队攻击样本和权限敏感样本。RAG 样本要标 query、gold evidence、reference answer、权限标签和是否允许拒答；Agent 样本要标目标、初始状态、可用工具、期望工具轨迹、参数、风险等级和成功条件。

## 183. LLM-as-judge 有哪些偏差，怎么缓解？

**面试答案：**  
常见偏差有位置偏差、长度偏差、风格偏好、自家模型偏好和业务规则理解不稳定。缓解方法是固定 judge prompt 和模型版本，写清 rubric，A/B 交换位置，多 judge 或多次评估，抽样人工复核，并用人工标注集校准。

## 184. OWASP LLM Top 10 里哪些风险最常被问？

**面试答案：**  
高频包括 prompt injection、敏感信息泄露、供应链风险、数据/模型投毒、输出处理不当、过度代理、系统提示泄露、向量和 embedding 弱点、错误信息和无限资源消耗。RAG 面试重点看注入、投毒和越权召回；Agent 面试重点看过度代理、工具越权和成本失控。

## 185. prompt injection 和 jailbreak 有什么区别？

**面试答案：**  
prompt injection 是通过输入或外部内容改变模型行为，让它违背系统指令或调用不该调用的工具。jailbreak 更偏绕过安全策略，让模型输出本该拒绝的内容。RAG/Agent 里更危险的是间接 prompt injection，因为恶意指令可能藏在网页、文档、邮件或 OCR 结果里。

## 186. RAG 怎么防间接 prompt injection？

**面试答案：**  
把检索文档当作不可信 evidence，而不是 instruction。系统 prompt 要明确区分系统指令、用户问题和外部证据；工具调用要在应用层做权限和参数校验；高风险操作需要人工确认；检索内容可做风险标签；红队集中加入藏在文档里的恶意指令。

## 187. 向量库投毒是什么，怎么防？

**面试答案：**  
向量库投毒是恶意文档或 chunk 进入知识库，被检索召回后影响模型回答或诱导工具调用。防护包括文档准入审核、来源和版本 metadata、权限标签、异常内容检测、索引版本化、召回结果审计、引用校验和可回滚索引。

## 188. Agent 的 excessive agency 怎么理解？

**面试答案：**  
过度代理指 Agent 拥有过多功能、权限或自主性，导致模型出错时能执行破坏性动作。比如自动删数据、发邮件、退款或修改权限。治理是最小权限、工具白名单、动态工具过滤、参数校验、最大步数、预算限制、HITL 和审计日志。

## 189. 系统提示泄露为什么危险？

**面试答案：**  
系统提示可能暴露内部策略、工具说明、路由逻辑甚至敏感配置。攻击者拿到这些信息后更容易绕过 guardrail 或构造精准攻击。系统提示里不应该放密钥、内部账号或不可泄露的信息，真正的权限要放在系统和工具层，不放在 prompt 里。

## 190. 模型输出进入下游系统前要做什么？

**面试答案：**  
要做格式校验、类型校验、业务规则校验、安全扫描和权限校验。不能把模型输出直接当 SQL、命令、HTML、代码或工具参数执行。高风险输出要进沙箱或人工确认，所有执行动作要记录 request id 和 trace。

## 191. 红队测试应该覆盖哪些攻击？

**面试答案：**  
覆盖 jailbreak、direct prompt injection、indirect prompt injection、system prompt leakage、PII 泄露、未授权工具调用、RAG poisoning、不安全输出处理、成本攻击和 misinformation。多模态系统还要测图片/OCR 中隐藏的恶意指令。

## 192. 红队报告应该包含哪些字段？

**面试答案：**  
至少包含 attack id、risk type、输入、期望安全行为、实际输出、工具 trace、严重程度、复现步骤、根因、修复方案和 regression case。红队样本不能测完就丢，要加入回归集。

## 193. 如何平衡安全拒答和可用性？

**面试答案：**  
不能只追求攻击拦截率，否则模型会什么都拒绝。要同时看 unsafe pass rate 和 false refusal rate。把拒答原因拆成证据不足、权限不足、安全禁止、问题不清，并为每类设计不同回复和补救路径。

## 194. 安全 SFT/DPO 能解决所有风险吗？

**面试答案：**  
不能。安全训练能提升模型倾向，但 RAG/Agent 风险很多来自系统层：外部文档、工具权限、向量库、业务执行、日志和用户身份。安全要同时做模型对齐、输入输出过滤、权限控制、红队 eval 和运行时监控。

## 195. NIST AI RMF / GenAI Profile 怎么落到项目？

**面试答案：**  
可以用 `Govern -> Map -> Measure -> Manage` 来讲。Govern 是明确责任、审批和事故响应；Map 是识别场景、用户、数据、权限和滥用方式；Measure 是用 eval、红队、日志和人工审核度量风险；Manage 是做防护、灰度、回滚和持续改进。

## 196. LLM 应用上线前的安全门禁是什么？

**面试答案：**  
上线前要过基础功能 eval、私有业务 eval、红队 eval、权限测试、工具调用测试、成本压力测试和回滚演练。高风险系统还要人工复核策略、日志审计和应急预案。

## 197. 线上监控安全风险看什么指标？

**面试答案：**  
看安全违规率、拒答率、误拒率、未授权工具调用拦截数、PII 检测命中数、prompt injection 命中数、工具失败率、P95/P99、token 成本、用户投诉和人工复核命中率。

## 198. LLM 应用出安全事故怎么复盘？

**面试答案：**  
先根据 request id 找到输入、检索片段、prompt 版本、模型版本、工具 trace、输出、安全策略和用户反馈。再判断根因是模型、prompt、索引、权限、工具 schema 还是业务规则。最后修复并把样本加入 regression eval，必要时回滚相关版本。

## 199. RAG/Agent 的灰度和回滚要回滚什么？

**面试答案：**  
不只回滚模型。还要能回滚 prompt、RAG 索引、embedding、reranker、工具 schema、安全策略、路由策略和模型网关配置。每个关键资产都要版本化，否则 bad case 很难定位。

## 200. LLM 安全的核心原则是什么？

**面试答案：**  
不信任模型，不信任用户输入，不信任外部内容，不把模型输出直接执行。所有高风险动作都要权限、校验、审计、人工确认和可回滚。安全不是一个过滤器，而是 eval、权限、trace、监控和流程共同组成的闭环。
