# 答案版 06：大模型系统设计追问题

这一份对应 `03_高频题单100题.md` 的第 141-160 题。目标不是背固定架构图，而是把系统设计题答成“有约束、有指标、有取舍、有上线闭环”的工程方案。

## 141. 设计一个企业级 RAG 知识库问答系统，你怎么讲？

**面试答案：**  
我会先澄清文档规模、更新频率、权限、延迟和是否需要引用。架构上分离线建库和在线问答。离线侧做文档采集、解析/OCR、清洗去重、结构化 chunk、metadata、embedding、向量索引和 BM25 索引，并维护版本。在线侧先做鉴权、query rewrite，再做 hybrid retrieval、rerank、context packaging，最后让模型基于证据回答并返回引用。评估上分检索、上下文、生成和业务四层，线上记录 trace，bad case 回流。

**追问补一句：**  
生产级 RAG 的重点不是“接个向量库”，而是可评估、可追溯、可更新、可控权限和成本。

## 142. RAG demo 和生产系统差在哪里？

**面试答案：**  
demo 通常是文档切分、embedding、向量检索、拼 prompt。生产系统还要处理文档解析质量、chunk 版本、混合检索、rerank、权限过滤、引用、拒答、冲突文档、缓存、限流、监控、私有 eval、bad case 回流和回滚。

**一句话：**  
demo 证明能跑，生产要证明可用、可信、可运维。

## 143. 文档更新后，RAG 索引怎么维护？

**面试答案：**  
每个文档和 chunk 要有稳定 id、版本、更新时间和来源。新增文档做增量 embedding 和入库；修改文档要删除或失效旧 chunk，再写入新版本；删除文档要同步删除向量索引、关键词索引和缓存。线上引用要能追溯到具体版本，避免回答引用过期内容。

**风险：**  
如果只追加不删除，旧制度和新制度会同时被召回，模型可能给出冲突答案。

## 144. RAG 系统又慢又贵，怎么优化？

**面试答案：**  
先分解耗时和成本：query rewrite、召回、rerank、prompt tokens、模型生成、网络。优化方向包括减少不必要 rewrite、控制 top-k、只对候选做 rerank、压缩 context、缓存高频 query 和固定前缀、按任务选择小模型、限制 max output tokens、流式输出提升体感速度。

**注意：**  
不能只为了省钱砍 top-k，要同时看 Recall@k 和 faithfulness 是否下降。

## 145. 设计一个能调用内部系统的 Agent，你怎么讲？

**面试答案：**  
我会把 Agent 设计成受控状态机。系统包含任务路由、规划/状态图、上下文构建、工具选择、参数校验、权限校验、工具执行、observation 解析、状态更新、终止判断和人工确认。工具注册表记录 schema、返回格式、权限等级和幂等性。高风险动作必须 human-in-the-loop，所有步骤都记录 trace。

**一句话：**  
模型提出动作，系统负责校验、执行、审计和兜底。

## 146. Agent 工具权限和审计怎么做？

**面试答案：**  
权限不能交给模型判断。工具要按用户身份、租户、任务类型和风险等级动态暴露。执行前校验参数、资源范围和业务权限；高风险动作如退款、删数据、发消息要人工确认；敏感字段脱敏；所有工具调用记录用户、工具、参数、返回、耗时、错误、审批人和 trace id。

**常见追问：**  
如果模型请求一个未授权工具，系统直接拒绝，并把原因返回给编排层或用户，而不是让模型继续猜。

## 147. MCP 在系统设计里放在哪一层？

**面试答案：**  
MCP 放在工具和上下文接入层。Agent Orchestrator 可以通过 MCP Client 连接多个 MCP Server，Server 暴露 Tools、Resources 和 Prompts。Function Calling 更偏模型输出工具名和参数，MCP 更偏工具、资源和提示模板如何标准化接入，Agent 框架负责状态和编排。

**一句话：**  
Function Calling 是“模型怎么说要调工具”，MCP 是“工具怎么被标准化接入系统”。

## 148. Agent memory 怎么设计？

**面试答案：**  
可以分短期记忆、工作记忆和长期记忆。短期记忆保存当前对话窗口；工作记忆保存任务状态、步骤和工具结果；长期记忆保存用户偏好、历史事实或可复用经验。长期记忆写入要谨慎，最好有置信度、来源和过期机制，避免错误记忆污染后续任务。

**项目话术：**  
我会把 memory 当成可检索的数据层，而不是无限追加到 prompt。

## 149. Agent 怎么评估才可靠？

**面试答案：**  
要评估轨迹，不只看最终答案。指标包括 task success rate、tool call accuracy、argument accuracy、平均步骤数、失败恢复率、人工确认触发率、成本、延迟和安全违规率。评估集要覆盖正常任务、工具为空、工具超时、参数缺失、权限不足、高风险动作和 prompt injection。

**一句话：**  
Agent 的质量在 action/observation 轨迹里，不只在 final answer 里。

## 150. 设计一个模型网关，需要哪些模块？

**面试答案：**  
模型网关要做鉴权、租户隔离、请求规范化、prompt/template 版本、模型路由、token 预算、限流、排队、熔断、重试、fallback、安全过滤、streaming、日志、成本统计、灰度和回滚。它是 LLM 应用的控制面，不只是 API 代理。

**追问补一句：**  
模型网关最好记录 prompt 版本和模型版本，否则线上 bad case 很难复现。

## 151. 为什么大模型服务要按 token 限流，而不只按 QPS？

**面试答案：**  
因为 LLM 请求的成本和延迟主要由输入输出 token 决定。同样 1 QPS，200 token 请求和 20k token 请求对 GPU、显存、外部 API 成本完全不同。所以要同时限制 QPS、input tokens/min、output tokens/min、并发数、max context 和 max generation tokens。

## 152. 流式输出中途失败怎么办？

**面试答案：**  
要看失败阶段。如果模型服务超时或断流，可以返回已生成内容并提示失败，也可以让用户重试；如果业务要求完整性，应该标记本次回答不完整。系统侧要记录 request id、已输出 token、错误码和模型状态。对有副作用工具的 Agent，不能因为断流就重复执行工具，必须保证幂等或通过 trace 恢复。

**关键点：**  
文本生成可以重试，外部副作用动作不能盲目重试。

## 153. 大模型和小模型怎么路由？

**面试答案：**  
可以按任务复杂度、风险等级、上下文长度、结构化程度、延迟和成本路由。简单分类、抽取、格式化走小模型；复杂推理、多跳 RAG、高风险回答走大模型；失败或低置信度时升级模型；高峰期可以降级到小模型或缩短输出。

**风险：**  
路由模型本身也要评估，否则错误路由会导致质量下降。

## 154. 设计高并发 LLM 推理服务，你会怎么讲？

**面试答案：**  
入口网关做鉴权、token 预算、限流和排队；调度层做 continuous batching；推理引擎管理 KV Cache，可用 PagedAttention 降低碎片；固定前缀用 prefix cache；长 prompt 用 chunked prefill 缓解阻塞；必要时用量化和 speculative decoding。监控 TTFT、TPOT、tokens/s、队列长度、batch size、cache hit rate、GPU 利用率、显存和错误率。

## 155. 线上 TTFT 变高，怎么排查？

**面试答案：**  
TTFT 主要看首 token 前的路径：入口排队、鉴权/安全过滤、prompt 过长、tokenizer、prefill batch 被长请求阻塞、prefix cache 未命中、GPU 利用率、显存碎片、网络和流式 flush。先看队列和输入长度分布，再看 prefill 耗时和 cache 命中率。

**一句话：**  
TTFT 更偏排队和 prefill。

## 156. 线上 TPOT 变高，怎么排查？

**面试答案：**  
TPOT 主要看 decode 阶段：batch size 是否过大、KV Cache 是否接近上限、显存带宽是否瓶颈、量化 kernel 是否高效、tensor parallel 通信是否过重、输出是否过长、speculative decoding 接受率是否低。先看 decode step latency、GPU memory bandwidth 和 batch 组成。

**一句话：**  
TPOT 更偏持续生成和内存带宽。

## 157. Continuous Batching 的取舍是什么？

**面试答案：**  
Continuous Batching 能让 decode 阶段请求动态进出 batch，提高 GPU 利用率和吞吐。但 batch 过大可能增加单请求延迟，长短请求混在一起也会影响公平性。调度器需要考虑优先级、最大等待时间、token 预算和 prefill/decode 平衡。

## 158. LLM 应用的日志和 trace 要记录什么？

**面试答案：**  
基础字段包括 request id、用户/租户、输入长度、输出长度、模型版本、prompt 版本、耗时、成本和错误。RAG 还要记录 query rewrite、retrieved chunk、rerank 分数、context tokens、引用和用户反馈。Agent 还要记录每步 action、arguments、validation、tool output、retry、human approval、状态快照和终止原因。

**一句话：**  
日志要能 replay，不只是能看报错。

## 159. 怎么防 prompt injection 和数据泄露？

**面试答案：**  
首先区分系统指令、用户输入和外部检索内容，不让外部内容覆盖系统策略。检索前后都做权限过滤，工具按用户和任务动态暴露，高风险动作人工确认。对用户上传文档和网页打风险标签，输出前做敏感信息检测和引用校验。还要限制工具次数、上下文长度和输出长度，防成本攻击。

## 160. LLM-as-judge 靠谱吗，怎么用？

**面试答案：**  
LLM-as-judge 可以辅助评估答案相关性、faithfulness、格式合规和安全风险，但不能当唯一真值。要固定 judge prompt 和模型版本，用人工标注集校准，关键样本人工复核。复杂业务规则、权限、安全红线最好用规则和人工兜底。

**一句话：**  
LLM-as-judge 是评估工具，不是绝对裁判。
