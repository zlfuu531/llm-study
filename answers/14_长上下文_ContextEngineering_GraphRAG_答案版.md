# 答案版 14：长上下文、Context Engineering 与 GraphRAG

对应 `03_高频题单100题.md` 的 301-320 题。

## 301. 长上下文为什么不是万能？

长上下文只是扩大输入窗口，不会自动选择正确证据。它会带来更高 prefill 延迟、KV Cache 显存压力、输入 token 成本、无关信息干扰和权限泄露风险。模型还可能受 lost in the middle 影响忽略中间证据，所以仍要做检索、排序、压缩和评估。

## 302. Lost in the Middle 是什么？

它指模型在长上下文里更容易利用开头和结尾的信息，中间位置的证据更容易被忽略。缓解方法包括 rerank 后把关键证据放到显著位置、结构化摘要、分步检索、引用约束，以及在 eval 里测试证据位于不同位置时答案是否稳定。

## 303. RAG 和长上下文怎么组合？

RAG 负责从大规模资料里选证据，长上下文负责在选出的证据内综合理解。常见做法是先检索和 rerank，再取 top evidence、摘要和必要历史拼入长上下文模型。长上下文不能替代权限过滤、版本管理、引用和评估闭环。

## 304. Context Engineering 和 Prompt Engineering 有什么区别？

Prompt Engineering 更关注指令怎么写，Context Engineering 更关注模型运行时拿到什么信息。它包括检索结果、记忆、工具返回、用户状态、历史对话、权限过滤、压缩、排序和 token budget。生产系统里上下文供应链往往比单个 prompt 更关键。

## 305. Context Builder 负责什么？

Context Builder 负责把候选上下文变成模型可用输入：收集检索、记忆、工具和历史；做权限过滤、相关性打分、去重、冲突检测、排序、压缩和预算分配；最后生成带 source id 的上下文，方便引用、审计和 bad case 回放。

## 306. token budget 怎么分配？

先保留系统规则、当前问题和输出预算，再给核心证据、任务状态、工具结果、历史摘要分配预算。原则是当前任务和高置信证据优先，长期历史摘要化，低相关内容不进入上下文，权限不确定的信息直接排除。预算不是固定比例，而是随任务动态调整。

## 307. 上下文压缩有哪些方法和风险？

方法包括抽取式压缩、摘要式压缩、结构化压缩、层级摘要和查询相关压缩。风险是漏掉 gold evidence、引入幻觉、丢失细节或改变事实。评估要看压缩后证据是否保留、faithfulness 是否下降、引用是否正确、成本是否真的降低。

## 308. Query rewrite、HyDE、Multi-query 分别解决什么？

Query rewrite 把用户问题改写得更适合检索，适合省略、代词和业务别名。HyDE 先生成假设答案再用它检索，适合短问题和语义稀疏问题。Multi-query 生成多个查询角度提高 recall，但会增加噪声和成本。都要用 Recall@k、MRR 和端到端质量评估。

## 309. GraphRAG 适合什么场景？

适合实体关系复杂、多跳问答、跨文档总结、组织关系、金融、法律、风控、医学等场景。普通 RAG 找 chunk，GraphRAG 找实体、关系、社区和路径。文档少、关系简单、普通 RAG 已够用时，不建议硬上 GraphRAG。

## 310. GraphRAG 的典型流程是什么？

文档先做实体和关系抽取，再构建图；之后做社区发现或层级摘要；查询时定位相关实体、子图或社区摘要，再把图证据和原文证据交给模型回答。关键是图中节点和边要能回链到原文，方便引用和纠错。

## 311. GraphRAG 错了会怎样？

实体抽取、消歧、关系边、社区摘要或权限过滤出错，都会导致多跳路径错、答案引用错、全局摘要偏，错误还会被结构化形式放大。治理要做 provenance、图谱版本化、实体关系评估、社区摘要抽检和原文回链。

## 312. RAPTOR 和 GraphRAG 有什么区别？

RAPTOR 是层级摘要检索：chunk 聚类、摘要、递归成树，检索时同时用叶子块和高层摘要。GraphRAG 是实体关系图检索。RAPTOR 更偏长文档/多文档层级总结，GraphRAG 更偏实体关系和多跳路径。

## 313. Agent memory 怎么分类？

短期记忆是当前对话原文，工作记忆是当前任务状态、计划和工具结果，长期记忆是用户偏好和稳定事实，检索记忆是可按需召回的历史和经验。面试要强调不同记忆生命周期不同，不能把所有聊天记录都塞进向量库。

## 314. Agent memory 怎么防污染和泄露？

写入前要有准入规则：只保存稳定、明确、允许保存的信息；敏感、越权、注入内容和工具错误结果不写长期记忆。读取时按相关性和权限过滤，标注来源和时间，允许用户删除或更正，并记录记忆命中 trace。

## 315. 长上下文能力怎么评估？

可以用 LongBench、RULER、needle-in-a-haystack、lost-in-the-middle 测试，也要做业务长文档 eval。指标包括答案正确率、证据定位、位置鲁棒性、输入 token 成本、延迟和不同长度分桶下的退化曲线。

## 316. Context Engineering 怎么评估？

看 context recall、context precision、faithfulness、citation accuracy、permission correctness、token cost 和 latency。还要做 ablation：naive top-k、+rerank、+query rewrite、+compression、+long context packaging、+GraphRAG/RAPTOR，比较每个模块贡献。

## 317. 什么时候用 GraphRAG，什么时候用普通 RAG？

普通事实问答、文档规模小、关系简单时用普通 RAG 更稳、更便宜。多跳关系、跨文档全局问题、实体关系复杂时用 GraphRAG。选择依据是业务问题是否依赖显式实体关系，而不是“GraphRAG 更新”。

## 318. 为什么上下文里要做权限过滤？

模型只要看到无权限信息，就可能泄露。RAG、GraphRAG、记忆和工具结果都要按用户权限过滤，尤其图谱节点、社区摘要和历史记忆也可能包含敏感信息。权限过滤要在进入上下文前完成，不能只靠模型自觉不说。

## 319. 长上下文项目怎么讲 8 分钟？

按“场景 -> 痛点 -> baseline -> Context Builder -> 压缩/GraphRAG -> 评估 -> 成本和风险”讲。强调没有直接塞所有文档，而是做召回、rerank、压缩、权限过滤和预算分配；用 context recall、faithfulness、引用正确率、P95 和成本证明效果。

## 320. Context Engineering 的核心原则是什么？

核心是给模型正确、必要、可追溯、权限安全、成本可控的上下文。不是越长越好，而是以任务为中心做选择、排序、压缩和评估。最终要能回答：为什么这些信息进入 prompt，为什么其他信息没进，出错时如何回放和修复。
