# 答案版 10：大模型数据工程与数据集构建

这一份对应 `03_高频题单100题.md` 的第 221-240 题。目标是把“数据怎么来、怎么清洗、怎么评估、怎么防污染”讲成可落地流程。

## 221. 预训练、SFT、偏好对齐分别需要什么数据？

**面试答案：**  
预训练需要大规模、多样、高质量文本/代码语料，目标是学习语言、知识和模式；SFT 需要 instruction/input/output 或 chat 格式数据，目标是学指令遵循；偏好对齐需要 prompt + chosen/rejected 数据，目标是学人类偏好、安全和帮助性。

## 222. 为什么数据清洗对大模型训练很重要？

**面试答案：**  
低质、重复、冲突、隐私、违法或格式混乱的数据会浪费训练预算，并让模型学到幻觉、重复、安全风险和错误格式。SFT 和偏好阶段数据量相对小，脏数据占比稍高就会明显影响模型行为。

## 223. 预训练数据质量过滤有哪些信号？

**面试答案：**  
常见信号包括文本长度、语言识别、乱码比例、重复字符、广告导航模板、困惑度、毒性、PII、代码/数学比例、来源可信度和 benchmark overlap。过滤目标是提高有效 token 占比，而不是盲目删数据。

## 224. exact dedup 和 near dedup 有什么区别？

**面试答案：**  
exact dedup 去完全相同文本，通常用 hash；near dedup 去近似重复，比如同一网页模板、轻微改写、镜像页面，可用 SimHash、MinHash、n-gram overlap 或 embedding similarity。大模型训练更需要 near dedup，因为重复变体很多。

## 225. benchmark contamination 是什么，怎么防？

**面试答案：**  
benchmark contamination 是训练数据包含评测题目或答案，导致评估虚高。防护包括 benchmark blacklist、exact/near match、n-gram overlap、MinHash、按来源过滤，以及按用户/时间/文档切分 train/test，避免同源泄露。

## 226. SFT 数据为什么要统一 chat template？

**面试答案：**  
chat template 决定 system/user/assistant 角色、特殊 token 和轮次边界。如果格式不统一，模型会学到混乱的角色标记，甚至输出用户内容或漏掉 assistant 边界。训练和推理 template 也要一致。

## 227. SFT loss mask 写错会怎样？

**面试答案：**  
如果 prompt 也算 loss，模型会学复述用户问题；如果 padding 没 mask，会污染梯度；如果多轮角色边界错，模型可能学用户说话或输出角色混乱。通常只对 assistant answer token 算 loss。

## 228. 偏好数据 chosen/rejected 怎么保证质量？

**面试答案：**  
要有清晰标注协议，维度包括正确性、帮助性、格式、简洁性、安全和证据引用。要避免长度偏置，过滤两个答案都错或差异太小的样本，抽样复核标注一致性，并保留 preference reason 便于审计。

## 229. DPO 数据里长度偏置有什么问题？

**面试答案：**  
如果 chosen 普遍比 rejected 长，模型可能学到“更长就是更好”，导致啰嗦和偏离任务。可以控制长度分布、加入简洁性标注、做长度分桶评估，或在 logprob 处理上考虑长度归一化变体。

## 230. 合成数据有什么用，风险是什么？

**面试答案：**  
合成数据可以扩充长尾任务、生成指令数据、构造偏好对、reasoning traces 和红队样本。风险是 teacher 幻觉、风格单一、难度不真实、错误自我强化和数据泄露。要过滤、去重、人工抽检，并用真实数据校准。

## 231. Self-Instruct 类方法的核心流程是什么？

**面试答案：**  
从少量 seed tasks 出发，让大模型生成新指令、输入和输出，再过滤低质、重复或不安全样本，形成更大的 instruction dataset。核心是自动扩展任务覆盖，但质量控制很关键。

## 232. 数据配比怎么影响模型能力？

**面试答案：**  
代码数据多会提升代码能力但可能影响自然语言风格；数学推理多会提升推理但可能更冗长；安全拒答多会降低 unsafe pass 但可能增加误拒；领域数据多会提升领域表现但可能损失通用能力。要用分桶验证集看 trade-off。

## 233. 如何缓解领域继续预训练的遗忘？

**面试答案：**  
可以混入通用 replay 数据，控制领域数据比例，降低学习率和训练步数，做阶段式训练，并用通用能力、领域能力、安全和格式遵循的验证集同时监控。不要只看领域 loss 下降。

## 234. 业务数据怎么划分 train/valid/test 才安全？

**面试答案：**  
不要简单随机切。可以按时间、用户、文档、工单、来源或业务实体切分，避免同一用户或同一文档的相似样本同时进入训练和测试。RAG eval 还要保证 reference answer 不泄露到 prompt。

## 235. RAG eval 数据需要哪些字段？

**面试答案：**  
至少包含 query、query_type、gold_doc_id/gold_chunk_id、reference_answer、must_cite、should_refuse、permission_label 和安全标签。这样才能分层评估召回、上下文、生成、引用和拒答。

## 236. Agent eval 数据需要哪些字段？

**面试答案：**  
需要 goal、initial_state、available_tools、expected_tool_trace、expected_arguments、success_criteria、risk_level 和是否需要人工确认。Agent 评估要看轨迹，不只看最终回答。

## 237. 数据版本化要记录什么？

**面试答案：**  
记录数据来源、抽取时间、清洗规则版本、去重规则、标注协议、标注者或 judge 模型、schema、train/valid/test split、数据 hash 和变更说明。这样实验可复现，出问题能回滚。

## 238. 线上 bad case 怎么回流到数据闭环？

**面试答案：**  
先打标签归因：数据缺失、检索失败、标注错误、prompt 问题、模型能力、安全策略或工具问题。再补充到 eval 或训练数据，跑回归评估，确认没有引入新问题后灰度上线。

## 239. 给你一批业务数据，你怎么判断 RAG、SFT 还是 DPO？

**面试答案：**  
如果问题是知识更新和可引用，优先 RAG；如果模型知道但不按格式或风格回答，考虑 SFT；如果已有多个回答和偏好判断，考虑 DPO/RLHF；如果只是少量规则任务，可能 prompt + 规则就够。先建 eval 再选方案。

## 240. 数据工程项目 8 分钟深挖版应该讲什么？

**面试答案：**  
讲业务目标、数据来源、数据审计、清洗去重、脱敏合规、schema、标注协议、train/valid/test 划分、eval 指标、bad case 回流、版本管理和上线效果。重点证明你不是“拿数据训练”，而是建立了数据闭环。
