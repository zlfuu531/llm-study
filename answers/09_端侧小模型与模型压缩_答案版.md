# 答案版 09：端侧小模型与模型压缩

这一份对应 `03_高频题单100题.md` 的第 201-220 题。目标是让你能把端侧部署、蒸馏、量化、剪枝和本地运行时讲成可落地方案。

## 201. 为什么还需要小模型和端侧部署？

**面试答案：**  
不是所有任务都需要最大模型。小模型在成本、延迟、隐私、离线可用和端侧交互上有优势。简单分类、抽取、改写、格式化、个性化助手可以用小模型；复杂推理、高风险决策和长上下文任务再路由到大模型。

## 202. 端侧 LLM 和云端 LLM 的指标有什么不同？

**面试答案：**  
云端更看 QPS、tokens/s、GPU 利用率和 P95；端侧还要看模型文件大小、峰值内存、冷启动、首 token 延迟、每 token 延迟、功耗、发热、降频、离线可用和设备兼容性。

## 203. 知识蒸馏的基本公式是什么？

**面试答案：**  
teacher 和 student 都用温度 softmax 得到平滑分布：

```text
p_i = exp(z_i / T) / sum_j exp(z_j / T)
L_distill = T^2 * KL(p_teacher^T || p_student^T)
```

实际训练常和真实标签交叉熵混合：

```text
L = alpha * CE + (1 - alpha) * L_distill
```

## 204. response distillation 和 logit distillation 有什么区别？

**面试答案：**  
logit distillation 学 teacher 的概率分布，能学习类别之间的暗知识；response distillation 用 teacher 生成的回答或指令数据训练 student，更适合 LLM 指令遵循、领域问答和格式输出。LLM 项目里 response/reasoning distillation 更常见。

## 205. reasoning distillation 有什么风险？

**面试答案：**  
风险是 teacher 的错误推理和幻觉会被 student 学进去；长 CoT 也会增加训练和推理成本；小模型容量不足时可能只学表面格式。要做数据过滤、可验证答案、短轨迹蒸馏和任务分桶 eval。

## 206. PTQ 和 QAT 怎么选？

**面试答案：**  
PTQ 是训练后量化，快、便宜，适合快速部署；QAT 是训练时模拟量化，精度更稳，但成本更高。一般先 PTQ，用业务 eval 看掉点，如果低比特精度不够再考虑 QAT 或少量校准微调。

## 207. 对称量化和非对称量化公式是什么？

**面试答案：**  
对称量化：

```text
q = round(x / scale)
x_hat = q * scale
```

非对称量化：

```text
q = round(x / scale + zero_point)
x_hat = (q - zero_point) * scale
```

非对称量化能处理分布不以 0 为中心的张量。

## 208. per-tensor、per-channel、group-wise 量化有什么区别？

**面试答案：**  
per-tensor 是整个张量共享一个 scale，简单但误差大；per-channel 是每个通道一个 scale，精度更好；group-wise 是按组共享 scale，在精度和开销之间折中。LLM 低比特权重量化常用 group-wise。

## 209. GPTQ、AWQ、SmoothQuant 大致区别是什么？

**面试答案：**  
GPTQ 用近似二阶信息补偿量化误差；AWQ 认为少量激活敏感通道很重要，会保护这些权重通道；SmoothQuant 通过平滑激活 outlier，把激活量化难度迁移到权重上，更适合 INT8 权重和激活量化。

## 210. GGUF / llama.cpp 适合什么场景？

**面试答案：**  
适合本地 CPU/GPU 快速部署和端侧原型验证。GGUF 支持多种量化级别，llama.cpp 生态成熟、部署简单。但生产端侧还要考虑平台 runtime、功耗、模型更新、安全、设备差异和业务 eval。

## 211. Ollama 在本地部署里解决什么问题？

**面试答案：**  
Ollama 更偏本地模型管理和服务封装，方便拉模型、运行模型、配置 Modelfile 和提供本地 API。它适合开发调试和原型验证，但面试里要补充底层仍需要关心量化格式、性能、内存和质量。

## 212. ONNX Runtime GenAI、ExecuTorch、MLC LLM 怎么区分？

**面试答案：**  
ONNX Runtime GenAI 偏跨平台 ONNX 生态和推理接口；ExecuTorch 是 PyTorch 面向移动和边缘设备的部署栈；MLC LLM 强调编译优化和多后端部署。选择取决于模型来源、目标设备、后端支持、性能和工程生态。

## 213. 剪枝为什么可能不加速？

**面试答案：**  
非结构化剪枝会产生不规则稀疏，如果硬件和 kernel 不支持稀疏计算，推理仍按密集矩阵跑，可能只省存储不省时间。结构化剪枝删除 head、channel、neuron 或 layer，更容易带来真实加速。

## 214. SparseGPT 类方法解决什么？

**面试答案：**  
SparseGPT 代表一类 one-shot 大模型剪枝方法，利用近似二阶信息在不完整重训练的情况下剪掉部分权重并补偿误差。它适合快速压缩大模型，但实际加速仍取决于稀疏 kernel 和部署后端。

## 215. 端侧量化后效果下降，怎么排查？

**面试答案：**  
先看量化格式、bit 数、group size、校准数据和 outlier；再看是否有敏感层需要保留高精度；然后用任务分桶 eval 看数学、代码、长上下文、工具参数和安全拒答是否下降。补救包括换 INT8、混合精度、更好校准集、QAT/LoRA 修复或路由到高精度模型。

## 216. 为什么 INT4 不一定比 FP16 快？

**面试答案：**  
INT4 减少存储和带宽，但速度取决于硬件是否支持低比特计算、kernel 是否高效、解量化开销、batch size、上下文长度和内存带宽。如果 kernel 不好，可能只省显存不提速。

## 217. 小模型怎么和大模型做混合路由？

**面试答案：**  
简单、低风险、高频和隐私任务先走本地小模型；低置信度、复杂推理、长上下文和高风险任务升级到云端大模型。路由要看任务类型、置信度、成本、延迟、隐私和安全等级。

## 218. 端侧 RAG 怎么设计？

**面试答案：**  
本地保存私有文档和 embedding，小向量库在端侧检索，简单问题由本地小模型回答；复杂问题可以脱敏后上云。关键是控制索引大小、embedding 成本、隐私权限、离线可用和本地模型质量。

## 219. 端侧部署项目怎么做评估？

**面试答案：**  
质量上看业务准确率、格式遵循、幻觉、安全和拒答；性能上看 TTFT、TPOT、tokens/s、峰值内存、冷启动、模型大小、功耗、发热和低端设备兼容；还要看长时间运行是否降频、更新回滚是否可靠。

## 220. 端侧小模型项目 8 分钟深挖版讲什么？

**面试答案：**  
讲场景和设备约束、为什么选小模型、teacher/student 或模型来源、蒸馏/微调/量化/剪枝方案、部署 runtime、性能指标、质量 eval、bad case、fallback 到云端大模型的策略，以及上线更新和回滚。
