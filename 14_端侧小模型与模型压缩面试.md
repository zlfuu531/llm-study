# 端侧小模型与模型压缩面试

这一章面向端侧 AI、AI Infra、模型压缩、推理部署、移动端/PC 本地智能和大模型应用岗位。它和“云端推理优化”不完全一样：云端更关注 GPU 并发、vLLM、KV Cache、吞吐；端侧更关注内存、功耗、冷启动、隐私、离线可用、CPU/NPU/GPU 后端和小模型质量。

端侧语音、唤醒词、VAD、低延迟 ASR 和语音助手可以配合 [31_SpeechAudioLLM_ASR_TTS与实时语音Agent面试.md](31_SpeechAudioLLM_ASR_TTS与实时语音Agent面试.md) 看。

如果被追问量化公式、GPTQ/AWQ/SmoothQuant、QLoRA、FP8、KV Cache 量化、INT4 为什么不一定快，跳到独立专题：[37_模型量化_低比特推理_GPTQ_AWQ_SmoothQuant面试.md](37_模型量化_低比特推理_GPTQ_AWQ_SmoothQuant面试.md)。本章负责端侧和压缩全局，新章负责低比特推理细节。

你要能回答：

- 为什么还需要小模型？
- 蒸馏、剪枝、量化、低秩分解分别解决什么？
- GPTQ、AWQ、SmoothQuant、GGUF 大致区别是什么？
- 端侧部署为什么不只是把模型转格式？
- 手机/PC 本地跑 LLM 要看哪些指标？
- 量化后效果下降怎么评估和补救？

## 一句话总览

端侧小模型的核心目标是：

```text
在有限内存、算力、功耗和延迟约束下，让模型达到“够用”的质量。
```

常见路径：

```text
大模型 teacher
  -> 数据/响应/推理轨迹蒸馏
  -> 小模型 student
  -> 量化/剪枝/低秩/编译优化
  -> GGUF/ONNX/ExecuTorch/MLC 等部署格式
  -> CPU/GPU/NPU 本地推理
  -> 端侧 eval + 性能功耗监控
```

面试口语版：

> 端侧部署不是把 7B 模型压成 INT4 就完了。要先判断任务是否适合小模型，再通过蒸馏、领域微调、量化和运行时优化，在质量、延迟、内存、功耗、隐私和离线可用之间做取舍。

## 为什么还需要小模型

大模型强，但不是所有任务都需要最大模型。

小模型适合：

- 高频简单任务：分类、抽取、改写、格式化。
- 隐私敏感：不希望把数据发到云端。
- 离线场景：弱网、断网、车载、工业设备。
- 低延迟交互：键盘输入、语音字幕、端侧助手。
- 成本敏感：云 API 成本太高。
- 个性化：本地用户偏好、私有上下文。

不适合：

- 复杂推理。
- 多跳知识问答。
- 高风险决策。
- 需要最新知识。
- 长上下文深度理解。

面试答案：

> 小模型的价值不是替代所有大模型，而是在合适场景里用更低成本、更低延迟和更好隐私完成够用任务。复杂任务可以走模型路由：小模型先处理，低置信度或高风险再升级到大模型。

## 模型压缩总图

| 方法 | 直觉 | 优点 | 风险 |
| --- | --- | --- | --- |
| 蒸馏 | 大模型教小模型 | 提升小模型质量 | 依赖 teacher 和数据质量 |
| 量化 | 低比特表示权重/激活 | 省内存、降带宽 | 精度下降、kernel 依赖 |
| 剪枝 | 删除不重要参数/结构 | 减参数和计算 | 非结构化剪枝不一定加速 |
| 低秩分解 | 用低秩矩阵近似大矩阵 | 减参数 | 压缩过强损质量 |
| 共享参数 | 多层共享权重 | 减参数 | 表达能力下降 |
| 编译优化 | 图优化、算子融合、后端调度 | 提升实际速度 | 依赖硬件和 runtime |

一句话：

> 蒸馏保质量，量化省内存，剪枝减结构，编译让压缩真的跑得快。

## 知识蒸馏

蒸馏用 teacher 模型提供软标签、回答或推理轨迹，让 student 模型学习。

### 1. Logit 蒸馏

teacher 输出概率分布，student 学习这个分布。

温度 softmax：

```text
p_i = exp(z_i / T) / sum_j exp(z_j / T)
```

蒸馏损失：

```text
L_distill = T^2 * KL(p_teacher^T || p_student^T)
```

通常和真实标签 CE 混合：

```text
L = alpha * CE(y, p_student) + (1 - alpha) * L_distill
```

为什么用温度？

- 温度变大，概率分布更平滑。
- student 能学到“猫比狗更像老虎，但不像汽车”这类暗知识。

### 2. Response 蒸馏

用大模型生成高质量问答数据训练小模型。

适合：

- 指令遵循。
- 领域问答。
- 格式输出。
- 工具调用示例。

风险：

- teacher 幻觉会被 student 学进去。
- 数据分布太窄会过拟合。
- 版权和合规风险。
- 小模型容量不足，学不到复杂推理。

### 3. Reasoning 蒸馏

蒸馏思维链、解题步骤或工具轨迹。

适合：

- 数学。
- 代码。
- 多步任务。
- Agent 工具调用。

注意：

> 推理轨迹不是越长越好。端侧小模型要兼顾质量和输出成本，很多时候蒸馏“短而可验证”的步骤更实用。

## 量化

量化把权重或激活从高精度变成低精度。

### 1. 基本公式

对称量化：

```text
q = round(x / scale)
x_hat = q * scale
```

非对称量化：

```text
q = round(x / scale + zero_point)
x_hat = (q - zero_point) * scale
```

关键概念：

- scale：浮点数和整数之间的比例。
- zero_point：非对称量化的零点。
- group size：一组权重共享 scale 的大小。
- per-tensor：整个张量一个 scale。
- per-channel：每个通道一个 scale，精度更好。
- weight-only：只量化权重。
- weight-activation：权重和激活都量化。

### 2. PTQ 和 QAT

| 方法 | 说明 | 优点 | 风险 |
| --- | --- | --- | --- |
| PTQ | 训练后量化 | 快，不重新训练 | 低比特精度损失大 |
| QAT | 量化感知训练 | 精度更稳 | 训练成本高 |

面试答案：

> PTQ 适合快速部署和试验，QAT 适合对精度要求高、能承担训练成本的场景。端侧低比特模型常先 PTQ，再用业务 eval 判断是否需要 QAT 或少量校准微调。

### 3. GPTQ、AWQ、SmoothQuant

GPTQ：

- 后训练 weight-only 量化。
- 利用二阶信息近似补偿量化误差。
- 常用于 GPU 上 3/4-bit 权重量化。

AWQ：

- Activation-aware Weight Quantization。
- 认为少量重要通道对输出影响大。
- 保护重要权重通道，提升 4-bit 效果。

SmoothQuant：

- 面向 weight-activation 量化。
- 把激活 outlier 的量化难度迁移到权重上。
- 更利于 INT8 权重和激活量化。

面试口语版：

> GPTQ 更像用二阶近似修量化误差，AWQ 更像保护激活敏感的重要通道，SmoothQuant 更关注把激活 outlier 平滑掉，让权重和激活都更容易 INT8。

### 4. GGUF 和 llama.cpp

GGUF 是 llama.cpp 生态常见模型文件格式，适合本地 CPU/GPU 混合推理和多种量化级别。

你不用背所有量化后缀，但要知道：

- Q8 通常质量更稳，文件更大。
- Q4 更省内存，质量风险更大。
- K-quants 是 llama.cpp 生态常见改进量化格式。
- 端侧速度取决于 CPU 指令集、内存带宽、线程数、GPU/NPU 后端和上下文长度。

面试答案：

> GGUF/llama.cpp 适合本地部署和快速验证，优势是生态成熟、量化格式多、CPU 可跑；但生产端侧还要考虑平台后端、模型更新、安全、功耗和业务 eval。

## 剪枝

剪枝删除“不重要”的参数或结构。

### 1. 非结构化剪枝

删除单个权重。

优点：

- 压缩率高。

风险：

- 稀疏模式不规则。
- 没有稀疏 kernel 支持时，不一定加速。

### 2. 结构化剪枝

删除 head、channel、FFN neuron、layer。

优点：

- 更容易实际加速。
- 适合硬件和编译器。

风险：

- 影响模型结构。
- 需要校准或再训练。

### 3. SparseGPT 类思路

SparseGPT 代表一类大模型 one-shot 剪枝方法：在不完整重训练的情况下，利用近似二阶信息对权重剪枝并补偿误差。

面试口语版：

> 剪枝不是“删参数就会快”。如果硬件和 kernel 不支持稀疏计算，非结构化剪枝可能只省存储不省延迟。端侧更偏好结构化剪枝或能被 runtime 利用的稀疏模式。

## 端侧部署栈

常见路线：

```text
PyTorch / HF model
  -> export / convert
  -> quantize
  -> runtime format
  -> backend acceleration
  -> app integration
```

常见工具：

| 工具 | 适合什么 | 面试说法 |
| --- | --- | --- |
| llama.cpp / GGUF | PC/CPU/本地快速部署 | 生态成熟，量化丰富 |
| Ollama | 本地模型管理和服务 | 适合开发和原型验证 |
| ONNX Runtime GenAI | 跨平台推理 | 适合 Windows/移动/边缘生态 |
| ExecuTorch | PyTorch 端侧部署 | 面向移动和边缘设备 |
| MLC LLM | 编译优化、多后端 | 支持多平台和 GPU/WebGPU 等 |
| TensorRT-LLM | NVIDIA GPU 高性能 | 更偏云端/边缘 GPU |

端侧不只是转换格式，还要处理：

- tokenizer。
- chat template。
- KV Cache。
- 采样参数。
- 线程和后端选择。
- 模型热加载。
- 内存峰值。
- 安全策略。
- 更新和回滚。

## 端侧指标

云端常看 QPS、tokens/s、GPU 利用率；端侧还要看：

- 首 token 延迟。
- 每 token 延迟。
- 峰值内存。
- 模型文件大小。
- 冷启动时间。
- 电量/功耗。
- 发热和降频。
- 离线可用。
- 隐私数据是否出端。
- 不同设备兼容性。

面试答案：

> 端侧评估不能只看平均 tokens/s。手机上还要看内存峰值、功耗、发热、冷启动、长时间运行降频和低端机兼容性。

## 端侧 RAG 和混合路由

端侧不一定只能跑一个模型。

常见架构：

```text
local small model
  -> simple task / privacy task / offline task
  -> confidence low or risk high
  -> cloud large model
```

端侧 RAG：

- 本地 embedding。
- 本地小向量库。
- 私有文档只在本机检索。
- 答案可由本地小模型生成，复杂问题升级云端。

混合路由策略：

- 简单任务本地处理。
- 长上下文或复杂推理走云端。
- 隐私字段脱敏后再上云。
- 弱网时降级到本地。
- 高风险任务要求人工确认或云端强模型。

## 量化后效果下降怎么排查

按层排查：

```text
量化格式 -> group size -> calibration data -> outlier
-> sensitive layers -> tokenizer/template -> kernel/runtime
-> task eval -> safety eval
```

常见补救：

- INT4 换 INT8 或混合精度。
- 调 group size。
- 保护 embedding、lm_head、attention output 等敏感层。
- 用更贴近业务的 calibration data。
- 少量 QAT 或 LoRA 修复。
- 数学/代码/安全任务路由到高精度模型。
- 对输出做格式和安全校验。

面试答案：

> 量化后不能只看平均分，要看任务分桶。数学、代码、长上下文、工具参数、安全拒答可能对量化更敏感。评估集要覆盖这些高风险样本。

## 高频问答

### Q1：蒸馏和微调有什么区别？

微调通常是让模型学习标注数据或指令数据；蒸馏强调 teacher-student，把大模型的分布、回答或推理过程迁移给小模型。两者可以结合：先用 teacher 生成数据，再 SFT student。

### Q2：量化一定能加速吗？

不一定。它一定能降低权重存储，但速度取决于硬件是否支持低比特计算、kernel 是否高效、batch 和上下文长度、解量化开销和内存带宽。端侧要实测。

### Q3：剪枝为什么可能不加速？

非结构化稀疏如果没有对应 kernel，硬件仍然按密集矩阵算，可能只省存储不省时间。结构化剪枝更容易带来真实加速。

### Q4：端侧小模型怎么做项目表达？

讲清场景、设备约束、模型选择、压缩方法、部署 runtime、指标和 fallback。比如“在 8GB 内存 PC 上用 GGUF INT4 跑 7B，监控模型大小、峰值内存、TTFT、tokens/s 和业务准确率，复杂问题路由云端”。

### Q5：端侧部署最大风险是什么？

不是单一风险。质量下降、功耗发热、内存峰值、设备碎片化、模型更新、隐私安全、越权工具和云端 fallback 都要考虑。

## 面试前背诵版

端侧小模型的核心是在内存、算力、功耗、延迟和隐私约束下做到“够用”。常见方法包括蒸馏、量化、剪枝、低秩分解和编译优化。蒸馏用 teacher 的 logits、回答或推理轨迹训练 student；量化用 INT8/INT4 等低比特表示权重或激活，PTQ 快但可能掉点，QAT 稳但成本高；GPTQ 用近似二阶补偿量化误差，AWQ 保护激活敏感通道，SmoothQuant 平滑激活 outlier；剪枝要区分非结构化和结构化，只有 runtime/kernel 能利用稀疏才会真的加速。端侧部署要看 TTFT、TPOT、峰值内存、模型大小、冷启动、功耗、发热、设备兼容和业务 eval。
