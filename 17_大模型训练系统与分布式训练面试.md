## 一句话总览

```text
训练显存 = 参数 + 梯度 + optimizer state + activation + 临时 buffer + 通信 buffer

并行目标：
数据并行切 batch
张量并行切单层矩阵
流水线并行切层
ZeRO/FSDP 切参数、梯度、优化器状态
专家并行切 MoE experts
```

一句话背诵：

> 大模型训练系统的核心矛盾是显存、计算和通信。显存不够时先看参数、梯度、Adam 状态和 activation；单卡放不下用 ZeRO/FSDP 切状态，单层太大用 tensor parallel，层数太多用 pipeline parallel，batch 要扩展用 data parallel，MoE 还要 expert parallel。真正上线训练要同时看 MFU、吞吐、通信占比、OOM 位置、checkpoint、容错和评估回归。

## 训练显存怎么估算

训练显存粗分：

| 部分 | 是什么 | 常见规模 |
| --- | --- | --- |
| 参数 weights | 模型权重 | FP16/BF16 约 `2P` bytes |
| 梯度 gradients | 反向传播梯度 | FP16/BF16 约 `2P` bytes |
| optimizer state | Adam 的 m/v、master weights | 常见约 `12P` bytes |
| activation | 前向保存给反向用的中间值 | 和 batch、seq、hidden、layers 相关 |
| temporary buffer | attention、matmul、通信临时区 | 和 kernel、并行策略有关 |
| fragmentation | 显存碎片和框架开销 | 工程上不能忽略 |

其中 `P` 是参数量。混合精度 Adam 的参数相关显存常用粗估：

```text
params: 2P
grads: 2P
master weights: 4P
Adam m: 4P
Adam v: 4P
合计约 16P bytes
```

例子：

```text
7B 全参 Adam 训练，仅参数/梯度/优化器状态：
16 * 7B = 112GB

14B：
16 * 14B = 224GB

这还没算 activation、buffer、checkpoint 和碎片。
```

为什么 LoRA/QLoRA 省显存：

- 不训练全部参数，梯度和 optimizer state 只给 LoRA 参数。
- 冻结基座不需要保存全量 Adam 状态。
- QLoRA 还把冻结基座权重量化存储。

为什么全参训练比推理贵很多：

- 推理主要放权重和 KV Cache。
- 训练除了权重，还要梯度、优化器状态、activation。
- 反向传播还会带来额外计算和中间 buffer。

面试口语：

> 我会先按参数量做粗估。混合精度 Adam 全参训练常用 16 bytes/param 估参数、梯度和优化器状态，再加 activation 和 buffer。比如 7B 就已经约 112GB，所以单卡很难全参训练，需要 ZeRO/FSDP、checkpointing、梯度累积或 LoRA/QLoRA。

## 数据并行 DDP

数据并行的想法：

```text
每张卡一份完整模型
不同卡处理不同 mini-batch
反向后 all-reduce 梯度
每张卡用同步后的梯度更新同样的参数
```

优点：

- 简单稳定。
- 适合模型能完整放进单卡。
- 扩 batch 和吞吐很自然。

缺点：

- 每张卡都保留完整参数、梯度、optimizer state。
- 显存不随卡数下降。
- 通信主要是梯度 all-reduce。
- batch 太大可能影响收敛，需要学习率和 warmup 调整。

DDP 适合：

- 模型单卡能放下。
- 训练 batch 可以扩大。
- 不需要切模型本身。

DDP 不适合：

- 单卡放不下模型或 optimizer state。
- 单层矩阵太大。
- 激活显存太高。

## ZeRO 和 FSDP

ZeRO 的核心思想：数据并行里很多状态在每张卡上重复保存，可以切分。

| 阶段 | 切什么 | 解决什么 |
| --- | --- | --- |
| ZeRO-1 | optimizer state | Adam 状态太大 |
| ZeRO-2 | optimizer state + gradients | 梯度也切分 |
| ZeRO-3 | optimizer state + gradients + parameters | 参数也切分 |

ZeRO-3/FSDP 的直觉：

```text
平时每张卡只保存参数 shard
需要计算某层时 all-gather 拼出该层参数
反向后 reduce-scatter 梯度
optimizer state 也按 shard 更新
```

FSDP 可以理解成 PyTorch 生态里的 fully sharded data parallel。它和 ZeRO-3 思路很接近：切参数、梯度和 optimizer state，用通信换显存。

优点：

- 显著降低每卡参数相关显存。
- 仍然保留数据并行的编程模型。
- 适合全参微调和中大模型训练。

代价：

- all-gather / reduce-scatter 通信增加。
- 小 batch 或网络慢时通信占比高。
- checkpoint 保存和恢复更复杂。
- wrap 粒度、prefetch、reshard 策略会影响性能。

面试口语：

> ZeRO/FSDP 不是减少总显存，而是把重复状态分摊到多卡。ZeRO-1 切 optimizer，ZeRO-2 再切 gradient，ZeRO-3 连 parameter 也切。显存省了，但每层计算前后要 all-gather 和 reduce-scatter，所以会增加通信，对网络和 overlap 很敏感。

## Tensor Parallel

Tensor Parallel，简称 TP，是把单层矩阵计算切到多卡。

Transformer 里主要切：

- attention 的 QKV projection。
- attention output projection。
- MLP 的 up/gate/down projection。

直觉：

```text
一个很大的 Linear 层
按 hidden 或 intermediate 维度切成几块
不同 GPU 算不同 shard
必要时 all-reduce / all-gather 汇总
```

适合：

- 单层矩阵太大，单卡放不下或算不动。
- 想降低单卡权重和 matmul 压力。
- 高速互联机器内，例如 NVLink/NVSwitch。

代价：

- 每层都会有通信。
- TP 通常更依赖单机内高速互联。
- TP degree 太大，通信开销会吃掉收益。

面试口语：

> Tensor parallel 切的是层内矩阵，不是切 batch。它适合单层 hidden 很大、单卡放不下或 matmul 压力太高的情况。代价是每层都有通信，所以一般放在节点内高速互联上，跨节点 TP 会更难。

## Pipeline Parallel

Pipeline Parallel，简称 PP，是把不同层放到不同 GPU。

直觉：

```text
GPU0: layers 0-7
GPU1: layers 8-15
GPU2: layers 16-23
GPU3: layers 24-31
```

为了避免 GPU 空等，会把 batch 切成 micro-batches 流水执行。

核心问题：

- pipeline bubble：流水线填充和排空时有空闲。
- stage imbalance：不同 stage 计算量不均。
- activation 跨 stage 传输。
- 调度复杂，例如 1F1B。

适合：

- 模型层数很多。
- 单机或多机都需要按层切分。
- 只用 TP/ZeRO 仍然放不下。

代价：

- bubble 降低利用率。
- micro-batch 太少，bubble 更严重。
- batch 调度、checkpoint、错误恢复更复杂。

面试口语：

> Pipeline parallel 切的是层。它能把深模型分到多卡，但会有流水线空泡，所以要用 micro-batch 和 1F1B 之类调度提高利用率。PP 的难点是 stage 平衡和 bubble，不是简单把层平均一切就完事。

## 3D 并行怎么组合

3D 并行通常指：

```text
Data Parallel × Tensor Parallel × Pipeline Parallel
```

组合方式：

- TP：节点内切大矩阵。
- PP：跨层切深模型。
- DP：复制若干个 TP×PP 的模型副本，处理不同数据。

例子：

```text
总 GPU = 64
TP = 4
PP = 4
DP = 4
因为 4 * 4 * 4 = 64
```

每个 DP replica 由 `TP * PP = 16` 张卡组成。共有 4 个 replica 处理不同 batch，最后做数据并行梯度同步或 ZeRO/FSDP 相关同步。

怎么选：

- 单层太大：先加 TP。
- 层数太多：加 PP。
- batch 和吞吐不够：加 DP。
- optimizer state 太大：加 ZeRO/FSDP。
- 节点内互联好：TP 放节点内。
- 跨节点带宽有限：避免跨节点高频 TP 通信。

面试回答结构：

> 我会先看模型是否单卡能放下。如果单层矩阵太大，用 TP；层数多导致模型整体放不下，用 PP；想扩吞吐，用 DP；optimizer state 和梯度太占显存，用 ZeRO/FSDP。实际 mapping 要考虑节点内 NVLink 和跨节点 IB，尽量把高频通信放在节点内。

## Activation Checkpointing

反向传播需要前向 activation。activation 显存和下面因素有关：

- batch size。
- sequence length。
- hidden size。
- layer 数。
- 是否保存 attention 中间结果。

activation checkpointing 的做法：

```text
前向时不保存所有中间 activation
只保存少量 checkpoint
反向时重新计算缺失的中间结果
```

优点：

- 明显降低 activation 显存。
- 对长序列、大 batch、深模型很有用。

代价：

- 反向时需要重算，训练变慢。
- checkpoint 粒度会影响速度和显存。
- 随机性、dropout、RNG 状态要处理好。

面试口语：

> Activation checkpointing 是用计算换显存。它不减少参数或 optimizer state，而是减少前向保存的中间激活。长上下文或 batch 较大时 activation 往往很大，所以 checkpointing 很常用，但会增加重算开销。

## 梯度累积和 Global Batch

梯度累积解决的是“单卡 micro-batch 放不下，但想要更大 global batch”。

公式：

```text
global_batch = micro_batch_per_gpu * data_parallel_size * grad_accum_steps
```

做法：

- 多个 micro-batch 前向/反向。
- 梯度不立刻 optimizer step。
- 累积 `grad_accum_steps` 次后再更新。

优点：

- 降低单步显存压力。
- 模拟更大 batch。
- 对显存紧张的微调很常用。

注意：

- loss 要按累积步数缩放。
- 学习率、warmup、weight decay 常要跟 global batch 协调。
- BatchNorm 类模型要小心，但 LLM 通常主要是 LayerNorm/RMSNorm。
- 累积不能提高单步吞吐，甚至可能降低 wall-clock 效率。

## 混合精度：FP16、BF16、FP8

混合精度目标：

- 降低显存。
- 提高 Tensor Core 利用率。
- 提高吞吐。

FP16：

- 精度和动态范围较小。
- 可能梯度下溢。
- 常需要 loss scaling。

BF16：

- exponent 范围接近 FP32。
- 更稳定，很多训练更偏好 BF16。
- mantissa 精度低于 FP16，但动态范围更好。

FP8：

- 更省显存和带宽，训练吞吐潜力更高。
- 需要框架、硬件、scale 管理和数值稳定策略。
- 常见于 Hopper/Transformer Engine 等生态。

面试口语：

> FP16 省显存也快，但动态范围小，容易下溢，所以常用 loss scaling。BF16 动态范围更大，训练更稳。FP8 进一步降低带宽和存储压力，但更依赖硬件和框架，需要处理 scaling 和数值稳定，不能简单说无损替换。

## 通信瓶颈在哪里

常见 collective：

| 操作 | 常见场景 |
| --- | --- |
| all-reduce | DDP 梯度同步、TP 部分汇总 |
| reduce-scatter | ZeRO/FSDP 梯度切分 |
| all-gather | ZeRO-3/FSDP 参数聚合 |
| all-to-all | MoE expert parallel token dispatch |
| broadcast | 参数初始化或同步 |

瓶颈来源：

- 跨节点带宽不足。
- TP degree 太高导致层内通信频繁。
- PP stage 不平衡，bubble 多。
- ZeRO-3/FSDP all-gather 没有和计算 overlap。
- MoE token 分发 all-to-all 负载不均。
- 小 batch 下计算太少，通信占比更高。

排查指标：

- GPU utilization / MFU。
- step time 分解。
- communication time。
- all-reduce / all-gather 时间。
- network bandwidth。
- bubble ratio。
- batch size 和 micro-batch 数。

面试口语：

> 通信瓶颈要按并行策略定位。DDP 主要看梯度 all-reduce，ZeRO/FSDP 看 all-gather 和 reduce-scatter，TP 看每层通信，PP 看 bubble 和 stage imbalance，MoE 看 all-to-all 和负载均衡。不是显存够就一定快。

## MoE 训练和 Expert Parallel

MoE 用很多 experts 扩大总参数量，但每个 token 只激活 top-k experts。

关键概念：

- router/gate：决定 token 送到哪些 expert。
- top-k：每个 token 选几个 expert。
- capacity factor：每个 expert 最多接多少 token。
- load balancing loss：防止 expert 负载极不均。
- expert parallel：不同 GPU 放不同 experts。

MoE 难点：

- 路由塌缩：token 都去少数 experts。
- 负载不均：某些 GPU 很忙，某些空闲。
- all-to-all 通信重。
- expert capacity 溢出导致 token drop 或质量下降。
- 训练不稳定，负载均衡 loss 和主任务 loss 要平衡。

面试口语：

> MoE 的优势是总参数大但每个 token 激活参数少；难点是路由和系统效率。Expert parallel 需要把 token 按 expert 分发到不同 GPU，核心通信是 all-to-all。实际训练要看负载均衡、capacity、token drop、专家利用率和通信开销。

## 训练慢或 OOM 怎么排查

OOM 排查：

```text
参数/optimizer state 太大 -> ZeRO/FSDP、LoRA、optimizer offload
activation 太大 -> checkpointing、减小 seq/batch、FlashAttention
临时 buffer 太大 -> 调 kernel、分块、减少 peak
碎片 -> empty cache 不能根治，关注 allocator、batch shape、重启
```

训练慢排查：

```text
GPU 利用率低 -> dataloader / CPU / IO / 小 batch
通信占比高 -> DP/TP/PP/ZeRO 配置、跨节点带宽、overlap
bubble 高 -> micro-batch 数、stage balance、PP schedule
kernel 慢 -> FlashAttention、fused ops、Transformer Engine
重算太多 -> checkpoint 粒度
```

Loss 异常排查：

- FP16 溢出/下溢。
- learning rate 太大。
- loss scaling 异常。
- 梯度裁剪缺失。
- 数据有坏样本。
- mask 或 label shift 错。
- DDP 梯度同步或累积缩放错。

## 训练项目怎么讲

如果你有训练或微调项目，按这个顺序讲：

```text
模型规模 -> 数据规模 -> 目标 -> 显存估算 -> 并行/省显存策略
-> batch 和学习率 -> 训练稳定性 -> 评估 -> checkpoint -> 成本
```

项目证据：

- 参数量、序列长度、global batch。
- 训练卡数、GPU 型号、显存。
- 用了 LoRA/全参、FSDP/DeepSpeed、checkpointing、BF16。
- step time、tokens/s、GPU 利用率。
- 训练曲线、验证集指标、bad case。
- checkpoint 保存、恢复、断点续训。

面试高分点：

- 能解释为什么选择 LoRA 而不是全参。
- 能估算全参训练显存，并说明为什么需要 ZeRO/FSDP。
- 能说明 batch、学习率、warmup、梯度累积的关系。
- 能说出训练失败时怎么定位 OOM、loss nan、吞吐低。

## 面试常见高压问题

### Q1：14B 模型 FP16 加载和全参 Adam 训练大概多少显存？

FP16 加载权重约 `14B * 2 = 28GB`，实际还要加 buffer。全参 Adam 混合精度训练仅参数、梯度和优化器状态可粗估 `14B * 16 = 224GB`，还没算 activation 和通信 buffer。所以 14B 全参训练通常需要多卡切分或用 ZeRO/FSDP，也可以改 LoRA/QLoRA 降低训练状态。

### Q2：ZeRO-3 为什么会变慢？

ZeRO-3 切了参数，显存更省，但每层计算前要 all-gather 参数，反向后要 reduce-scatter 梯度。如果网络慢、batch 小、overlap 不好或 wrap 粒度不合适，通信开销会明显增加，所以可能比 ZeRO-2 慢。

### Q3：TP 和 PP 怎么选？

TP 切单层矩阵，适合 hidden/intermediate 很大、单层放不下或算不动；PP 切层，适合层数多、模型整体太深。TP 通信频繁，最好放节点内高速互联；PP 有 bubble，要靠 micro-batch 和调度降低空泡。

### Q4：显存不够时先用什么？

先定位显存主要花在哪里。如果是 activation，先用 activation checkpointing、减 batch/seq、FlashAttention；如果是 optimizer state 和梯度，用 ZeRO/FSDP 或 LoRA；如果只是想大 batch，用梯度累积；如果单层都放不下，考虑 TP。

### Q5：训练吞吐低怎么定位？

看 step time 分解、GPU 利用率、dataloader、通信时间、micro-batch、PP bubble、kernel 时间和网络带宽。根据并行策略定位：DDP 看 all-reduce，FSDP/ZeRO 看 all-gather/reduce-scatter，TP 看层内通信，MoE 看 all-to-all。

## 面试前背诵版

训练显存由参数、梯度、Adam 状态、activation、buffer 和碎片组成；混合精度 Adam 全参训练可粗估 16 bytes/param。DDP 切 batch，但每卡保存完整模型状态；ZeRO/FSDP 切 optimizer、gradient、parameter，用通信换显存；TP 切层内矩阵，PP 切层，3D 并行组合 DP/TP/PP。Activation checkpointing 用重算换激活显存，梯度累积用多次 micro-batch 模拟大 global batch，BF16 比 FP16 更稳，FP8 更依赖硬件和 scaling。训练慢要看 GPU 利用率、通信、bubble、kernel、dataloader；OOM 要先判断是参数状态还是 activation。
