# 大模型数据工程与数据集构建面试

这一章面向大模型算法、SFT/对齐、RAG/Agent 应用、算法评测和项目深挖。很多同学能背 LoRA、DPO、RAG，但一被问“数据怎么来、怎么清洗、怎么评估、怎么防污染”，就容易空。

你要能回答：

- 预训练、SFT、偏好对齐、RAG eval 分别需要什么数据。
- 数据清洗为什么比堆量更重要。
- 去重、质量过滤、PII 脱敏、污染检测怎么做。
- 合成数据有什么用，风险是什么。
- 数据配比、课程学习、难例挖掘怎么讲。
- 如果给你一批业务数据，你怎么构造成可训练/可评估数据集。

## 一句话总览

大模型数据工程不是“收集一堆文本”，而是把数据变成可训练、可评估、可追溯、可迭代的资产。

```text
数据源 -> 清洗 -> 去重 -> 质量过滤 -> 脱敏/合规
-> 格式化 -> 标注/合成 -> 划分 train/valid/test
-> 训练/评估 -> bad case 回流 -> 版本管理
```

面试口语版：

> 模型能力上限很大程度由数据决定。预训练看规模和覆盖，SFT 看指令质量和格式，偏好对齐看 chosen/rejected 是否稳定，RAG eval 看 gold evidence 和 reference answer 是否可信。数据工程的核心是质量、覆盖、去重、防污染和可追溯。

## 数据类型地图

| 阶段 | 数据形态 | 目标 | 高频风险 |
| --- | --- | --- | --- |
| 预训练 | 大规模网页、书籍、代码、论文、论坛 | 学语言、知识、推理模式 | 低质、重复、污染、版权、PII |
| Continue pretrain | 领域语料 | 补领域语言和知识 | 灾难性遗忘、领域过窄 |
| SFT | instruction/input/output | 学指令遵循和任务格式 | 回答低质、格式混乱、loss mask 错 |
| 偏好对齐 | prompt + chosen/rejected | 学偏好和安全边界 | 标注不一致、长度偏置、偏好噪声 |
| RAG eval | query + gold evidence + reference answer | 评估检索和生成链路 | gold 不准、答案不可验证 |
| Agent eval | goal + expected tool trace | 评估工具使用和任务完成 | 工具轨迹标注难、环境不稳定 |
| 安全红队 | attack input + safe behavior | 测拒答和权限 | 只测 jailbreak，不测系统链路 |

## 预训练数据

预训练数据强调：

- 规模。
- 多样性。
- 质量。
- 去重。
- 低污染。
- 合规。

常见来源：

- 网页文本。
- 书籍和百科。
- 代码。
- 学术论文。
- 问答论坛。
- 多语言语料。
- 数学和代码数据。

### 质量过滤

常见过滤信号：

- 文本长度过短或过长。
- 重复字符、乱码、模板页。
- 广告、导航、垃圾内容。
- 语言识别不符合目标。
- 困惑度过高或过低。
- 毒性、成人、违法内容。
- PII 和敏感信息。

面试答案：

> 预训练不是数据越多越好。低质和重复数据会浪费训练预算，污染 benchmark 会造成虚高，PII 和版权问题会带来合规风险。现代数据工程通常会做去重、质量打分、语言过滤、PII 过滤和污染检测。

### 去重

去重分几层：

- exact dedup：完全重复。
- near dedup：近似重复。
- document-level dedup：文档级。
- paragraph-level dedup：段落级。
- benchmark decontamination：去掉和评测集重叠的内容。

常见方法：

- hash。
- SimHash。
- MinHash。
- n-gram overlap。
- embedding similarity。

为什么重要：

- 降低训练浪费。
- 减少模型记忆重复样本。
- 防止验证集泄露。
- 减少 benchmark contamination。

## SFT 数据

SFT 数据通常形如：

```json
{
  "instruction": "请把下面内容总结成三点",
  "input": "...",
  "output": "1. ...\n2. ...\n3. ..."
}
```

或者 chat 格式：

```json
[
  {"role": "system", "content": "..."},
  {"role": "user", "content": "..."},
  {"role": "assistant", "content": "..."}
]
```

### 好的 SFT 数据

- 指令清楚。
- 答案正确。
- 格式稳定。
- 覆盖真实任务。
- 有拒答和边界样本。
- 多轮对话角色清晰。
- 不包含隐私和违法内容。

### SFT 数据常见问题

- instruction 太泛。
- output 过长但没信息。
- answer 事实错误。
- 同一任务格式不统一。
- prompt 和 answer 角色边界错。
- 把用户输入也算 loss。
- 合成数据风格过单一。

面试口语版：

> SFT 数据少但影响行为很大。高质量 SFT 更像“教模型怎么回答”，所以格式、边界、拒答、任务覆盖和 loss mask 都很关键。

## 偏好数据

偏好数据形如：

```text
prompt
chosen response
rejected response
preference reason
```

来源：

- 人工标注。
- 线上 A/B。
- 用户反馈。
- 大模型 judge 预筛。
- 合成偏好数据。

质量关键：

- chosen 真的比 rejected 好。
- 标注标准一致。
- 避免长度偏置。
- 覆盖安全、事实、格式、帮助性。
- 有标注原因，便于审计。

常见噪声：

- 标注员偏好不一致。
- 更长答案被误判为更好。
- chosen/rejected 差异太小。
- 两个答案都错。
- 偏好标准和业务目标不一致。

面试答案：

> DPO/RLHF 不只是算法，偏好数据质量决定上限。我要看标注协议、inter-annotator agreement、长度分布、拒答样本、安全样本和 bad case 分桶。

## 合成数据

合成数据是让模型生成训练或评估样本。

用途：

- 扩充长尾任务。
- 生成指令数据。
- 构造偏好对。
- 生成 reasoning traces。
- 生成红队攻击样本。
- 生成 RAG query。

典型流程：

```text
seed tasks / docs
-> teacher model generate
-> rule filter / model filter
-> dedup
-> human spot-check
-> train/eval split
```

风险：

- teacher 幻觉。
- 风格单一。
- 难度不真实。
- 数据泄露。
- 模型自我强化错误。
- eval 被 synthetic bias 污染。

面试答案：

> 合成数据能补覆盖，但不能无脑加入。要做过滤、去重、人工抽检和真实数据校准。训练集可以多用合成，核心评估集最好保留人工或真实业务样本。

## 数据配比和课程学习

数据配比决定模型学什么。

例子：

- 代码数据多，代码能力提升，但可能影响自然语言风格。
- 数学推理数据多，推理提升，但可能输出冗长。
- 安全拒答数据多，安全提升，但误拒可能上升。
- 领域数据过多，领域表现提升，但通用能力可能下降。

常见策略：

- 按任务重要性采样。
- 按质量分加权。
- 对长尾任务 oversampling。
- 难例挖掘。
- 阶段式训练：通用 -> 领域 -> 指令 -> 偏好。
- replay 通用数据，缓解遗忘。

面试口语版：

> 数据配比不是拍脑袋。要用验证集分桶看效果：通用能力、领域任务、格式遵循、安全、拒答、数学/代码分别是否变好或变差。

## 数据污染和泄露

数据污染常见于：

- 训练集包含 benchmark 题目或答案。
- SFT 数据包含测试集样本。
- RAG eval 的 reference answer 出现在 prompt。
- 同一用户/文档同时进 train 和 test。
- 多轮对话切分时上下文泄露。

检测方法：

- exact match。
- n-gram overlap。
- MinHash。
- embedding similarity。
- benchmark blacklist。
- 按 source/user/time 做 split。

面试答案：

> 如果测试集泄露，评估会虚高。尤其大模型容易记忆，不能只随机切分。业务数据最好按时间、用户、文档或来源切分，保证测试集能代表未来数据。

## RAG / Agent Eval 数据

RAG eval 样本：

```text
query
query_type
gold_doc_id / gold_chunk_id
reference_answer
must_cite
should_refuse
permission_label
```

Agent eval 样本：

```text
goal
initial_state
available_tools
expected_tool_trace
expected_arguments
success_criteria
risk_level
```

构造重点：

- gold evidence 要可验证。
- reference answer 不要靠模型猜。
- 应拒答样本不能缺。
- 工具失败和权限不足样本要覆盖。
- 每个 bad case 要能归因：检索、rerank、生成、工具、权限、数据。

## 数据版本化

数据要像代码一样版本化。

记录：

- 数据来源。
- 抽取时间。
- 清洗规则版本。
- 去重规则版本。
- 标注协议。
- 标注人员或模型。
- 样本 schema。
- train/valid/test split。
- 数据 hash。

为什么：

- 复现实验。
- 解释效果变化。
- 回滚坏数据。
- 审计合规。
- 跟踪 bad case 是否修复。

面试答案：

> 如果一次训练效果变差，没有数据版本和清洗规则记录，就很难判断是模型参数、数据配比、标注噪声还是评估集变化导致的。

## 给你一批业务数据，怎么做

可以按这个流程答：

1. 明确任务：问答、抽取、分类、摘要、工具调用还是对齐。
2. 数据审计：来源、权限、隐私、格式、规模、时间范围。
3. 清洗：去重、去低质、脱敏、统一 schema。
4. 划分：按用户/时间/文档切 train/valid/test，防泄露。
5. 构造：SFT、偏好、RAG eval 或 Agent eval。
6. 标注：规则、人工、模型辅助，保留标注理由。
7. 评估：自动指标 + 人工抽检 + bad case 分桶。
8. 训练/上线：小步实验、灰度、监控、回流。

面试口语版：

> 我不会拿到业务数据就直接微调。先做数据审计和 eval，再判断 RAG、SFT、DPO 还是 prompt 就够。数据闭环比单次训练更重要。

## 高频问答

### Q1：数据量和数据质量哪个更重要？

预训练阶段规模很重要，但低质数据会浪费算力；SFT/偏好阶段质量更关键，少量高质量数据可能比大量噪声数据更有效。

### Q2：怎么判断数据清洗有效？

看清洗前后验证集效果、重复率、低质率、PII 命中率、benchmark overlap、训练 loss 曲线、bad case 分布，而不是只看删了多少数据。

### Q3：合成数据能不能用于评估集？

可以辅助，但核心评估集最好有真实业务或人工标注样本。合成 eval 容易带 teacher 偏差，导致评估和真实用户分布不一致。

### Q4：怎么做偏好标注协议？

写清维度：正确性、帮助性、简洁性、格式、安全、引用证据。给正反例，要求标注理由，并抽样复核一致性。

### Q5：数据闭环怎么讲进项目？

讲“评估集 -> bad case -> 数据补充/清洗 -> 训练或索引更新 -> 回归评估 -> 灰度”。不要只说“我收集了一些数据”。

## 面试前背诵版

大模型数据工程要按数据类型讲：预训练数据重规模、多样性、质量、去重和污染检测；SFT 数据重指令清晰、答案正确、格式稳定和 loss mask；偏好数据重 chosen/rejected 质量、标注一致性和长度偏置；RAG/Agent eval 重 gold evidence、reference answer、工具轨迹和安全样本。数据清洗包括去重、低质过滤、PII 脱敏、格式统一、benchmark decontamination。合成数据能补长尾，但要过滤、去重、人工抽检和真实数据校准。数据版本化用于复现实验、定位回归和支持上线回滚。
