# 大模型评测与实验设计面试

这一章面向大模型算法、Post-training、RAG/Agent 应用、模型平台和算法评测岗。面试里“怎么评测”通常不是让你背几个 benchmark，而是看你能不能把模型能力、业务目标、数据集、指标、judge、ablation、上线回归讲成闭环。

Reasoning model、RLVR、verifier 和 test-time scaling 的评估可以配合 [29_ReasoningPostTraining_RLVR_Verifier_TestTimeScaling面试.md](29_ReasoningPostTraining_RLVR_Verifier_TestTimeScaling面试.md) 看，重点补 pass@1/pass@k、cons@k、verifier accuracy、cost per solved problem 和 reward hacking。

你需要能回答：

- 公开 benchmark 和私有 eval 分别解决什么。
- LLM-as-judge 怎么设计，偏置怎么控制。
- RAG、Agent、函数调用、安全、推理部署分别怎么评估。
- ablation 怎么做，怎么证明某个模块真的有用。
- 线上 A/B、灰度、回归测试和 bad case 回流怎么接起来。

## 一句话总览

```text
目标定义 -> 任务分桶 -> 样本构造 -> 指标/裁判 -> 评测协议
-> baseline 对比 -> ablation -> bad case 分析 -> 上线门禁
-> 线上反馈 -> eval set 版本迭代
```

一句话背诵：

> 大模型评测不是只跑榜单。公开 benchmark 看通用能力，私有 eval 看业务可靠性；自动指标看可重复，LLM-as-judge 看开放生成质量，人工评审负责校准。真正有说服力的评测要有固定数据版本、明确 rubric、baseline、ablation、置信度、bad case 分类和上线回归。

## 评测对象地图

| 评测对象 | 重点问题 | 常用评测 |
| --- | --- | --- |
| 基座模型 | 知识、推理、数学、代码、长上下文 | MMLU/MMLU-Pro、GPQA、HumanEval/MBPP、LongBench |
| 指令模型 | 是否听指令、回答是否有帮助 | MT-Bench、AlpacaEval、Arena 类 pairwise |
| Post-training | SFT/DPO/GRPO 后是否提升 | 私有任务集、pairwise win rate、格式遵循、安全回归 |
| RAG | 检索到没、证据对不对、答案是否忠实 | Recall@k、MRR、context precision/recall、faithfulness、引用正确率 |
| Agent | 是否完成任务、工具调用是否正确 | task success、tool call accuracy、argument accuracy、step cost |
| Function Calling | 工具选择和参数是否对 | function name accuracy、schema exact match、argument F1 |
| 安全 | 是否越权、泄露、被注入 | jailbreak success rate、policy violation rate、红队通过率 |
| 推理部署 | 延迟、吞吐、成本、稳定性 | TTFT、TPOT、tokens/s、P95/P99、错误率、成本/千 token |
| 线上产品 | 用户是否真的受益 | 解决率、转人工率、满意度、留存、投诉率、A/B |

面试要先问清楚“评测谁”：同一句“评估效果”，如果对象是 RAG、SFT、Agent、端侧部署或模型网关，指标完全不同。

## 公开 Benchmark 怎么讲

公开 benchmark 的价值：

- 横向比较模型通用能力。
- 快速发现短板，例如数学、代码、知识、推理、长上下文。
- 作为模型选型的第一层过滤。
- 给私有 eval 设计提供维度参考。

常见 benchmark：

| Benchmark | 大致考什么 | 面试表达 |
| --- | --- | --- |
| MMLU / MMLU-Pro | 多学科知识与推理 | 看知识覆盖和复杂选择题能力，MMLU-Pro 更强调难度和鲁棒性 |
| GPQA | 高难科学问答 | 更接近专家级知识推理，能区分强模型 |
| HumanEval / MBPP | 代码生成 | 看 pass@k，但不能等价于真实软件工程 |
| SWE-bench | 真实 GitHub issue 修复 | 更接近 agentic coding 和工程修复能力 |
| MT-Bench | 多轮对话 | 常配合 LLM judge，看 helpfulness 和 follow-up 能力 |
| AlpacaEval | 指令跟随 pairwise 评估 | 常看 win rate，注意长度偏置和 judge 偏置 |
| Chatbot Arena | 人类偏好对战 | 看真实用户偏好，通常用 Elo/Bradley-Terry 类排名 |
| GAIA | 多步工具和真实世界任务 | 更像通用助手/Agent 评测 |
| BFCL | Function Calling | 看工具选择、参数、schema 和多轮函数调用 |
| HELM | 多维度评测框架 | 强调准确率、鲁棒性、公平性、效率等多维评估 |

局限：

- 容易被污染，模型可能见过题。
- 不能覆盖你的业务数据、约束和风险。
- 单个分数会掩盖分桶差异。
- Leaderboard 可能被 prompt、采样参数和 judge 策略影响。
- 开放生成题中，长度、格式、位置、模型自偏好都会影响裁判。

面试回答：

> 我会把公开 benchmark 当第一层筛选，而不是最终证据。比如模型选型可以先看 MMLU-Pro、GPQA、HumanEval、SWE-bench 和 Arena 类结果，但上线前一定要做私有 eval：按业务场景分桶，固定数据版本，设置 baseline 和上线门槛，分析 bad case，避免只拿公开榜单分数证明业务效果。

Prompt 改动、结构化输出、schema 合规和 parser/retry 的评估见：[27_PromptEngineering_结构化输出与ConstrainedDecoding面试.md](27_PromptEngineering_结构化输出与ConstrainedDecoding面试.md)。

## 私有 Eval 怎么构造

私有 eval 的目标不是“凑一批题”，而是让你知道模型在真实业务里会不会出事。

构造流程：

```text
业务目标 -> 用户任务 -> 风险场景 -> 样本分桶
-> 标注 gold evidence / reference answer / rubric
-> 训练集隔离 -> 评测脚本固定 -> 人工抽检
-> 版本化 -> bad case 回流
```

样本来源：

- 真实线上 query，脱敏后抽样。
- 历史客服、搜索、工单、文档问答。
- 产品经理/业务专家整理的高频任务。
- 线上 bad case 和投诉样本。
- 对抗样本，例如越权、诱导泄露、无答案、冲突证据。
- 合成样本，但要人工抽检，核心评估集不能完全依赖合成。

分桶维度：

- 任务类型：问答、总结、改写、抽取、决策、工具调用。
- 难度：简单事实、多跳推理、长文档、歧义问题。
- 数据来源：文档、表格、数据库、网页、图片。
- 风险：安全、隐私、越权、幻觉、错误引用。
- 用户类型：新手、专家、内部员工、外部客户。
- 时效性：稳定知识、动态数据、实时状态。

每条样本建议字段：

```yaml
id: eval_001
task_type: rag_qa
difficulty: hard
user_query: "..."
gold_evidence:
  - doc_id: "policy_2026_v3"
    span: "..."
reference_answer: "..."
rubric:
  correctness: "必须包含 A/B，不能说 C"
  faithfulness: "答案只能依据 gold evidence"
  refusal: "证据不足时应拒答"
risk_tags: ["hallucination", "permission"]
source_split: "test"
```

注意事项：

- test set 不能参与 prompt 调参、SFT、DPO 或 few-shot 示例选择。
- valid set 用来调参，test set 用来最终报告。
- RAG 评估要防止 train/test 文档同源泄露。
- 线上 bad case 可以回流进 train 或 valid，但进入 test 要有版本记录。
- 小数据集要优先保证代表性和高风险覆盖，而不是随机大而空。

## 指标怎么选

指标分三层：

1. 组件指标：检索、rerank、工具调用、模型输出各自是否工作。
2. 端到端指标：用户任务是否完成。
3. 业务指标：是否真的降低成本、提升满意度或提升转化。

常见指标：

| 场景 | 指标 | 解释 |
| --- | --- | --- |
| 分类/抽取 | Accuracy、Precision、Recall、F1 | 标签或字段是否正确 |
| 搜索/RAG 检索 | Recall@k、MRR、NDCG | gold 文档是否被召回，排序是否靠前 |
| RAG 生成 | faithfulness、answer relevance、引用正确率 | 答案是否忠实证据、是否答到问题 |
| 开放生成 | win rate、平均分、rubric pass rate | 用人工或 LLM judge 打分 |
| 代码 | pass@k、unit test pass rate | 代码是否通过测试 |
| Agent | task success、tool accuracy、step count | 是否完成任务、工具轨迹是否合理 |
| 安全 | violation rate、jailbreak success rate | 是否违反策略 |
| 服务 | TTFT、TPOT、P95/P99、error rate | 性能和稳定性 |
| 产品 | solve rate、CSAT、转人工率、投诉率 | 业务价值 |

不要只报一个平均分：

- 平均分可能掩盖高风险场景。
- 要按任务、难度、来源、风险、语言、长度分桶。
- 关键业务场景可以设置 hard gate，例如安全违规率必须为 0 或低于阈值。

## LLM-as-Judge 怎么设计

LLM-as-judge 适合开放生成、对话质量、总结质量、RAG 忠实性、Agent 轨迹解释等场景。它的优势是便宜、快、可重复；问题是会有偏置。

基本流程：

```text
输入样本 -> 模型输出 A/B -> judge prompt + rubric
-> pointwise 或 pairwise 打分 -> 多次/多 judge 聚合
-> 人工抽检校准 -> 记录 judge 版本
```

Pointwise：

- 对单个回答打分。
- 优点：实现简单，适合 rubric 明确的任务。
- 缺点：不同 judge 的绝对分数不稳定。

Pairwise：

- 比较 A/B 哪个更好。
- 优点：更接近偏好判断，适合模型对比。
- 缺点：需要控制位置偏置，成本更高。

常见偏置：

- 位置偏置：更偏向第一个或第二个答案。
- 长度偏置：更长回答看起来更“有料”。
- 格式偏置：结构化回答更讨喜，但未必更正确。
- 自增强偏置：judge 偏向同系列模型风格。
- 过度自信：judge 可能忽略事实错误。
- rubric 漂移：prompt 或模型版本变化导致分数不可比。

控制方法：

- A/B 顺序随机，并做交换位置复评。
- 使用明确 rubric，而不是“哪个更好”。
- 要求 judge 输出理由和可检查的错误类型。
- 多 judge 或多次采样取多数/均值。
- 加入 gold answer 或 gold evidence，降低凭空评价。
- 对关键样本做人工抽检，计算 judge-human agreement。
- 固定 judge 模型、prompt、temperature 和评测脚本版本。

面试回答：

> 我会用 LLM-as-judge 做开放生成的规模化初筛，但不会完全相信它。评测时会采用 pairwise 或 rubric pointwise，控制位置和长度偏置；同时固定 judge 版本，对部分样本人工复核，统计和人工的一致率。上线门禁里，安全、合规和关键事实类问题不能只靠 judge，要有规则、证据校验或人工抽检。

## Win Rate、Elo 和 Bradley-Terry

开放生成模型对比常用 pairwise 偏好。

Win rate：

```text
win_rate(A over B) = A 胜场数 / 有效比较总数
```

如果有 tie：

```text
win_rate = (wins + 0.5 * ties) / total
```

Elo 思想：

- 两个模型对战，强者预期胜率更高。
- 如果弱者赢了，分数变化更大。
- Arena 类榜单常用对战偏好建模。

Bradley-Terry 思想：

```text
P(A beats B) = exp(s_A) / (exp(s_A) + exp(s_B))
```

其中 `s_A`、`s_B` 是模型隐含能力分。它把大量 pairwise 比较转成一个全局排名。

面试不用推很深，但要知道：

- win rate 适合固定 A/B 对比。
- Elo/BT 适合多模型多人群对战排名。
- 排名要看置信区间，不能只看小数点后差异。
- 数据分布变化后，历史排名不一定代表你的业务。

## RAG 怎么评测

RAG 不能只看最终回答，要拆链路。

检索层：

- Recall@k：gold 文档是否在前 k 个候选里。
- MRR：第一个正确文档排得多靠前。
- NDCG：多个相关文档时看排序质量。

上下文层：

- context precision：检索上下文里有多少是有用的。
- context recall：需要的证据是否被覆盖。
- context relevance：上下文是否和问题相关。

生成层：

- faithfulness：答案是否忠实上下文。
- answer relevance：是否回答用户问题。
- citation accuracy：引用是否真的支持答案。
- refusal correctness：证据不足时是否拒答。

端到端：

- task success。
- 用户满意度。
- 转人工率。
- 平均轮次。
- 响应延迟和成本。

排查链路：

```text
答案错 -> 看 gold 文档是否被召回
没召回 -> 查 chunk / embedding / hybrid / rerank
召回了但答案错 -> 查 prompt / context packing / 模型能力 / 引用约束
答案编造 -> 查无答案样本、拒答策略、faithfulness judge
```

## Agent 和 Function Calling 怎么评测

Agent 评测要看“任务是否完成”和“过程是否安全”。

任务级指标：

- task success rate。
- end-to-end solve rate。
- 平均步骤数。
- 成本和耗时。
- 重试次数。

工具级指标：

- tool selection accuracy。
- argument exact match / F1。
- schema validity。
- tool execution success。
- permission violation rate。

轨迹级指标：

- 是否遵守最大步数。
- 是否重复调用同一工具。
- 是否泄露 tool result。
- 是否在需要人工确认时停下来。
- 是否把工具错误当成真实结果。

Function Calling 样本字段：

```yaml
user_query: "帮我查一下订单 123 的物流"
available_tools:
  - get_order_status(order_id)
expected_tool: "get_order_status"
expected_args:
  order_id: "123"
forbidden_tools:
  - refund_order
success_criteria: "只查询物流，不触发退款"
```

面试回答：

> Agent 评测不能只看最终文本。我会构造任务集，记录 expected tool trace、参数、权限、最大步数和成功条件。指标上同时看 task success、tool selection、argument accuracy、schema validity、平均步骤数、成本和安全违规率。线上还要保留 trace，方便 bad case 回放。

## 安全和红队评测

安全评测要和业务权限绑定。

维度：

- prompt injection。
- jailbreak。
- PII 泄露。
- 越权访问。
- 工具滥用。
- RAG 数据投毒。
- 不安全建议。
- 合规拒答。

指标：

- attack success rate。
- policy violation rate。
- refusal false positive / false negative。
- sensitive info leakage rate。
- unsafe tool call rate。
- red-team pass rate。

注意：

- 安全样本不能只用公开 jailbreak prompt，要结合业务工具和权限。
- 拒答不是越多越好，要同时看过拒和漏拒。
- 安全 eval 是上线门禁，不是可选分析项。

## 实验设计和 Ablation

Ablation 的目标是证明“哪个模块带来了提升”。

基本写法：

```text
baseline
+ 改动 A
+ 改动 B
+ 改动 C
full system
```

RAG ablation 示例：

| 实验 | 目的 |
| --- | --- |
| vector only | baseline |
| + BM25 hybrid | 看关键词召回是否提升 |
| + reranker | 看排序是否提升 |
| + query rewrite | 看复杂问题是否提升 |
| + citation/refusal prompt | 看幻觉是否下降 |

SFT ablation 示例：

| 实验 | 目的 |
| --- | --- |
| base model | 原始能力 |
| + SFT | 指令跟随是否提升 |
| + clean data | 数据清洗贡献 |
| + hard negatives | 边界和拒答是否提升 |
| + DPO | 偏好质量是否提升 |

实验注意：

- 一次只改一个关键变量。
- 固定模型、prompt、采样参数、数据版本。
- 同时看总体指标和分桶指标。
- 报告 bad case，不只报提升。
- 小样本要给置信区间或至少说明抽样误差。

面试回答：

> 我会先定 baseline，再逐项加入模块做 ablation。比如 RAG 先跑向量召回，再加 BM25、rerank、query rewrite、引用约束，分别看 Recall@k、MRR、faithfulness 和延迟成本。这样能证明提升来自哪个模块，而不是凭主观感觉说系统变好了。

## 统计显著性和置信区间

面试不会要求复杂统计推导，但要知道“小样本差 1 分”不一定是真提升。

常见做法：

- 固定 test set，多次运行估计方差。
- bootstrap 估计置信区间。
- pairwise 任务看 win/tie/loss 分布。
- A/B 看样本量、转化率、置信区间和业务风险。
- 对高风险指标设置硬阈值。

口语版：

> 如果两个方案只差 1%-2%，我不会直接宣布新方案更好。我会看样本量、分桶结果和置信区间；如果是生成类任务，还会看 win/tie/loss 和人工抽检。对安全违规、越权调用这类指标，则宁可用硬门槛，而不是平均分掩盖风险。

## 上线回归和 EvalOps

上线前评测：

- 固定 regression eval set。
- 对比当前线上版本。
- 看核心指标、成本、延迟、安全。
- 检查高风险 bad case 是否复现。
- 生成评测报告和上线结论。

上线后监控：

- 用户反馈。
- 转人工率。
- 投诉和风险事件。
- 工具错误率。
- RAG 无答案率。
- judge 分数漂移。
- 延迟和成本。

版本化要记录：

- eval set 版本。
- 数据来源和脱敏规则。
- prompt 和 system message。
- 模型版本。
- judge 模型和 judge prompt。
- 采样参数。
- 评测脚本 commit。
- 结果和 bad case 标签。

EvalOps 简化闭环：

```text
offline eval -> regression gate -> canary / gray
-> online metrics -> human feedback -> bad case labeling
-> eval set update -> prompt/model/data iteration
```

## 面试常见高压问题

### Q1：公开 benchmark 高但业务效果差，怎么办？

先承认公开榜单和业务分布不同。排查：业务任务是否被 benchmark 覆盖；私有 eval 是否有代表性；RAG/工具/权限链路是否拖后腿；prompt、数据、延迟和安全策略是否影响体验。解决：构建私有 eval，按场景分桶，做 baseline、ablation 和 bad case 回流。

### Q2：LLM-as-judge 能不能替代人工？

不能完全替代。它适合规模化初筛和回归，但要用人工校准。关键场景要抽检，安全合规要有规则和人工兜底。要记录 judge 版本，避免模型升级后分数不可比。

### Q3：RAG 项目只说“准确率提升”有什么问题？

太粗。RAG 要拆检索、上下文、生成、引用、拒答和端到端。准确率提升可能来自模型更会猜，不一定更忠实。应该报告 Recall@k、MRR、faithfulness、引用正确率、无答案拒答率、延迟和成本。

### Q4：怎么证明 reranker 有用？

做 ablation：vector only、hybrid、hybrid+reranker。看 MRR/NDCG、top-k gold 文档位置、faithfulness 和端到端解决率，同时看 reranker 带来的延迟成本。如果只提升很小但 P95 延迟大涨，可能不值得上线。

### Q5：如何做模型替换的上线门禁？

新模型必须通过 regression eval：通用能力不退化，核心业务分桶提升或持平，安全违规不升高，延迟和成本在预算内。再做灰度，监控转人工率、投诉、错误率和延迟，出现高风险 bad case 可以回滚模型、prompt、retriever 或工具策略。

## 面试前背诵版

大模型评测要按对象拆：基座模型看公开 benchmark，业务应用看私有 eval，RAG 拆检索和生成，Agent 拆任务成功和工具轨迹，安全看违规率和红队，部署看延迟吞吐成本。私有 eval 要有分桶样本、gold evidence/reference answer/rubric、固定数据版本、baseline、ablation、bad case 和上线门禁。LLM-as-judge 适合规模化开放生成评测，但要控制位置偏置、长度偏置和 judge 版本漂移，并用人工抽检校准。
