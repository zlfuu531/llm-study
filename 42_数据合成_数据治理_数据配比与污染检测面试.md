- DataComp-LM、FineWeb、Dolma、Nemotron-CC 这类数据工作给面试带来什么启发。
- 合成数据、Self-Instruct、Magpie、UltraFeedback 怎么讲。
- 数据配比、难例挖掘、可验证数据和 verifier filter 怎么做。
- benchmark contamination、train/test leakage、decontamination 怎么防。
- 数据版本、data card、质量看板和 bad case 回流怎么支撑项目复盘。

## 一句话总览

大模型数据工程进阶不是“多爬点数据”，而是把数据当成可实验、可审计、可回滚的系统。

```text
source mix -> quality filter -> dedup -> decontamination
-> schema/template -> synthetic generation -> verifier/judge filter
-> mixture schedule -> train/eval split -> data version
-> training/eval -> bad case -> data recipe update
```

面试口语版：

> 现在的大模型数据准备更像做 data recipe。预训练数据看覆盖、质量、去重和污染；SFT/偏好数据看任务分布、格式、可验证性和偏好一致性；合成数据看 teacher 质量、过滤和真实样本校准。最终要能用分桶 eval 证明这版数据配方真的让目标能力变好。

## 1. 为什么数据从“规模”走向“配方”

早期大家更强调 token 数量，但越到后期越发现：

- 低质 token 会浪费训练预算。
- 重复数据会造成记忆和过拟合。
- 数据污染会让 benchmark 虚高。
- 不合规数据会带来隐私、版权和安全风险。
- 错误 SFT/偏好数据会直接改变模型行为。
- 数据配比不当会让某些能力提升、另一些能力退化。

所以面试里不要只说“我收集了多少条数据”，要说：

- 数据来源是什么。
- 为什么这些来源适合目标任务。
- 清洗和去重怎么做。
- eval/test 怎么防污染。
- 数据配比怎么设计。
- 哪些 bad case 回流到了数据里。

一句话：

> 数据规模是下限，数据配方决定有效学习信号。

## 2. Data Recipe 怎么讲

Data recipe 可以理解成一套可复现的数据配方：

```text
sources + filters + dedup + mixture weights + sampling schedule
+ templates + label policy + split policy + version
```

面试可以拆成 6 个问题：

| 问题 | 你要回答什么 |
| --- | --- |
| 数据从哪来 | 网页、代码、论文、业务日志、人工标注、合成 |
| 为什么可信 | 来源质量、授权、时间范围、领域覆盖 |
| 怎么清洗 | 语言、长度、乱码、模板、PII、毒性、广告 |
| 怎么去重 | exact、near、文档级、段落级、benchmark 级 |
| 怎么配比 | 通用/领域/代码/数学/安全/格式/长尾任务 |
| 怎么证明 | 分桶 eval、ablation、bad case、线上指标 |

口语版：

> 我会把数据配方写成可版本化配置，而不是一次性脚本。每次模型变化都能追溯到来源、过滤规则、采样权重、模板和切分策略。

## 3. 预训练数据质量过滤

预训练数据过滤常见信号：

- 语言识别：是否目标语言。
- 长度：过短、过长、空白、模板页。
- 字符质量：乱码、重复字符、HTML 残留。
- 低质内容：广告、导航、SEO、采集站。
- 内容安全：毒性、色情、违法、仇恨。
- PII：手机号、邮箱、身份证、地址、密钥。
- 困惑度或质量模型分数。
- 来源权重：高质量站点、教育/科研/代码仓库等。
- 去重和 benchmark overlap。

DataComp-LM、FineWeb、Dolma 这类工作给面试的启发是：

- 数据质量可以通过系统实验比较，而不是拍脑袋。
- 过滤策略、去重策略、数据源配比会显著影响模型能力。
- 公开数据集也要看许可证、处理流程和污染风险。

面试表达：

> 预训练数据过滤不是删得越多越好，而是提高有效 token 密度。要在质量、覆盖、多样性和偏差之间权衡。

## 4. 去重：Exact、Near、Semantic

去重层次：

| 类型 | 方法 | 解决什么 |
| --- | --- | --- |
| Exact dedup | hash / checksum | 完全重复 |
| Near dedup | SimHash / MinHash / n-gram overlap | 镜像页、模板页、轻微改写 |
| Semantic dedup | embedding similarity | 语义重复、改写重复 |
| Benchmark decontamination | n-gram / MinHash / exact QA match | 评测污染 |

注意点：

- 预训练可做文档级和段落级去重。
- 代码数据要处理 fork、vendor、生成文件和模板代码。
- SFT/偏好数据要去掉重复 prompt 和高度相似输出。
- eval/test 要和 train 做跨集合去重。

风险：

- 去重太弱：浪费训练、污染 eval。
- 去重太强：丢掉常见表达和领域覆盖。
- 只按 exact 去重：近似重复仍然很多。

面试口语：

> 我会把去重分成训练去重和评测防污染两类。训练去重关注有效 token，评测去重关注不要让模型提前见过答案。

## 5. Benchmark Contamination

污染来源：

- 公开 benchmark 题目或答案混进预训练。
- SFT 数据包含 test prompts。
- prompt few-shot 示例来自测试集。
- RAG eval 的 reference answer 被放入上下文。
- 同一用户、文档、工单跨 train/test 切分。
- 代码 benchmark 的题解和测试出现在 GitHub。

检测方法：

```text
exact match
n-gram overlap
MinHash / SimHash
embedding similarity
prompt-answer pair matching
time-based split
source blacklist
```

面试要说清：

> 污染检测不是只查题目字符串。答案、解析、相似改写、同源文档和 few-shot 示例都可能泄露。

评估报告里要标注：

- 数据 cutoff。
- benchmark 版本。
- decontamination 方法。
- 是否用过公开榜单做调参。
- 私有 eval 是否与训练隔离。

## 6. 合成数据的价值

合成数据适合补：

- 长尾任务。
- 复杂指令。
- 多轮对话。
- 工具调用轨迹。
- 偏好对。
- reasoning traces。
- 安全红队。
- RAG query 和 hard negatives。
- 领域格式和结构化输出样本。

典型流程：

```text
seed tasks / documents / schemas
-> teacher model generate
-> rule filter
-> judge/verifier filter
-> dedup
-> human spot-check
-> mix with real data
-> train/eval ablation
```

一句话：

> 合成数据不是替代真实数据，而是用更便宜的方式扩大覆盖，再用过滤和真实数据校准控制风险。

## 7. Self-Instruct、Evol-Instruct、Magpie

### Self-Instruct

从少量 seed tasks 出发，让模型生成更多 instruction/input/output。

优点：

- 快速扩任务覆盖。
- 适合冷启动 SFT。

风险：

- 风格单一。
- 难度不足。
- teacher 错误被放大。

### Evol-Instruct

让模型把简单指令逐步改写成更复杂、更有约束的任务。

适合：

- 提升复杂指令遵循。
- 增加多约束、多步骤任务。

风险：

- 复杂但不真实。
- 输出冗长。
- 评估和过滤更难。

### Magpie

核心直觉是从强 instruction-tuned model 的行为中自动生成指令数据，减少人工 seed 依赖。

面试表达：

> 这些方法共同点是用 teacher 扩数据；区别在于任务生成方式。真正关键不是名字，而是生成后怎么过滤、去重、分桶、抽检和做 ablation。

## 8. UltraFeedback 和偏好数据

UltraFeedback 类数据关注 response feedback / preference。

偏好数据构造要关注：

- prompt 分布是否真实。
- chosen/rejected 差异是否足够。
- 标注维度是否清楚：正确性、帮助性、安全、格式、引用。
- 是否有长度偏置。
- 是否两个答案都错。
- 是否存在 judge model 偏差。

偏好数据过滤：

```text
remove low-agreement labels
remove both-bad pairs
control length distribution
dedup prompts
bucket by task and risk
spot-check judge bias
```

面试口语：

> DPO/RLHF 的难点不只是 loss，而是偏好标签能不能代表目标行为。如果 chosen 只是更长、更像 teacher，模型会学到错误偏好。

## 9. 可验证合成数据

对数学、代码、检索、工具调用，优先做可验证数据。

例子：

| 任务 | Verifier |
| --- | --- |
| 数学 | 最终答案、符号计算、单元测试 |
| 代码 | test cases、lint、执行结果 |
| RAG | gold evidence、引用覆盖 |
| Tool calling | schema、参数、环境状态 |
| JSON | parser、schema、字段规则 |
| 安全 | policy classifier、人工复核 |

好处：

- 不完全依赖 teacher 自评。
- 能过滤 hallucinated traces。
- 能构造 hard negatives。
- 能支持 RLVR / verifier / rejection sampling。

一句话：

> 只要任务能验证，就优先让 verifier 参与数据过滤。可验证数据比纯自然语言 judge 更可靠。

## 10. 数据配比和 Mixture Tuning

数据配比回答模板：

```text
先定义目标能力 -> 建分桶 eval -> 设计 source/task mixture
-> 小规模 ablation -> 看 trade-off -> 调权重
-> 固化 recipe -> 训练大模型
```

常见配比维度：

- 通用文本。
- 领域文本。
- 代码。
- 数学。
- 多语言。
- 安全拒答。
- 工具调用。
- 长上下文。
- 结构化输出。
- RAG/Agent trace。

风险：

- 领域数据过多，通用能力下降。
- 安全数据过多，误拒上升。
- reasoning traces 过多，回答变啰嗦。
- 合成数据过多，风格变单一。
- 代码数据过多，自然语言风格变化。

面试表达：

> 数据配比必须靠分桶 eval 和 ablation，不是凭经验调。每次改配比都要看目标能力、通用能力、安全、格式、成本是否同时可接受。

## 11. 难例挖掘和 Active Learning

难例来源：

- 线上 bad case。
- eval failed cases。
- judge disagreement。
- 低置信输出。
- 用户差评。
- 安全红队样本。
- 工具调用失败 trace。
- RAG 检索 miss。

流程：

```text
collect failures
-> label failure type
-> cluster/dedup
-> decide fix path: data / prompt / retrieval / model / tool
-> add to eval first
-> add to training if stable
-> regression test
```

关键原则：

> 先进入 eval，再决定是否进训练。不要把所有 bad case 直接塞进训练集。

## 12. 数据切分：Train / Valid / Test

安全切分原则：

- 按用户切，避免同一用户泄露。
- 按文档/工单/实体切，避免同源样本泄露。
- 按时间切，模拟未来数据。
- 按来源切，测试跨域泛化。
- RAG eval 要隔离 gold evidence、query 和 reference answer。
- Agent eval 要隔离任务模板和环境状态。

面试口语：

> 随机切分常常不安全。大模型数据里相似样本太多，必须按业务实体、时间和来源隔离，否则验证集会虚高。

## 13. Data Card 和版本化

Data card 记录：

- 数据来源和许可证。
- 时间范围和 cutoff。
- 采集方式。
- 清洗规则。
- 去重方法。
- PII / safety 处理。
- schema 和模板。
- 标注协议。
- 合成 teacher / judge 版本。
- train/valid/test split。
- 数据量、token 数、语言/任务分布。
- known limitations。

版本化要记录：

```text
dataset_version
source_snapshot
filter_config
dedup_config
template_version
label_policy
split_seed / split_rule
hash / manifest
changelog
```

一句话：

> 没有 data card 和版本化，数据实验就不可复现；模型回归也很难定位。

## 14. 数据质量看板

建议看这些指标：

- token 数、样本数、来源分布。
- 语言分布、领域分布、任务分布。
- 平均长度、P95 长度。
- exact/near duplicate ratio。
- PII hit rate。
- toxicity / unsafe rate。
- schema valid rate。
- judge pass rate。
- human spot-check pass rate。
- contamination match rate。
- train/valid/test overlap。

面试项目里可以说：

> 我不会只报“清洗后剩多少条”，还会报重复率、过滤率、任务分布、污染命中率和抽检通过率。

## 15. 数据问题怎么定位模型回归

模型突然变差，按这条链路查：

```text
eval bucket regression
-> changed data source?
-> changed filter/dedup?
-> changed mixture weight?
-> changed template/loss mask?
-> changed synthetic teacher/judge?
-> train/test leakage?
-> label noise or policy change?
```

典型问题：

- 模板变了，模型输出格式乱。
- 数据配比变了，某个能力桶下降。
- 新合成数据有 teacher 幻觉。
- 过滤规则太激进，领域样本被删。
- 去重误删高频合法模式。
- eval 被污染，指标虚高。

## 16. 高频追问快答

**Q：合成数据能不能进评估集？**  
可以做辅助 stress test，但核心上线 eval 最好用真实业务或人工高质量样本。合成 eval 容易继承 teacher bias。

**Q：怎么证明合成数据有效？**  
做 ablation：只用真实、真实+合成、不同合成比例；看目标任务、通用能力、安全、格式和 bad case 分桶。

**Q：数据配比怎么调？**  
先用小规模实验和分桶 eval 找方向，再固定 recipe 大规模训练。不要直接在全量训练上盲调。

**Q：污染检测是不是 exact match 就够？**  
不够。还要看近似改写、答案泄露、同源文档、few-shot 泄露和按时间/来源切分。

**Q：数据闭环最重要的原则？**  
bad case 先进入 eval，确认稳定复现和归因后，再决定是否进入训练数据。

## 17. 项目 8 分钟讲法

```text
1. 业务目标：
   要提升什么能力？领域问答、格式遵循、代码、数学、Agent 工具还是安全？

2. 数据来源：
   真实业务、公开语料、人工标注、合成数据、线上日志、RAG 文档。

3. 数据治理：
   清洗、去重、PII、质量过滤、合规、污染检测。

4. 数据配方：
   source mixture、任务配比、真实/合成比例、难例比例、replay 数据。

5. 标注和合成：
   teacher/judge/verifier、人工抽检、偏好协议、schema 校验。

6. 切分和评估：
   train/valid/test 按用户/文档/时间隔离，分桶 eval 和私有 benchmark。

7. 结果和复盘：
   哪些桶提升、哪些桶退化、bad case 怎么回流、数据版本怎么回滚。
```

收束句：

> 这个项目我会强调“数据配方 + 评估闭环”，而不是只说我收集和清洗了数据。
