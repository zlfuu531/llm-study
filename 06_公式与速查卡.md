## Attention

```text
Attention(Q,K,V) = softmax(QK^T / sqrt(d_k)) V
```

从输入开始：

```text
input_ids: [B, T]
embedding -> X: [B, T, C]
Q = XW_q, K = XW_k, V = XW_v
Q/K/V: [B, T, D]
score = QK^T: [B, T, T]
A = softmax(score / sqrt(D) + mask)
O = A V: [B, T, D]
```

记忆：

- Q：我要找什么。
- K：我怎么被匹配。
- V：我提供什么内容。
- 缩放：防止 softmax 饱和。
- mask：softmax 前加 `-inf`。

逐元素：

```text
score[i,j] = q_i · k_j = sum_m q_i[m] * k_j[m]
o_i = sum_j softmax(score[i,j]) * v_j
```

## MHA Shape

```text
x:     [B, T, C]
qkv:   [B, T, 3C]
view:  [B, T, 3, H, D]
q/k/v: [B, H, T, D]
score: [B, H, T, T]
out:   [B, T, C]
```

`D = C / H`

参数量：

```text
W_q + W_k + W_v + W_o ≈ 4 * C * C
```

复杂度：

```text
QKV/O projection: O(B * T * C^2)
Attention:        O(B * T^2 * C)
```

记忆：

## DPO

```text
pi_logratio = logp_policy_chosen - logp_policy_rejected
ref_logratio = logp_ref_chosen - logp_ref_rejected
loss = -logsigmoid(beta * (pi_logratio - ref_logratio))
```

记忆：让 policy 相对 reference 更偏好 chosen。

`beta`：

- 大：偏好优化更猛，可能偏离 reference。
- 小：更新保守，偏好学习不明显。

## GRPO

同一个 prompt 采样 `G` 个回答：

```text
rewards: [B, G]
A_i = (r_i - mean(r_1...r_G)) / (std(r_1...r_G) + eps)
```

记忆：

- 同题多答案。
- 组内比较。
- 高于平均的鼓励，低于平均的压低。
- 减少 value model 依赖。

## Sequence Logprob

```text
log_probs = log_softmax(logits, dim=-1)
token_logp = gather(log_probs, labels)
sequence_logp = sum(token_logp * answer_mask)
```

易错：

- next-token 任务要 shift。
- prompt/pad 不算 loss。
- DPO chosen/rejected 长度差异大时要说明 sum 还是 mean。

## Cross Entropy / PPL

```text
CE = - sum_i y_i log p_i
one-hot: CE = -log p_true
softmax + CE gradient: dL/dz = p - y
PPL = exp(mean token CE)
```

记忆：

- CrossEntropyLoss 通常输入 logits，不要先手动 softmax。
- PPL 低不等于指令遵循、事实性、安全性好。
- LLM SFT 常只算 assistant answer loss，prompt/pad 不算 loss。
- packing 要检查 attention mask、loss mask、position id 和样本边界。

## ML / DL 数学基础速查

Loss 选择：

```text
单标签多分类: CrossEntropyLoss(logits, class_id)
二分类/多标签: BCEWithLogitsLoss(logits, labels)
回归: MSE / Huber
Embedding/Reranker: InfoNCE / pairwise / listwise
```

Logits：

```text
logits -> softmax/sigmoid -> probability -> threshold/decision
训练 loss 通常吃 logits，展示概率时再归一化
```

最大似然：

```text
max sum log p_theta(y|x)
等价于 min -sum log p_theta(y|x)
one-hot 分类下就是 CE
```

类别不均衡：

```text
accuracy 可能骗人
看 precision / recall / F1 / PR-AUC / Recall@K
可用 class weight / sampling / focal loss / threshold tuning
```

指标：

```text
Precision = TP / (TP + FP)
Recall = TP / (TP + FN)
F1 = 2PR / (P + R)
TPR = TP / (TP + FN)
FPR = FP / (FP + TN)
AUC ~= 随机正样本得分高于随机负样本的概率
```

阈值：

```text
threshold 上升: precision 通常上升, recall 通常下降
threshold 下降: recall 通常上升, precision 通常下降
```

校准：

```text
模型说 0.8 概率的样本，真实正确率是否约等于 80%
p = softmax(logits / T)
T > 1: 分布变平, 置信度降低
T < 1: 分布变尖, 置信度升高
```

InfoNCE：

```text
L_i = -log exp(sim(q_i, d_i+)/tau)
          / sum_j exp(sim(q_i, d_j)/tau)
```

面试压缩：

```text
loss 是训练信号
metric 是评价目标
threshold 是上线决策
calibration 是置信度可信度
bad case 是改进方向
```

loss 降了但线上没涨：

```text
数据分布 -> 标签质量 -> loss/mask -> 指标选择
-> 阈值/校准 -> 线上链路一致性 -> bad case 分桶
```

## AdamW

```text
m_t = beta1*m_{t-1} + (1-beta1)*g_t
v_t = beta2*v_{t-1} + (1-beta2)*g_t^2
theta = theta - lr * m_hat / (sqrt(v_hat) + eps)
theta = theta - lr * weight_decay * theta
```

记忆：

- Adam：一阶动量 + 二阶动量。
- AdamW：weight decay 和自适应梯度解耦。
- bias / LayerNorm / RMSNorm scale 常不做 weight decay。
- warmup 解决初期不稳定，cosine decay 解决后期收敛。
- gradient clipping 是安全带，根因还要查数据、学习率、loss mask 和混合精度。

## PyTorch 训练工程速查

标准训练循环：

```text
model.train()
for batch in loader:
    batch -> device
    optimizer.zero_grad(set_to_none=True)
    loss = model(batch).loss
    loss.backward()
    clip_grad_norm_(model.parameters(), max_norm)
    optimizer.step()
    scheduler.step()
```

验证循环：

```text
model.eval()
with torch.no_grad():
    forward + metric
```

`train/eval/no_grad`：

```text
model.train(): Dropout/BatchNorm 等进入训练行为
model.eval(): Dropout/BatchNorm 等进入评估行为
torch.no_grad(): 不记录 autograd graph
torch.inference_mode(): 更强推理模式，纯推理更省
```

Autograd：

```text
requires_grad=True -> 记录操作
grad_fn -> tensor 的生成操作
leaf tensor -> 参数通常是 leaf
loss.backward() -> 梯度累积到 param.grad
detach() -> 切断历史计算图
retain_graph=True -> 多次 backward 才需要，容易涨显存
```

梯度累积：

```text
effective_batch = micro_batch * grad_accum_steps * world_size
loss = loss / grad_accum_steps
loss.backward()
if step % grad_accum_steps == 0:
    optimizer.step()
    optimizer.zero_grad(set_to_none=True)
```

AMP + GradScaler：

```text
with torch.amp.autocast("cuda", dtype=torch.float16):
    loss = model(batch).loss
scaler.scale(loss).backward()
scaler.unscale_(optimizer)
clip_grad_norm_(...)
scaler.step(optimizer)
scaler.update()
```

FP16 / BF16：

```text
FP16: 动态范围小，常配 GradScaler
BF16: 动态范围接近 FP32，训练更稳，常不需要 scaler
```

DataLoader：

```text
Dataset: 样本怎么取
Sampler: 样本顺序
DistributedSampler: DDP 切分数据
collate_fn: 样本列表 -> batch，padding/mask/label mask
num_workers: 多进程加载
pin_memory + non_blocking: 加速 H2D copy
```

Checkpoint 清单：

```text
model.state_dict()
optimizer.state_dict()
scheduler.state_dict()
scaler.state_dict()
global_step / epoch
rng_state / cuda_rng_state
config / tokenizer / chat template / data version / code version
```

DDP：

```text
每卡完整模型
每卡不同 batch
backward 时 gradient all-reduce
每卡各自 optimizer.step()
DistributedSampler.set_epoch(epoch)
rank0 logging / checkpoint
```

DDP + 梯度累积：

```text
非更新 micro-step: model.no_sync()
更新 micro-step: 正常 backward 触发 all-reduce
```

## Tokenizer / BPE / SentencePiece / Token 预算速查

基本链路：

```text
text
-> normalizer
-> pre-tokenizer
-> subword model(BPE / WordPiece / Unigram)
-> post-processor(special tokens)
-> token ids
-> embedding lookup
```

BPE 训练：

```text
初始 symbol(char/byte)
-> 统计相邻 pair 频率
-> 合并最高频 pair
-> 更新语料表示
-> 重复直到 vocab size
```

BPE 编码：

```text
输入文本 -> 初始 symbol 序列
-> 按 merge rank 合并可合并 pair
-> 输出 subword token ids
```

BPE / WordPiece / Unigram：

| 算法 | 面试一句话 | 常见场景 |
| --- | --- | --- |
| BPE | 高频相邻 pair 逐步 merge，词表可控 | GPT/Llama/Qwen 等 decoder LLM |
| WordPiece | 合并更关注似然提升，词内片段常带 `##` | BERT 系列 |
| Unigram | 从大候选词表删 token，选概率最大切分 | SentencePiece 常见 |

SentencePiece：

```text
直接从原始文本学习 subword
空格用 ▁ 表示
适合中文、日文等无天然空格语言
```

Byte-level BPE / byte fallback：

```text
覆盖任意 Unicode 字符
减少 <unk>
代价是罕见字符、中文、emoji、乱码可能被切得更碎
```

Special tokens：

```text
BOS: 序列开始
EOS: 序列结束 / 停止生成
PAD: batch padding，loss/attention 要 mask
UNK: 未知 token
SEP: 片段分隔
MASK: MLM 掩码
role/tool tokens: chat template 和工具调用协议
```

Token budget：

```text
total_input_tokens =
```

## 解码策略 / Sampling / Beam Search 速查

自回归生成：

```text
prompt tokens
-> logits over vocab
-> processors / warpers
-> choose next token
-> append
-> stop on EOS / stop / max_new_tokens
```

Softmax：

```text
p_i = exp(z_i) / sum_j exp(z_j)
```

Temperature：

```text
p_i = softmax(z_i / T)
T < 1: 更尖，更确定
T > 1: 更平，更多样
T -> 0: 接近 greedy
```

Greedy：

```text
x_{t+1} = argmax_i p_i
```

Sampling：

```text
x_{t+1} ~ Categorical(p)
```

Top-k：

```text
S = top k tokens by probability
p'_i = p_i / sum_{j in S} p_j, i in S
p'_i = 0, otherwise
sample from p'
```

Top-p / nucleus：

```text
sort p descending
S = smallest prefix with cumulative probability >= top_p
sample from normalized p over S
```

Top-k vs Top-p：

```text
Top-k: 固定候选数量
Top-p: 动态候选集合，控制累计概率质量
```

Beam Search：

```text
score(seq) = sum_t log p(x_t | x_<t)
每步扩展 beams，保留 top num_beams 条路径
```

Length penalty：

```text
score = sum_logprob / length^alpha
```

Repetition penalty：

```text
if generated token has logit > 0: logit = logit / penalty
if generated token has logit < 0: logit = logit * penalty
```

Logits processing 链路：

```text
raw logits
-> bad words / min length / no-repeat ngram / forced tokens
-> repetition / presence / frequency penalty
-> temperature / top-k / top-p
-> argmax or multinomial sampling
```

Stopping：

```text
EOS token
stop strings
max_new_tokens
timeout
tool call boundary
```

参数选择：

| 任务 | 常见策略 |
| --- | --- |
| JSON / 工具调用 | 低温 + constrained decoding |
| RAG 问答 | 低温 + top_p + 输出长度限制 |
| 创意写作 | 较高 temperature + top_p |
| 代码生成 | 低温 pass@1，多采样 pass@k |
| 翻译 / 摘要 | beam 或低温 sampling |
| 复杂推理 | 多采样 + verifier / Self-Consistency |

常见排查：

```text
重复 -> prompt/上下文/stop/惩罚参数/模型训练数据
幻觉 -> temperature/top_p 过高/RAG 证据不足/输出太长
格式错 -> prompt 不够/schema 约束缺失/stop token 错
成本高 -> output tokens 多/多采样/beam/schema 解码慢
不可复现 -> seed/model/tokenizer/config/batch/kernel/服务版本
```

更完整的 generation loop、temperature、top-k/top-p、beam search、LogitsProcessor、constrained decoding、stop strings、pass@k 和项目讲法见：[36_解码策略_Sampling_BeamSearch与LogitsProcessor面试.md](36_解码策略_Sampling_BeamSearch与LogitsProcessor面试.md)。

## Embedding / Reranker / 向量检索速查

相似度：

```text
dot(q, d) = q^T d
cos(q, d) = q^T d / (||q|| ||d||)
L2(q, d) = ||q - d||_2

若 ||q|| = ||d|| = 1:
cos(q, d) = q^T d
||q - d||_2^2 = 2 - 2cos(q, d)
```

BM25：

```text
BM25(q, D) =
Σ IDF(q_i) * f(q_i, D) * (k1 + 1)
  / (f(q_i, D) + k1 * (1 - b + b * |D| / avgdl))
```

RRF 融合：

```text
RRF(d) = Σ 1 / (k + rank_i(d))
```

索引取舍：

```text
Flat: 精确但慢
HNSW: 图搜索，低延迟高召回，内存高
IVF: 聚类分桶，nprobe 越大召回越高延迟越高
PQ: 向量压缩，省内存但有量化误差
```

Rerank：

```text
Bi-encoder: q = E(q), d = E(d), score = q·d，快，适合召回
Cross-encoder: score = f([q; d])，准，慢，适合 top-k 重排
```

召回低排查：

```text
query -> rewrite -> chunk -> metadata -> embedding/BM25
-> ANN 参数 -> filter -> fusion -> rerank -> context
```

- 高风险工具：最小权限 + 人工确认 + 审计日志。

## 推理指标

- TTFT：首 token 延迟。
- TPOT：每 token 耗时。
- QPS：每秒请求数。
- tokens/s：每秒 token 吞吐。
- P95/P99：尾延迟。

排查：

```text
TTFT 高：网关排队 -> tokenizer/prompt -> RAG 上下文过长
       -> prefix cache 未命中 -> prefill 调度 -> GPU 队列

TPOT 高：decode batch -> KV Cache 长度/带宽 -> kernel
       -> 输出过长 -> 后处理/安全过滤
```

## vLLM / 推理优化

```text
PagedAttention: block 管理 KV Cache，减少碎片
Continuous Batching: decode 阶段动态进出 batch
Prefix Cache: 复用相同前缀 KV
Chunked Prefill: 长 prompt 分块，改善短请求 TTFT
Speculative Decoding: 小模型草拟，大模型验证
```

别混：

- FlashAttention：优化 attention 计算和 IO。
- PagedAttention：优化推理 KV Cache 管理。

更完整的推理引擎专题见：[24_推理引擎_vLLM_SGLang_TensorRTLLM面试.md](24_推理引擎_vLLM_SGLang_TensorRTLLM面试.md)。

### 推理引擎核心公式和链路

KV Cache 显存：

```text
KV bytes ≈ 2 * layers * active_requests * seq_len
           * kv_heads * head_dim * bytes_per_element
```

记忆：

- `2` 是 K 和 V。
- MHA 的 `kv_heads` 通常等于 attention heads。
- MQA/GQA/MLA 通过减少 KV 表示降低显存和带宽。
- 长上下文、高并发、长输出都会线性放大 KV Cache。

Serving 链路：

```text
API/Gateway -> Queue/Admission -> Scheduler -> Tokenizer
-> Prefill -> KV Cache Manager -> Decode Loop
-> Sampler -> Streamer -> Metrics/Trace
```

Prefill vs Decode：

| 阶段 | 主要做什么 | 瓶颈 | 指标 |
| --- | --- | --- | --- |
| Prefill | 处理完整 prompt，生成 KV | prompt 长、RAG 上下文、算力、排队 | TTFT |
| Decode | 逐 token 生成，读写 KV | KV Cache、显存带宽、batch 调度 | TPOT/ITL |

PagedAttention vs FlashAttention：

```text
FlashAttention  = attention kernel 的计算/IO 优化
PagedAttention  = serving KV Cache 的 block/page 管理
```

Prefix Cache / RadixAttention：

- Prefix Cache：前缀 token 完全一致时复用 KV。
- RadixAttention：把共享前缀组织成 radix tree，更适合多轮、Agent、结构化 workflow。
- 命中低常见原因：时间戳、随机 ID、用户信息放在前缀、RAG 排序变化、cache 太小、多租户隔离。

Speculative Decoding：

```text
draft model 草拟 gamma 个 token
target model 并行验证
接受率高 + draft 便宜 + 验证并行好 -> 加速
接受率低 + draft 太重 + 显存挤占 -> 可能变慢
```

更完整专题见：[38_推测解码_DraftModel_EAGLE_Medusa_MTP面试.md](38_推测解码_DraftModel_EAGLE_Medusa_MTP面试.md)。

推测解码速查：

```text
prefix
-> proposer 草拟 y_1...y_gamma
-> target 一次 forward 验证
-> 从左到右接受连续前缀
-> 拒绝时修正采样
-> 全部接受时可额外输出一个 target token
```

sampling 接受概率：

```text
p: target distribution
q: draft/proposer distribution
y: draft sampled token

a(y) = min(1, p(y) / q(y))
```

拒绝后的修正分布：

```text
r(x) ∝ max(0, p(x) - q(x))
```

加速粗估：

```text
C_T: target 生成 1 token 成本
C_D: draft 生成 1 token 成本
C_V: target 验证 gamma token 成本
gamma: 草拟 token 数
A: 平均接受 token 数

speedup ≈ ((A + 1) * C_T) / (gamma * C_D + C_V + overhead)
```

方法区别：

```text
draft model: 独立小模型草拟，要求便宜且像 target
n-gram/suffix: 从上下文重复片段草拟，便宜但适用面窄
Medusa: 多个 decoding heads + tree attention
EAGLE: feature-level speculation
MTP: 多 token 预测模块/目标，可作为原生 proposer
```

关键指标：

```text
accept rate = accepted draft tokens / proposed draft tokens
mean accepted length = 每轮平均接受 token 数
tokens per target step = target 每次验证平均推进 token 数
```

什么时候可能变慢：

```text
accept rate 低
draft 太重
gamma/tree/topk 太大
高 QPS / 大 batch 下 target 已饱和
瓶颈在 prefill / 排队 / tokenizer / 网络
guided decoding / JSON schema 让 proposer 常生成非法 token
双模型显存、KV Cache、调度和 CUDA graph overhead 变大
```

Prefill/Decode disaggregation：

```text
Prefill worker: prompt -> KV Cache
KV transfer
Decode worker: KV Cache -> token stream
```

适合规模较大、长上下文和混合流量明显的系统；难点是 KV 传输、cache locality、调度和故障恢复。

更完整专题见：[39_分布式推理_PD分离_KVCache传输与MoEServing面试.md](39_分布式推理_PD分离_KVCache传输与MoEServing面试.md)。

PD 分离速查：

```text
Aggregated:
  same GPU pool = prefill + decode
  simple, no KV transfer
  but prefill/decode interfere

Disaggregated:
  prefill pool -> KV transfer -> decode pool
  independent scaling
  but KV handoff is hard
```

Prefill vs Decode：

```text
Prefill:
  full prompt
  compute / matmul heavy
  affects TTFT

Decode:
  one token per step
  KV Cache / HBM / scheduling heavy
  affects TPOT / ITL
```

KV Cache：

```text
KV bytes ≈ B * S * L * H_kv * D * 2 * bytes
```

KV transfer 难点：

```text
KV 很大
prefill/decode TP size 可能不同
KV layout / page / block 映射要一致
decode 侧要预分配 KV slots
取消 / 超时 / 失败要清理两侧资源
跨节点要 RDMA / UCX / NIXL / Mooncake 等高效通道
```

并行策略：

```text
TP: 切层内矩阵，通信是 all-reduce/all-gather
PP: 切层，注意 pipeline bubble
DP/Replica: 多副本扩 QPS
EP: 切 MoE experts，通信是 all-to-all
```

PD 路由看：

```text
prompt length
max_new_tokens
prefill queue
decode queue
KV capacity
KV transfer bandwidth
prefix cache hit
tenant / SLO / priority
```

MoE serving：

```text
router -> top-k experts -> dispatch -> expert FFN -> combine

难点:
- all-to-all
- hot experts
- load imbalance
- dynamic shape
- tail latency
```

## GPU Systems / CUDA / Triton / FlashAttention

完整专题见：[25_GPU_CUDA_Triton与FlashAttention面试.md](25_GPU_CUDA_Triton与FlashAttention面试.md)。

瓶颈分类：

```text
compute-bound: tensor core/SM 忙，主要缺算力
memory-bound: HBM bandwidth 高，SM 等数据
launch/CPU-bound: 小 kernel 多，GPU timeline 有空洞
communication-bound: NCCL/all-reduce/all-gather/all-to-all 占比高
```

Roofline 直觉：

```text
Arithmetic intensity = FLOPs / bytes moved
performance <= min(peak compute, memory bandwidth * arithmetic_intensity)
```

GPU 内存层次：

```text
register -> shared memory/SRAM -> L2 -> HBM/global memory -> CPU memory
```

CUDA 执行模型：

```text
grid -> block/CTA -> warp -> thread
```

Kernel 优化关键词：

- coalesced access：相邻线程访问连续地址，提升 HBM 带宽利用。
- tiling：把数据块搬到 shared memory 复用。
- occupancy：活跃 warp 程度，能隐藏延迟，但不是越高越好。
- warp divergence：同一 warp 内分支不同会串行化。
- kernel fusion：减少 launch 和中间张量写回 HBM。

FlashAttention：

```text
标准 attention: QK^T -> softmax -> PV
问题: 显式写 N x N attention matrix，HBM IO 大
FlashAttention: tiling + online softmax + 不写完整 N x N
性质: exact attention，不是近似；优化 attention 计算 IO
```

Triton：

```text
program_id: 当前 program/block 编号
BLOCK_SIZE: tile 大小
mask: 处理边界，避免越界 load/store
tl.load/tl.store: 显式访存
```

Profiler 排查：

```text
PyTorch Profiler: op 级 CPU/GPU 时间
Nsight Systems: 全局 timeline、CPU-GPU 空洞、NCCL、同步
Nsight Compute: 单 kernel 的 occupancy、bandwidth、stall、bank conflict
```

NCCL collective：

```text
all-reduce: DDP 梯度 / TP 结果合并
all-gather: FSDP 参数聚合 / 分片收集
reduce-scatter: ZeRO/FSDP 切分状态
all-to-all: MoE expert parallel token 分发
```

## 搜索推荐广告 / Ranking / LTR 速查

工业链路：

```text
request/query/user/context
-> 多路召回
-> 粗排
-> 精排
-> 重排/策略
-> 展示
-> 点击/转化/反馈回流
```

目标函数：

```text
pCTR = P(click | user, item, context)
pCVR = P(conversion | click, user, item, context)
pCTCVR = P(click and conversion | user, item, context)
eCPM = bid * pCTR * pCVR * calibration
score = w1 * relevance + w2 * pCTR + w3 * pCVR + w4 * value - penalties
```

CTR/CVR 常用 BCE：

```text
L = - y log p - (1 - y) log(1 - p)
```

FM 二阶交叉：

```text
y = w0 + sum_i w_i x_i + sum_{i<j} <v_i, v_j> x_i x_j
```

双塔召回：

```text
q = E_q(query/user)
d = E_d(item)
score = q · d
```

Pairwise LTR：

```text
P(i > j) = sigmoid(s_i - s_j)
L_pair = -log sigmoid(s_i - s_j)
```

NDCG：

```text
DCG@K = sum_{i=1..K} (2^{rel_i} - 1) / log2(i + 1)
NDCG@K = DCG@K / IDCG@K
```

MRR / MAP：

```text
MRR = mean(1 / rank_first_relevant)
AP = mean(Precision@k for each relevant item position k)
MAP = mean(AP over queries)
```

AUC / GAUC：

```text
AUC ~= P(score_pos > score_neg)
GAUC = sum_u impressions_u * AUC_u / sum_u impressions_u
```

MMR 重排：

```text
score(item) = lambda * relevance(item)
              - (1 - lambda) * max_similarity(item, selected)
```

IPS 反事实直觉：

```text
unbiased_reward ~= observed_reward / propensity
```

面试答题链路：

```text
目标是什么 -> 候选怎么来 -> 排序怎么做 -> 指标怎么选
-> 偏差怎么处理 -> 线上怎么 A/B -> LLM 放哪一层 -> 成本和合规怎么控
```

## VLM / 多模态速查

典型架构：

```text
image/video -> vision encoder -> projector/Q-Former/resampler
-> visual tokens -> LLM -> answer/bbox/action
```

ViT patch 数：

```text
num_patches = (H / P) * (W / P)
```

高分辨率 / AnyRes 成本：

```text
num_patches = ceil(H / P) * ceil(W / P)
visual_tokens ≈ global_tokens + sum(tile_tokens)
video_visual_tokens ≈ num_frames * tokens_per_frame
```

直觉：

- 分辨率翻倍，patch/token 数大约变成 4 倍。
- AnyRes/tiling 保留局部细节，但会增加 token、prefill 和 KV Cache 成本。
- dynamic resolution 是给 OCR、截图、票据、GUI 多预算，给普通图像少预算。

CLIP 对比学习：

```text
s_ij = cosine(v_i, t_j) / tau
L = (CE(image_to_text) + CE(text_to_image)) / 2
```

连接模块：

- projector：视觉特征映射到 LLM hidden space。
- Q-Former：learnable query 从视觉特征抽取信息。
- resampler：把大量视觉 token 压成固定数量 token。

OCR 排查：

```text
image quality/resolution -> crop/tiling -> vision encoder
-> token compression -> prompt -> generation -> OCR tool/rule check
```

Grounding：

```text
bbox = [x1, y1, x2, y2]
x_norm = x / image_width
y_norm = y / image_height
IoU = area(pred ∩ gt) / area(pred ∪ gt)
```

## 模型量化 / 低比特推理速查

主线：

```text
低精度表示权重 / 激活 / KV Cache
-> 降低显存和带宽
-> 质量取决于误差、outlier、校准、粒度
-> 速度取决于硬件、kernel、dequant 和端到端瓶颈
```

权重显存粗估：

```text
P = 参数量
FP32: 4P bytes
FP16/BF16: 2P bytes
INT8: 1P bytes
INT4: 0.5P bytes
```

实际还要加：

```text
scale / zero point / group metadata
runtime buffer
KV Cache
未量化层
adapter
```

对称量化：

```text
q = clip(round(x / s), q_min, q_max)
x_hat = s * q

s ≈ max(|x|) / (2^(b-1)-1)
```

非对称量化：

```text
s = (x_max - x_min) / (q_max - q_min)
z = round(q_min - x_min / s)
q = clip(round(x / s + z), q_min, q_max)
x_hat = s * (q - z)
```

PTQ vs QAT：

```text
PTQ: 训练后量化，成本低，上线快，依赖校准数据
QAT: 训练时模拟量化误差，质量可能更稳，成本高
```

量化对象：

```text
weight-only: 只量化权重，GPTQ/AWQ/GGUF 常见
W8A8: 权重和激活都 INT8，SmoothQuant 常见
KV quant: 量化推理缓存，长上下文/高并发收益明显
FP8: 浮点低精度，依赖硬件、scaling 和混合精度框架
```

粒度：

```text
per-tensor: 一个张量一个 scale，简单但容易被 outlier 伤害
per-channel: 每个通道一个 scale，更准但 scale 更多
group-wise: 每组一个 scale，LLM 4-bit 常用折中
```

outlier 直觉：

```text
大多数值在 [-1, 1]，但有一个 20
-> scale 被 20 撑大
-> 普通值只能落在很少整数档位
-> 量化误差变大
```

GPTQ：

```text
目标: 3/4-bit weight-only
核心: 用校准激活和近似二阶信息补偿权重量化误差
一句话: 不是简单 round，而是尽量让 XW_hat 接近 XW
```

AWQ：

```text
目标: 4-bit weight-only
核心: activation-aware，保护少量敏感通道
一句话: 重要权重不只看权重大小，还看它乘到的激活
```

SmoothQuant：

```text
目标: W8A8
核心: 平滑激活 outlier，把难度迁移到权重

Y = XW = (X / s) * (sW)
```

QLoRA：

```text
base model: 4-bit quantized + frozen
LoRA adapter: trainable
NF4: 适合近似正态分布权重
double quant: 再量化量化常数
paged optimizer: 缓解 optimizer state 显存峰值
```

FP8：

```text
E4M3: 精度更高，动态范围较小
E5M2: 动态范围更大，精度较低
需要: hardware Tensor Core + scaling recipe + framework/kernel support
```

KV Cache 显存：

```text
KV cache ≈ B * S * L * H_kv * D * 2 * bytes

B: batch / 并发序列数
S: cache token 数
L: layers
H_kv: KV heads，GQA/MQA 会降低
D: head_dim
2: K 和 V
```

为什么 INT4 不一定快：

```text
省权重显存 != 一定省端到端延迟

原因:
- 解包和 dequant 开销
- scale/zero point metadata 访存
- 低比特 GEMM kernel 是否高效
- prefill/decode 瓶颈不同
- batch/seq shape 不同
- GPU/CPU/NPU 低比特支持不同
- 非量化部分仍占开销
- 质量下降可能需要多采样 / rerank / fallback
```

量化评估：

```text
quality:
  PPL / benchmark / 私有 eval / 数学 / 代码 / 长上下文 / JSON / RAG

system:
  显存 / TTFT / TPOT / tokens/s / concurrency / P95/P99 / OOM / cost

baseline:
  FP16 -> INT8/W8A8 -> GPTQ/AWQ INT4 -> KV quant -> 不同 group size
```

排查链路：

```text
固定 tokenizer / chat template / decoding config
-> 对比 FP16 baseline
-> 检查 calibration data
-> 调 bitwidth / group size / sensitive layers
-> 逐个开关 weight quant / activation quant / KV quant
-> 分桶看数学、代码、长文、结构化输出
-> 必要时 LoRA repair / 蒸馏 / QAT
```

## 端侧小模型与压缩速查

端侧目标：

```text
有限内存 + 有限算力 + 功耗/发热约束 + 隐私/离线
-> 够用质量 + 可接受延迟
```

蒸馏：

```text
p_i = exp(z_i / T) / sum_j exp(z_j / T)
L_distill = T^2 * KL(p_teacher^T || p_student^T)
L = alpha * CE + (1 - alpha) * L_distill
```

量化：

```text
对称: q = round(x / scale), x_hat = q * scale
非对称: q = round(x / scale + zero_point)
        x_hat = (q - zero_point) * scale
```

量化粒度：

- per-tensor：简单，误差大。
- per-channel：更准，scale 更多。
- group-wise：LLM 低比特常用折中。

方法区分：

- GPTQ：近似二阶补偿量化误差。
- AWQ：保护激活敏感通道。
- SmoothQuant：平滑激活 outlier，利于 W8A8。
- GGUF：llama.cpp 生态本地推理格式。

剪枝：

- 非结构化剪枝：省参数，但不一定加速。
- 结构化剪枝：删 head/channel/neuron/layer，更易加速。

端侧指标：

- 模型文件大小。
- 峰值内存。
- 冷启动。
- TTFT / TPOT / tokens/s。
- 功耗和发热。
- 长时间运行降频。
- 低端设备兼容性。
- 业务质量和安全 eval。

## 大模型训练系统速查

训练显存组成：

```text
参数 + 梯度 + optimizer state + activation
+ temporary buffer + communication buffer + fragmentation
```

混合精度 Adam 全参训练粗估：

```text
params: 2P
grads: 2P
master weights: 4P
Adam m: 4P
Adam v: 4P
total ~= 16P bytes
```

例子：

- 7B：仅参数/梯度/Adam 状态约 `16 * 7B = 112GB`。
- 14B：约 `16 * 14B = 224GB`。
- 还要额外加 activation、buffer、通信和碎片。

并行切分：

- DDP：切 batch，每卡完整模型，梯度 all-reduce。
- ZeRO-1：切 optimizer state。
- ZeRO-2：再切 gradients。
- ZeRO-3 / FSDP：再切 parameters。
- TP：切层内矩阵。
- PP：切层。
- EP：切 MoE experts。

3D 并行：

```text
total_gpus = DP * TP * PP
```

选择：

- 单层太大：TP。
- 层数太多：PP。
- batch/吞吐要扩：DP。
- optimizer/gradient/parameter 太占显存：ZeRO/FSDP。
- MoE experts 太多：EP。

Activation checkpointing：

```text
少存 activation，反向时重算
省 activation 显存，增加计算时间
```

Global batch：

```text
global_batch = micro_batch_per_gpu * data_parallel_size * grad_accum_steps
```

混合精度：

- FP16：快、省显存，可能下溢，常需要 loss scaling。
- BF16：动态范围大，LLM 训练更稳。
- FP8：更省带宽和存储，依赖硬件、框架和 scaling。

通信排查：

- DDP：all-reduce。
- ZeRO/FSDP：all-gather、reduce-scatter。
- TP：层内 all-reduce / all-gather。
- PP：pipeline bubble、stage imbalance。
- MoE/EP：all-to-all、负载不均。

OOM 排查：

```text
参数状态大 -> ZeRO/FSDP/LoRA/offload
activation 大 -> checkpointing/减 seq 或 batch/FlashAttention
通信或临时 buffer 大 -> 调并行策略和 kernel
碎片 -> 查 allocator、batch shape、重启复现
```

## DeepSeek / MoE / MLA / Reasoning 速查

V3 / R1：

- DeepSeek-V3：高效 MoE 通用模型，重点是 MoE、MLA、MTP、训练和推理效率。
- DeepSeek-R1：reasoning model，重点是 R1-Zero、cold-start、多阶段 RL、GRPO、蒸馏。

MoE：

```text
y = shared_expert(x) + sum_{i in TopK(router(x))} gate_i(x) * expert_i(x)
```

- 总参数多，激活参数少。
- 每个 token 只路由到 top-k experts。
- 难点：负载均衡、expert collapse、capacity、all-to-all、尾延迟。

MLA：

```text
MHA: 每个 head 缓存 K/V
MQA: 所有 query heads 共享 K/V
GQA: 分组共享 K/V
MLA: latent 压缩 K/V 相关信息
```

MTP：

```text
next-token: predict x_{t+1}
multi-token: 辅助预测 x_{t+1}, x_{t+2}, ...
```

GRPO：

```text
同一 prompt 采样一组回答
A_i = (r_i - mean(r_group)) / std(r_group)
```

- PPO：常需要 value/critic。
- DPO：离线 chosen/rejected。
- GRPO：组内相对优势，减少 value model 依赖。

R1-Zero / R1：

- R1-Zero：用 RL 激发推理，可能有可读性和语言稳定性问题。
- R1：加 cold-start 和多阶段训练，提升可读性、稳定性和通用能力。

Reasoning Model：

- 收益：数学、代码、多步规划、自我检查。
- 训练：reasoning SFT、可验证 reward、RL、rejection sampling、distillation。
- 风险：reward hacking、长思考成本、过程不忠实、简单题过度推理。

DeepSeek 为什么便宜：

```text
MoE 稀疏激活
+ MLA 降 KV Cache
+ GRPO 降低 reasoning RL 复杂度
+ 蒸馏迁移能力
+ 推理服务缓存/批处理/路由
```

## Reasoning Post-training / RLVR / Verifier 速查

RLVR：

```text
sample response
-> verifiable reward(checker/test/verifier)
-> RL update
```

- 适合：数学、代码、工具调用、规则任务。
- 风险：checker 漏洞、reward hacking、答案对但过程错。

Reward 来源：

```text
Verifiable reward: 程序/测试/规则
Reward Model: 学出来的打分模型
LLM-as-Judge: 大模型按 rubric 评审
```

ORM / PRM：

```text
ORM: outcome reward, 看最终答案
PRM: process reward, 看中间步骤
```

- ORM：便宜，适合 Best-of-N。
- PRM：细粒度，适合 search / 早停 / 过程监督，但标注贵。

Verifier：

```text
Train-time: samples -> verifier -> RL / rejection sampling / distillation
Test-time: sample N -> verifier score -> choose best / continue search
```

Best-of-N：

```text
sample N answers -> verifier scores -> choose max
```

Self-Consistency：

```text
sample N reasoning paths -> extract final answers -> majority / consistency vote
```

Test-Time Scaling：

```text
更长思考 + 多采样 + verifier + search + tool feedback
```

收益：

- 难题覆盖更多路径。
- verifier 筛掉坏解。
- 数学/代码/规划收益更明显。

代价：

- token 成本。
- P95/P99 延迟。
- verifier 调用成本。
- 简单题过度思考。

Compute routing：

```text
simple -> normal model / short answer
medium -> short reasoning / small N
hard -> long reasoning / Best-of-N / verifier / tool
high-risk -> human-in-the-loop
```

GRPO in RLVR：

```text
同一 prompt 采样 G 个回答
reward_i = checker(response_i)
A_i = (reward_i - mean(group)) / std(group)
```

评估：

```text
pass@1: 一次答对
pass@k: k 个候选里至少一个正确
cons@k: k 次采样投票后答对
cost per solved problem: 每解出一题的成本
```

Reward hacking 治理：

```text
hidden tests + 多 verifier + 人工抽检
+ adversarial eval + held-out tasks
+ reward/业务指标相关性监控
```

## Diffusion / DiT / 文生图视频生成速查

DDPM 前向加噪：

```text
q(x_t | x_{t-1}) = N(sqrt(1-beta_t)x_{t-1}, beta_t I)
alpha_t = 1 - beta_t
alpha_bar_t = product_{s=1}^t alpha_s
x_t = sqrt(alpha_bar_t)x_0 + sqrt(1-alpha_bar_t)epsilon
epsilon ~ N(0, I)
```

训练目标：

```text
epsilon_pred = epsilon_theta(x_t, t, condition)
L = E ||epsilon - epsilon_pred||^2
```

预测目标：

```text
epsilon prediction: 预测噪声
x0 prediction: 预测干净样本
v prediction: 混合参数化，常更稳定
```

CFG：

```text
epsilon_hat = epsilon_uncond + s * (epsilon_cond - epsilon_uncond)
```

- `s` 小：prompt adherence 弱。
- `s` 大：更听 prompt，但可能过饱和、质量下降、多样性变差。

Latent Diffusion：

```text
image -> VAE encoder -> latent
latent denoising
latent -> VAE decoder -> image
```

U-Net vs DiT：

```text
U-Net: 卷积、多尺度、skip connection
DiT: latent patch tokens + Transformer denoiser
```

Control：

```text
ControlNet: pose/depth/edge/seg 等结构条件
LoRA: 低秩个性化/风格适配
DreamBooth: 学特定主体，效果强但易过拟合
Textual Inversion: 学新 token embedding
Inpainting: image + mask + prompt 局部重绘
```

Video diffusion：

```text
image: H x W
video: T x H x W
难点: temporal consistency / identity / motion / physics / cost
```

Rectified Flow：

```text
x_t = (1-t)x_0 + t x_1
learn v_theta(x_t,t) ~= x_1 - x_0
```

评估：

```text
FID: 分布质量
CLIPScore: 图文对齐
Human preference: 主观质量
Video: temporal consistency / flicker / motion
Production: safety + latency + cost + user acceptance
```

## 长上下文 / Context Engineering / GraphRAG 速查

长上下文不是万能：

- prefill 慢。
- KV Cache 大。
- 输入 token 成本高。
- lost in the middle。
- 噪声干扰。
- 权限和隐私风险。

RoPE：

```text
theta_i = base^(-2i / d)
angle(m, i) = m * theta_i

[x_2i'  ]   [cos(m theta_i)  -sin(m theta_i)] [x_2i  ]
[x_2i+1'] = [sin(m theta_i)   cos(m theta_i)] [x_2i+1]
```

RoPE scaling：

```text
Position Interpolation: m' = m / s, s = L_target / L_train
NTK-aware: adjust RoPE base / frequency spectrum
YaRN: frequency-aware interpolation/scaling
LongRoPE: searched/non-uniform scaling + long-context adaptation
```

长上下文成本：

```text
prefill attention ~= O(L^2 * d)
KV bytes = 2 * layers * batch * seq_len * kv_heads * head_dim * bytes
```

Ring Attention：

```text
split sequence across devices
K/V blocks circulate in a ring
each device accumulates blockwise attention for its Q block
```

StreamingLLM：

```text
keep first sink tokens + keep recent window
drop middle old tokens
```

提醒：StreamingLLM 适合流式长生成，不等于完整长文档任意位置问答。

Context Engineering：

```text
用户任务 -> 检索/记忆/工具/历史
-> 权限过滤 -> 排序/去重/压缩
-> token budget -> context packaging
-> 引用/trace/eval
```

Prompt Engineering vs Context Engineering：

- Prompt Engineering：怎么写指令。
- Context Engineering：给模型什么信息，以及怎么选、排、压缩、过滤、评估。

Context Builder 输入：

- 当前问题。
- 系统规则。
- 用户权限和状态。
- 对话历史。
- RAG 证据。
- 工具结果。
- Agent memory。

压缩风险：

- 漏掉 gold evidence。
- 摘要幻觉。
- 丢失细节。
- 权限信息被混入摘要。

Query 增强：

- query rewrite：补全和改写问题。
- HyDE：生成假设答案再检索。
- multi-query：多个角度扩召回。

GraphRAG：

```text
文档 -> 实体/关系抽取 -> 图构建
-> 社区发现/层级摘要 -> 子图/社区检索
-> 图证据 + 原文证据 -> answer
```

适合：

- 多跳问答。
- 跨文档关系。
- 组织、金融、法律、风控、医学。
- 全局摘要。

不适合：

- 文档少。
- 关系简单。
- 普通 RAG 已够用。
- 图谱更新和抽取质量不可控。

长上下文 eval：

- Needle-in-a-haystack：证据位置敏感性。
- LongBench：多任务长上下文理解。
- RULER：检索、聚合、多跳和长度外推合成评测。
- NeedleBench：needle / multi-needle 长上下文检索。
- NoLiMa：减少词面匹配捷径，测真正理解。
- 相关专题：[41_长上下文进阶_RoPEScaling_YaRN_LongRoPE_RingAttention面试.md](41_长上下文进阶_RoPEScaling_YaRN_LongRoPE_RingAttention面试.md)

Agent memory：

- short-term：当前对话。
- working：当前任务状态。
- long-term：稳定用户偏好和事实。
- retrieval memory：可检索历史和经验。

评估：

- context recall。
- context precision。
- faithfulness。
- citation accuracy。
- permission correctness。
- token cost / latency。
