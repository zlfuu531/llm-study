# 推理部署与性能优化

## 面试目标

这一章要达到的状态：

- 能区分 prefill 和 decode。
- 能解释吞吐、延迟、并发、显存之间的取舍。
- 能讲清量化、KV Cache、Continuous Batching、PagedAttention、Speculative Decoding。
- 能回答“线上部署一个大模型服务要考虑什么”。

## 推理阶段

大模型推理通常分两段：

### Prefill

处理用户 prompt。

特点：

- 输入长度可能很长。
- 可以并行计算所有 token 的 hidden states。
- 主要瓶颈常是计算。

### Decode

逐 token 生成回答。

特点：

- 每步只生成一个新 token。
- 依赖 KV Cache。
- 主要瓶颈常是内存带宽和 KV Cache 读写。

面试表达：

> Prefill 是把 prompt 一次性喂进模型，decode 是自回归逐 token 生成。二者瓶颈不同，所以优化手段也不同：prefill 更关注矩阵计算吞吐，decode 更关注 KV Cache、批处理和内存访问效率。

## 延迟和吞吐

常见指标：

- TTFT：Time To First Token，首 token 延迟。
- TPOT：Time Per Output Token，每个输出 token 时间。
- QPS：每秒请求数。
- Tokens/s：每秒生成 token 数。
- P50/P95/P99 延迟。

取舍：

- 批处理能提高吞吐，但可能增加单请求延迟。
- 长上下文会增加 prefill 成本和 KV Cache 显存。
- 高并发需要调度、限流、缓存和降级。

## KV Cache 优化

问题：

- KV Cache 随请求数和上下文长度增长。
- 长上下文、多轮对话、高并发时显存压力大。

优化：

- MQA/GQA/MLA 减少 K/V head 数或压缩 K/V。
- PagedAttention 管理 KV Cache 内存碎片。
- prefix cache 复用相同前缀。
- context truncation 或 sliding window。
- 量化 KV Cache。

## PagedAttention

直觉：

- 把 KV Cache 管理成分页内存。
- 避免为每个请求预留大块连续显存。
- 提高显存利用率。

面试表达：

> PagedAttention 借鉴操作系统分页思想，把 KV Cache 拆成 block 管理，解决不同请求长度不一致导致的显存碎片和浪费问题。它是 vLLM 高吞吐推理的重要基础之一。

## Continuous Batching

传统 batching：

- 等一批请求一起跑。
- 批内请求结束时间不同，容易浪费。

Continuous batching：

- 动态把新请求加入正在运行的 batch。
- 已完成请求退出，未完成请求继续。
- 更适合流式生成和请求长度不一致场景。

面试表达：

> Continuous batching 不是静态凑齐一批再跑，而是在 decode 过程中动态调度请求，提高 GPU 利用率和整体吞吐。

## 量化

目的：

- 降低显存。
- 提高推理速度。
- 降低部署成本。

常见类型：

- FP16/BF16
- INT8
- INT4
- GPTQ
- AWQ

注意：

- 量化可能带来精度损失。
- 对不同任务、模型和层的影响不同。
- 需要用业务评测集验证。

面试表达：

> 量化不是无脑压到越低越好。它降低显存和带宽压力，但可能损失效果，所以要结合业务评测看 accuracy、hallucination、格式遵循和安全性。

## Speculative Decoding

思想：

- 用小模型快速草拟多个 token。
- 大模型并行验证这些 token。
- 如果接受，就一次推进多个 token。

优点：

- 在不明显损失输出分布的前提下降低延迟。

限制：

- 小模型预测要足够接近大模型。
- 系统复杂度增加。

## 推理服务设计

一个线上大模型服务需要考虑：

- 模型加载和热启动。
- GPU 显存规划。
- 请求队列和调度。
- 流式输出。
- 超时和取消。
- 限流和鉴权。
- prompt 模板管理。
- 日志、trace、监控。
- 评估和灰度。
- 敏感信息和安全。

## 显存估算

主要组成：

- 模型权重。
- KV Cache。
- 激活和临时 buffer。
- 框架开销。

粗略：

```text
权重显存 ≈ 参数量 * bytes_per_param
KV cache ≈ batch * seq_len * layers * kv_heads * head_dim * 2 * bytes
```

例子：

- 7B FP16 权重大约 14GB。
- INT4 权重大约 3.5GB 加额外 scale/metadata。
- 真实部署还要考虑 KV Cache 和运行时开销。
