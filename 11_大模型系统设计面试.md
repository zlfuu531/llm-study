# 大模型系统设计面试

这章专门准备白板类、项目深挖类、AI 应用工程类面试题。很多 2025-2026 的大模型岗位不再只问“RAG 是什么”“Agent 是什么”，而是问：

- 如果让你设计一个企业知识库问答系统，你怎么做？
- 如果 Agent 要调用内部系统，怎么保证权限、审计和失败恢复？
- 如果线上 TTFT/TPOT 变高，你怎么排查？
- 如果 RAG 召回正确但答案仍然错，系统哪里可能有问题？

系统设计题不是背架构图，而是展示你能把模型、检索、工具、评估、成本、安全和上线闭环放到同一个工程系统里考虑。

## 一套通用答题框架

遇到系统设计题，先不要急着画组件。按这个顺序回答最稳：

```text
1. 澄清场景和约束
2. 定义指标和 SLO
3. 拆离线链路和在线链路
4. 画核心架构
5. 讲关键取舍
6. 讲评估、监控和 bad case 回流
7. 讲安全、权限、降级和回滚
```

### 1. 澄清场景和约束

面试官问“设计一个 RAG 系统”时，你可以先反问或主动补充：

- 面向谁：内部员工、客服、开发者、普通用户？
- 知识来源：PDF、网页、数据库、工单、代码仓库、实时 API？
- 数据规模：几万文档还是千万级 chunk？
- 更新频率：每天批量更新，还是分钟级同步？
- 准确性要求：允许拒答吗？需要引用吗？
- 延迟要求：P95 2 秒、5 秒还是 15 秒？
- 成本约束：单次请求成本是否敏感？
- 风险等级：是否涉及权限、财务、医疗、代码执行？

这一步的作用是把“泛泛设计”变成“有约束的工程设计”。

### 2. 定义指标

系统设计题一定要先说指标，否则后面很容易变成堆组件。

| 类型 | 指标 | 面试里怎么解释 |
| --- | --- | --- |
| 检索质量 | Recall@k、MRR、NDCG、答案证据命中率 | 先保证找得到证据 |
| 生成质量 | faithfulness、answer relevance、citation correctness | 再保证回答忠于证据 |
| Agent 成功率 | task success rate、tool call accuracy、argument accuracy | 看任务轨迹，不只看最终文本 |
| 性能 | TTFT、TPOT、P95/P99、tokens/s、QPS | 区分首 token 延迟和持续生成速度 |
| 成本 | tokens/request、cost/request、cache hit rate | 大模型系统必须算账 |
| 安全 | 拒答准确率、越权调用率、prompt injection 成功率 | 上线系统不能只看效果 |
| 运营 | 用户满意度、转人工率、bad case 数量 | 连接真实业务反馈 |

面试口语版：

> 我会把指标分成质量、性能、成本和安全四类。RAG 先看检索是否命中 gold evidence，再看答案是否忠于证据；Agent 不能只看最终答案，要看工具选择、参数、步骤数、失败恢复和权限违规。

## 大模型应用通用架构

一个生产级 LLM 应用可以这样讲：

```text
Client / UI
  -> API Gateway
      -> Auth / Tenant / Rate Limit
      -> Model Router
      -> Orchestrator
          -> Context Builder
              -> RAG Retriever / Memory / User Profile
          -> Tool Runtime
              -> Tool Registry / MCP Client / Permission Guard
          -> Prompt & Policy Layer
          -> LLM Serving / External Model API
      -> Post-processing
      -> Logs / Traces / Eval / Feedback
```

每层职责：

- API Gateway：鉴权、租户隔离、限流、请求大小限制、灰度。
- Model Router：根据任务难度、成本、延迟和安全等级选择模型。
- Orchestrator：编排 RAG、Agent、工具、上下文和状态。
- Context Builder：负责检索结果、历史记忆、用户信息、工具输出的选择和压缩。
- Tool Runtime：执行工具调用，做参数校验、权限、超时、重试和审计。
- LLM Serving：自部署或外部 API，关注 TTFT、TPOT、batch、KV Cache 和成本。
- Eval/Trace：记录每次请求的完整链路，用于复盘和回归测试。

面试里要强调：大模型只是系统里的一个组件，不是整个系统。

推理服务细节可以跳到独立专题：[24_推理引擎_vLLM_SGLang_TensorRTLLM面试.md](24_推理引擎_vLLM_SGLang_TensorRTLLM面试.md)。系统设计题里只要把核心链路讲清，但推理部署/AI Infra 岗会继续追问 prefill/decode、KV Cache、PagedAttention、continuous batching、chunked prefill、prefix cache、spec decoding 和 TTFT/TPOT 排查。

如果系统设计继续追多卡 serving、PD 分离、KV Cache 传输和 MoE serving，跳到：[39_分布式推理_PD分离_KVCache传输与MoEServing面试.md](39_分布式推理_PD分离_KVCache传输与MoEServing面试.md)。

## 企业级 RAG 系统设计

### 1. 离线建库

```text
数据源
  -> 文档采集
  -> 解析/OCR/表格抽取
  -> 清洗去重
  -> 结构化 chunk
  -> metadata 标注
  -> embedding
  -> 向量索引 + 关键词索引
  -> 版本管理和质量抽检
```

关键点：

- 文档解析要保留标题、章节、页码、来源、更新时间、权限标签。
- chunk 不是固定长度硬切，优先按标题、段落、表格和语义单元切。
- 对 PDF、表格、图片、代码块要有特殊处理。
- 建索引时保留 `doc_id`、`chunk_id`、`version`、`source`、`permission`。
- 文档更新后要能增量重建、删除旧 chunk、失效缓存。

### 2. 在线问答

```text
query
  -> 权限和意图识别
  -> query rewrite / multi-query
  -> hybrid retrieval
  -> rerank
  -> context packaging
  -> LLM generation
  -> citation / refusal / consistency check
  -> trace + feedback
```

关键取舍：

- dense retrieval 擅长语义相似，BM25 擅长关键词、编号、专有名词。
- rerank 能把真正含答案的 chunk 排前，但会增加延迟。
- top-k 不是越大越好，太大会带入噪声和成本。
- 长上下文不是 RAG 的替代品，长上下文也需要上下文选择和证据排序。
- 证据不足时应该拒答，不应该让模型凭常识补全。

更完整的 embedding、BM25、hybrid retrieval、HNSW/IVF/PQ、reranker、向量库生产能力和检索评估见：[22_Embedding_Reranker与向量检索面试.md](22_Embedding_Reranker与向量检索面试.md)。

### 3. RAG 从 demo 到生产的差距

demo 常见写法：

```text
PDF -> split -> embedding -> vector db -> prompt -> answer
```

生产级系统要补上：

- 文档解析质量评估。
- chunk 策略和版本管理。
- hybrid retrieval、rerank、权限过滤。
- 引用、拒答、冲突处理。
- 离线 eval 和线上反馈。
- bad case 回流。
- 缓存、限流、成本监控。
- 数据安全和 prompt injection 防护。

面试口语版：

> demo 只证明链路能跑，生产要证明它可评估、可更新、可追溯、可控成本、可处理权限和失败。

### 4. RAG 线上问题排查

用户说“答错了”，不要直接改 prompt。按层排查：

```text
query 是否表达清楚
  -> rewrite 是否改错
  -> gold evidence 是否存在
  -> chunk 是否包含完整答案
  -> embedding/BM25 是否召回
  -> rerank 是否排前
  -> context packaging 是否截断证据
  -> prompt 是否要求基于证据
  -> 模型是否忠于证据
  -> 引用和后验检查是否发现问题
```

对应指标：

- 召回没命中：看 Recall@k、query rewrite、chunk、embedding、hybrid。
- 召回命中但排后：看 rerank、NDCG、top-k。
- 上下文有证据但答错：看 faithfulness、prompt、模型能力、上下文冲突。
- 答案没引用或引用错：看 citation correctness 和证据 span。

## Agent 系统设计

### 1. 先判断是否需要 Agent

不要为了炫技上 Agent。

| 场景 | 推荐方案 |
| --- | --- |
| 流程固定、规则清晰 | 普通工作流 |
| 需要高一致性、高安全 | 规则引擎 + 人工确认 |
| 需要多步探索、工具反馈 | Agent |
| 需要在不确定环境中规划 | Agent / 状态图 |
| 只是问答 | Chatbot 或 RAG |

面试口语版：

> Agent 适合目标明确但执行路径不固定的任务。如果流程固定，我会优先用 workflow，因为更稳定、更便宜、更好评估。

### 2. 生产级 Agent 架构

```text
User Goal
  -> Task Router
  -> Planner / State Graph
  -> Context Builder
  -> Tool Selector
  -> Argument Validator
  -> Permission Guard
  -> Tool Executor
  -> Observation Parser
  -> State Update
  -> Stop Judge / Human Approval
  -> Final Response
```

必须说清的工程模块：

- 状态：任务目标、当前步骤、已调用工具、工具结果、失败次数。
- 工具注册表：工具描述、参数 schema、返回 schema、权限等级。
- 参数校验：类型、必填、枚举、范围、业务规则。
- 权限控制：用户身份、租户、数据范围、高风险动作确认。
- 终止条件：最大步数、成功条件、失败次数、成本预算。
- 失败恢复：重试、换工具、降级、询问用户、人工接管。
- 观测：trace、每步 action/observation、耗时、成本、错误码。

### 3. Agent Loop 不是裸循环

错误写法：

```text
while not done:
  ask model what to do
  execute tool
```

生产写法要有边界：

```text
while not done and steps < max_steps and budget_ok:
  state = read_state()
  tools = filter_tools_by_task_and_permission(state, user)
  context = build_context(state, memory, retrieved_docs)
  action = model_decide(context, tools)
  checked = validate_action(action, schema, permission)
  result = execute_with_timeout_retry_or_approval(checked)
  state = update_state(state, action, result)
  done = judge_by_state_and_task_criteria(state)
```

核心点：模型只负责提出动作建议，系统负责校验、执行、审计和兜底。

### 4. MCP 在系统设计里怎么讲

MCP 可以放在工具接入层：

```text
Agent Orchestrator
  -> MCP Client
      -> MCP Server A: database tools
      -> MCP Server B: file resources
      -> MCP Server C: ticket prompts/tools
```

你要能区分三层：

- Function Calling / Tool Calling：模型输出结构化工具名和参数。
- MCP：工具、资源、提示模板如何被标准化发现和调用。
- Agent 框架：负责任务状态、规划、工具编排、评估和失败恢复。

面试口语版：

> 我不会把 MCP 说成替代 Function Calling。Function Calling 更偏模型输出格式，MCP 更偏外部工具和上下文接入协议，Agent 编排层可以用 Function Calling 表达动作，再通过 MCP Client 调 MCP Server 的工具或资源。

### 5. Agent 评估

Agent 不能只看“最终答得像不像”。要看轨迹：

- task success rate：任务是否完成。
- tool call accuracy：工具选得是否正确。
- argument accuracy：参数是否正确。
- step efficiency：是否绕路。
- recovery rate：工具失败后是否恢复。
- human approval rate：人工确认触发是否合理。
- safety violation rate：是否越权、误操作、泄露数据。
- cost per task：token、工具调用和延迟成本。

评估集要覆盖：

- 正常任务。
- 工具返回空。
- 工具超时。
- 参数缺失。
- 权限不足。
- 高风险动作。
- prompt injection。
- 多轮任务变更。

## 模型网关设计

模型网关不是简单转发 API。它通常负责：

```text
鉴权/租户
  -> 请求规范化
  -> 模板和 prompt 版本
  -> 模型路由
  -> token 预算
  -> 限流/排队/熔断
  -> 安全过滤
  -> streaming
  -> retry/fallback
  -> 日志/成本/监控
  -> 灰度/回滚
```

### 1. 为什么按 token 限流

LLM 请求成本不只和 QPS 有关，还和输入输出 token 有关。

两个请求都是 1 QPS：

- 请求 A：输入 200 token，输出 100 token。
- 请求 B：输入 20k token，输出 4k token。

对 GPU、外部 API 成本和延迟的压力完全不同。所以生产系统通常同时限制：

- requests per minute。
- input tokens per minute。
- output tokens per minute。
- concurrent requests。
- max context length。
- max generation tokens。

### 2. 模型路由

常见路由策略：

- 简单问题用小模型，复杂问题用大模型。
- 高风险问题用更强安全模型或人工确认。
- 长上下文任务走支持长上下文的模型。
- 结构化抽取走便宜稳定的模型。
- 大模型失败或超时时 fallback 到备用模型。

路由不能只看成本，还要看质量和失败后果。

更完整的 LLMOps、模型网关、OpenTelemetry GenAI、trace、prompt 版本、成本治理、灰度回滚和事故复盘见：[20_LLMOps模型网关与可观测性面试.md](20_LLMOps模型网关与可观测性面试.md)。

## 推理服务系统设计

### 1. Prefill 和 Decode

推理分两段：

- prefill：处理输入 prompt，计算首轮 KV Cache，主要和输入长度有关。
- decode：逐 token 生成，复用 KV Cache，常受显存带宽和 batch 调度影响。

指标：

- TTFT：Time To First Token，用户等第一个 token 的时间。
- TPOT：Time Per Output Token，生成每个 token 的平均时间。
- tokens/s：吞吐。
- P95/P99 latency：尾延迟。

### 2. 高并发架构

```text
Gateway
  -> Request Queue
  -> Scheduler
      -> Prefill Workers
      -> Decode Workers
  -> Inference Engine
      -> KV Cache Manager
      -> Continuous Batching
      -> Prefix Cache
  -> Streaming Response
```

关键技术：

- Continuous Batching：decode 阶段动态合批，提高 GPU 利用率。
- PagedAttention：把 KV Cache 用 block 管理，减少碎片，提高可服务请求数。
- Prefix Cache：相同前缀复用 KV，适合固定系统 prompt、少量模板。
- Chunked Prefill：长 prompt 分块，缓解长请求阻塞短请求。
- Speculative Decoding：小模型草拟，大模型验证，接受率高才明显加速。
- 更完整的 draft model、EAGLE、Medusa、MTP 和 accept rate 排查见：[38_推测解码_DraftModel_EAGLE_Medusa_MTP面试.md](38_推测解码_DraftModel_EAGLE_Medusa_MTP面试.md)。
- 量化：降低权重和 KV Cache 显存，但可能损失精度。

### 3. TTFT 变高怎么排查

```text
入口排队
  -> 鉴权/安全过滤耗时
  -> prompt 过长
  -> tokenizer 慢
  -> prefill batch 被长请求阻塞
  -> prefix cache 未命中
  -> GPU 利用率/显存碎片
  -> 网络和流式 flush
```

### 4. TPOT 变高怎么排查

```text
decode batch 太大
  -> KV Cache 显存压力
  -> 内存带宽瓶颈
  -> 量化 kernel 不高效
  -> tensor parallel 通信开销
  -> 输出过长
  -> speculative decoding 接受率低
```

面试重点：TTFT 更偏 prefill 和排队，TPOT 更偏 decode 阶段持续生成。

## 评估和可观测性

### 1. 每次请求要记录什么

RAG 请求：

```text
request_id
user/tenant
query
query_rewrite
retrieved_chunk_ids
rerank_scores
context_tokens
prompt_version
model_version
answer
citations
latency
cost
user_feedback
```

Agent 请求：

```text
request_id
goal
state snapshots
tool list
chosen tool
arguments
validation result
tool output
error/retry
human approval
final status
step count
cost/latency
```

有了这些日志，才能做 replay、回归测试和 bad case 复盘。

### 2. LLM-as-judge 怎么用

LLM-as-judge 可以用来做初筛，例如判断答案相关性、忠实性、格式是否合规。但不能盲信：

- judge prompt 要固定版本。
- 需要少量人工标注集校准。
- 复杂业务规则要用规则或人工复核。
- 高风险场景不能只靠模型评判模型。

面试口语版：

> 我会把 LLM-as-judge 当成辅助评估器，不当成唯一真值。关键样本要人工标注，线上 bad case 要回流到固定 eval 集。

## 安全和权限

大模型系统常见风险：

- prompt injection：检索文档或网页里藏恶意指令。
- 数据泄露：模型把无权限上下文返回给用户。
- 工具越权：Agent 调用了不该调用的工具。
- 向量库投毒：恶意文档进入索引影响答案。
- jailbreak：绕过安全策略。
- 成本攻击：诱导超长上下文和超长输出。
- 幻觉执行：模型编造工具结果或执行不存在的动作。

治理方法：

- 检索前和检索后都做权限过滤。
- 工具按用户身份和任务动态暴露。
- 高风险动作必须 human-in-the-loop。
- 工具执行前做参数校验和业务规则校验。
- RAG prompt 明确区分“证据内容”和“系统指令”。
- 输出前做敏感信息检测和引用校验。
- 对外部网页、用户上传文档做隔离和风险标签。
- 对 token、上下文长度和工具次数设置预算。

## 高频系统设计题模板

### 题 1：设计企业知识库问答系统

回答结构：

```text
场景约束 -> 指标 -> 离线建库 -> 在线问答 -> 权限 -> 评估 -> 监控 -> 迭代
```

可以这样说：

> 我会先定义业务问题、文档规模、更新频率、权限和延迟指标。离线侧做文档解析、清洗、结构化 chunk、metadata、embedding、向量索引和 BM25 索引，并维护版本。在线侧先做鉴权和 query rewrite，再做 hybrid retrieval、rerank、context packaging，最后让模型基于证据回答并返回引用。评估上分检索、上下文、生成和业务四层；线上记录完整 trace，bad case 回流到 eval 集和索引优化。

### 题 2：设计一个能操作内部系统的客服 Agent

回答结构：

```text
任务边界 -> 工具清单 -> 状态机 -> 权限 -> HITL -> 失败恢复 -> 评估
```

可以这样说：

> 我会把 Agent 设计成受控状态机，而不是让模型自由调用工具。工具注册表里定义每个工具的 schema、权限等级和幂等性。执行前做参数校验和权限校验；查信息类工具可以自动执行，改订单、退款、发通知这类高风险动作要人工确认。每一步记录 action、arguments、observation、错误和耗时。评估看任务完成率、工具选择准确率、参数准确率、平均步骤、人工确认触发率和安全违规率。

### 题 3：设计高并发大模型服务

回答结构：

```text
流量模型 -> 网关 -> 调度 -> 推理引擎 -> 缓存 -> 限流降级 -> 监控
```

可以这样说：

> 我会先拆 TTFT、TPOT、tokens/s 和 P95 指标。入口通过网关做鉴权、token 预算、限流和排队；调度层做 continuous batching，避免 GPU 空转；推理引擎用 KV Cache、PagedAttention、prefix cache 和必要的量化；长 prompt 可以用 chunked prefill 缓解阻塞；线上监控队列长度、batch size、cache 命中率、GPU 利用率、显存、TTFT、TPOT 和错误率。降级策略包括缩短 max tokens、切小模型、关闭复杂工具、排队或拒绝低优先级请求。

### 题 4：设计私有 eval 和灰度上线流程

回答结构：

```text
样本来源 -> 标注字段 -> 离线回归 -> 小流量灰度 -> 监控 -> 回滚
```

可以这样说：

> 私有 eval 样本来自真实 query、历史工单、bad case 和人工构造边界样本。RAG 样本要标 gold evidence、reference answer、是否允许拒答和权限标签；Agent 样本要标期望工具轨迹、参数和高风险动作。每次改 prompt、chunk、embedding、rerank、模型或工具 schema 都跑离线回归。上线先小流量灰度，监控成功率、P95、成本、拒答、安全违规和用户反馈，异常时能回滚模型、prompt、索引和工具版本。

## 面试常见追问

### Q1：为什么你的系统不直接用长上下文？

长上下文能放更多材料，但不能自动保证材料正确、排序合理、权限正确、成本可控。RAG 的价值不只是省上下文，还包括证据选择、引用、权限过滤、版本管理和评估闭环。实际项目可以 RAG + 长上下文结合，但仍要做上下文选择。

### Q2：RAG 召回正确但答案错，怎么办？

说明问题可能在 rerank 后的 context packaging、prompt 约束、模型能力、上下文冲突或引用校验。先确认 gold evidence 是否进入最终 prompt，再看是否被截断、是否有冲突证据、模型是否忠于证据。治理包括更强证据约束、结构化引用、答案-证据一致性检查和拒答策略。

### Q3：Agent 为什么要有状态机？

因为纯 prompt 驱动的 Agent 难控制、难恢复、难评估。状态机能明确当前步骤、允许的工具、终止条件、失败次数和人工确认点，让系统可观测、可回放、可回滚。

### Q4：工具调用失败是让模型重试还是系统重试？

分情况。网络超时、临时 5xx 可以系统重试；参数缺失或格式错误可以把错误反馈给模型修正；权限不足不应该重试，要拒绝或要求授权；高风险操作失败要记录并交给人工处理。

### Q5：怎么证明一次改动真的提升了系统？

不要只看单个例子。用固定离线 eval 集跑回归，看质量、延迟、成本、安全是否同时满足门槛；再小流量灰度，看线上反馈和 bad case。RAG/Agent 改动还要看分层指标，避免最终答案偶然变好但召回或安全变差。

## 一页背诵版

系统设计题的核心不是画很多框，而是讲清“约束、指标、链路、取舍、评估、安全、上线”。RAG 系统分离线建库和在线问答，重点是文档解析、chunk、hybrid retrieval、rerank、引用、权限、评估和 bad case 回流。Agent 系统要从裸 prompt 升级成受控 Harness：状态、工具注册表、参数校验、权限、HITL、失败恢复、trace 和 eval。模型服务要区分 TTFT 和 TPOT，关注 prefill/decode、KV Cache、continuous batching、PagedAttention、prefix cache、chunked prefill、限流和降级。生产级大模型系统一定要讲监控、私有 eval、灰度和回滚。
